## Q01. 트랜잭션 — 원자성 / COMMIT / ROLLBACK

### 지문 키워드

> 은행에서 A 계좌 출금 + B 계좌 입금을 **하나의 트랜잭션**으로 처리
> ① 여러 작업을 묶어 **모두 성공하거나 모두 실패**하도록 보장하는 특성
> ② 결과를 DB에 **영구 반영**하는 명령어
> ③ 오류 시 **이전 상태로 되돌리는** 명령어

### 왜 이게 답인가?

| 빈칸 | 지문 핵심 키워드            | 연결 포인트                     | 정답            |
| --- | -------------------- | -------------------------- | ------------- |
| ①  | 모두 성공 or 모두 실패 (All or Nothing) | 원자성 = 쪼갤 수 없음 | **원자성(Atomicity)** |
| ②  | 결과를 **영구 반영**        | 확정 명령                      | **COMMIT**    |
| ③  | 이전 상태로 **되돌림**       | 취소/복귀 명령                   | **ROLLBACK**  |

### 메커니즘 — 트랜잭션 ACID

| 특성                | 의미                          | 관련 명령           |
| ----------------- | --------------------------- | --------------- |
| **A**tomicity(원자성) | 전부 반영 or 전부 취소             | COMMIT/ROLLBACK |
| **C**onsistency(일관성) | 실행 전후 DB 무결성 유지          | 제약조건            |
| **I**solation(격리성) | 동시 트랜잭션 간 간섭 없음           | LOCK/격리수준       |
| **D**urability(지속성) | COMMIT된 결과는 영구 보존         | 로그/체크포인트        |

```
[TCL 명령어]
COMMIT    → 변경 확정 (영구 반영)
ROLLBACK  → 변경 취소 (트랜잭션 시작 시점으로)
SAVEPOINT → 롤백 지점 지정 (부분 취소용)
```
> ⚠️ 이 문제는 원자성 위반 상황(출금은 됐는데 입금 실패) → ROLLBACK으로 출금까지 취소해야 원자성 보장

### ✅ 정답

① 원자성(Atomicity)  ② COMMIT  ③ ROLLBACK

## Q02. 문맥 교환 (Context Switching)

### 지문 키워드

> 인터럽트/스케줄러에 의해 **다른 프로세스로 전환**
> 현재 프로세스의 **상태 정보를 저장**하고, 새 프로세스의 **상태를 복원**

### 왜 이게 답인가?

| 지문 키워드                    | 연결 포인트                   | 정답        |
| ------------------------- | ------------------------ | --------- |
| 상태 저장 + 복원, 프로세스 전환       | Context(문맥) = 프로세스 상태 정보 | 문맥 교환     |

### 메커니즘

```
[문맥 교환 과정]
프로세스 A 실행 중
   ↓ 인터럽트/타임슬라이스 만료
A의 상태(PC, 레지스터 등) → PCB에 저장
   ↓
B의 PCB에서 상태 복원
   ↓
프로세스 B 실행

* 저장되는 정보 = 문맥(Context): 프로그램 카운터(PC), 레지스터 값,
  스택 포인터 등 → PCB(Process Control Block)에 보관
```
> ⚠️ 문맥 교환 자체는 CPU 작업만 소모하는 **오버헤드**(순수 부가비용) → 너무 잦으면 성능 저하

### ✅ 정답

문맥 교환(Context Switching)

## Q03. C언어 — 소수 판별 + 삼항 증가

### 문제

```c
int temp = 2, flag;
while (temp < 8) {
    flag = 2;
    while (flag < temp && temp % flag != 0) {
        flag++;
    }
    if (flag == temp || temp == 2)
        printf("%d ", temp);
    temp += (temp % 3 == 0) ? 1 : 2;
}
```

### 핵심 개념 — 무엇을 하는 코드인가?

안쪽 while은 `flag`를 2부터 늘리며 `temp`를 나눠본다.
- `temp`가 소수면 → 나누어지는 값이 없어 `flag`가 `temp`까지 도달 → `flag == temp` 참 → 출력
- `temp`가 2면 무조건 출력 (`temp == 2` 조건)
- 증가폭: `temp`가 3의 배수면 +1, 아니면 +2

> ⚠️ **핵심**: 소수 판별 + 불규칙 증가폭 조합 → 값을 직접 추적해야 함

### 단계별 트레이싱

| temp | 안쪽 while 결과 flag | 출력 조건               | 출력? | 다음 증가폭          |
| ---- | --------------- | ------------------- | --- | -------------- |
| 2    | (temp==2 즉시)    | temp==2 → 참         | **2 출력** | 2%3≠0 → +2 → 4 |
| 4    | flag: 2, 4%2==0 → 멈춤, flag=2 | flag(2)==temp(4)? ✗, temp==2? ✗ | ✗   | 4%3≠0 → +2 → 6 |
| 6    | flag: 2, 6%2==0 → 멈춤, flag=2 | 2==6? ✗             | ✗   | 6%3==0 → +1 → 7 |
| 7    | flag: 2→3→4→5→6→7(7<7 거짓 종료), flag=7 | 7==7? ✓ | **7 출력** | 7%3≠0 → +2 → 9 |
| 9    | 9 < 8 거짓 → 루프 종료 | -                   | -   | -              |

> 💡 4는 flag=2에서 `4%2==0`이라 안쪽 while이 바로 멈추고 flag가 2에 머무름 → 소수 아님.
> 6도 마찬가지. 7만 끝까지 돌아 flag=7 → 소수.

### ✅ 정답

```
2 7 
```

## Q04. Java — 생성자 호출 순서 + 오버라이딩 + 필드 초기화 시점

### 문제

```java
class Parent {
    public Parent() {
        System.out.println("1");
        display();               // ← 여기가 함정
    }
    public void display() { System.out.println("Super"); }
}
class Child extends Parent {
    private String info = "Child Info";
    private int count = 100;
    public Child() { System.out.println("2"); }
    public void display() {
        System.out.println(info);
        System.out.println(count);
    }
}
class Main {
    public static void main(String[] args) {
        Parent obj1 = new Parent();
        Parent obj2 = new Child();
    }
}
```

### 🤔 이 문제의 3가지 함정

**함정 1 — 생성자 안에서 `display()` 호출 시 어느 것이 실행되나?**
`Parent` 생성자가 `display()`를 부르지만, 실제 객체가 `Child`면 **오버라이딩된 Child.display()가 실행**된다 (동적 바인딩).

**함정 2 — 필드 초기화 순서**
자바 객체 생성 순서는:
```
1) super() 생성자 실행 (부모 생성자 완료)
2) 자식 필드 초기화 (info="Child Info", count=100)
3) 자식 생성자 본문 실행
```
즉 **부모 생성자가 돌 때는 자식 필드가 아직 초기화 전** → `info=null`, `count=0`(기본값)

**함정 3 — obj1과 obj2는 다르다**
`new Parent()`는 진짜 Parent → `Parent.display()` = "Super"
`new Child()`는 Child → 부모 생성자 속 `display()`는 `Child.display()` 실행

### 단계별 트레이싱

**`obj1 = new Parent()`**
```
Parent() 실행 → "1"
display() → Parent.display() → "Super"
```

**`obj2 = new Child()`**
```
Child() 호출 → 먼저 super() = Parent() 실행
  Parent(): "1" 출력
  display() 호출 → 실제 타입 Child → Child.display() 실행
     info 출력 → 아직 초기화 전 → null
     count 출력 → 아직 초기화 전 → 0
  (Parent 생성자 종료)
  이제 Child 필드 초기화: info="Child Info", count=100
  Child 생성자 본문: "2" 출력
```

### ✅ 정답

```
1
Super
1
null
0
2
```
> ⚠️ `null`과 `0`이 핵심 — "부모 생성자 → 자식 필드 초기화"라는 순서를 모르면 "Child Info / 100"으로 틀림

## Q05. SQL — GROUP BY + HAVING + 집계함수

### 문제 (요구 조건)

- GROUP BY, HAVING 절 **반드시** 사용
- WHERE 절 **사용 금지**
- SELECT에 **별칭(Alias)** 포함
- 집계함수(MIN, MAX) 사용
- **최고점수 - 최저점수 ≥ 10** 인 과목만 출력

원하는 결과:

| 과목이름 | 최고점수 | 최저점수 |
| ---- | ---- | ---- |
| DB   | 97   | 83   |
| OS   | 75   | 60   |

### 왜 이렇게 쓰나?

| 조건               | SQL 구현                         |
| ---------------- | ------------------------------ |
| 과목별로 묶기          | `GROUP BY 과목이름`               |
| 최고/최저 점수         | `MAX(점수)`, `MIN(점수)`          |
| 별칭               | `AS 최고점수`, `AS 최저점수`          |
| 차이 10 이상 필터      | `HAVING MAX(점수) - MIN(점수) >= 10` |

> ⚠️ **핵심 함정**: "차이 10 이상"은 **집계 결과에 대한 조건**이므로 WHERE가 아니라 **HAVING**을 써야 한다.
> WHERE는 그룹화 **전** 개별 행을 거르고, HAVING은 그룹화 **후** 집계값을 거른다.

### 정답 SQL

```sql
SELECT 과목이름,
       MAX(점수) AS 최고점수,
       MIN(점수) AS 최저점수
FROM GRADE
GROUP BY 과목이름
HAVING MAX(점수) - MIN(점수) >= 10;
```

**검산**: JAVA(85,88,91) 차이=6 → 제외 / DB(92,97,83) 차이=14 → 포함 / OS(75,60) 차이=15 → 포함 ✅

### ✅ 정답 → 위 SQL

## Q06. 서브넷 비트 수 계산

### 문제

> 10.0.0.0/8 대역. 최소 **200개 서브넷**, 각 서브넷 **1,000대 이상 호스트**.
> 네트워크/브로드캐스트 주소 제외. 서브넷 낭비 없이 **최소 서브넷 비트 수**는?

### 핵심 개념 — 비트 배분

`/8`이므로 **호스트로 쓸 수 있는 비트 = 32 - 8 = 24비트**.
이 24비트를 **서브넷 비트(s) + 호스트 비트(h)** 로 나눠야 한다.

**① 호스트 조건: 서브넷당 1,000대 이상**
사용 가능 호스트 = `2^h - 2 ≥ 1000`
- h=10 → 2^10 - 2 = 1022 ✅ (h=9는 510 → 부족)
→ **호스트 비트 h = 10**

**② 서브넷 조건: 최소 200개**
서브넷 수 = `2^s ≥ 200`
- s=8 → 2^8 = 256 ✅ (s=7은 128 → 부족)
→ **서브넷 비트 s = 8**

**③ 비트 합 확인**: s + h = 8 + 10 = 18 ≤ 24 ✅ (여유 6비트, 조건 만족)

### 계산 요약표

| 조건        | 부등식             | 최소 비트    |
| --------- | --------------- | -------- |
| 호스트 1000+ | 2^h − 2 ≥ 1000  | h = 10   |
| 서브넷 200+  | 2^s ≥ 200       | **s = 8** |

> ⚠️ 문제가 묻는 것은 **서브넷 비트 수** → 답은 **8**
> (전체 프리픽스로 보면 /8 + 8 = /16)

### ✅ 정답

```
8
```

## Q07. Land Attack + 가용성

### 지문 키워드

> **출발지 IP와 목적지 IP가 동일**하게 설정된 비정상 패킷 다량 유입
> 시스템 자원 고갈 → 정상 서비스 불가
> ① 이런 공격의 명칭  ② 침해되는 정보보안 3대 요소

### 왜 이게 답인가?

| 지문 키워드                   | 연결 포인트                          | 정답        |
| ------------------------ | ------------------------------- | --------- |
| **출발지 IP = 목적지 IP**      | 자기 자신에게 응답하게 만드는 Land Attack의 정의 | ① Land Attack |
| **자원 고갈, 서비스 불가**        | 서비스를 못 쓰게 함 = 가용성 침해            | ② 가용성      |

### 메커니즘 — 정보보안 3대 요소(CIA)

| 요소                    | 의미            | 침해 예시              |
| --------------------- | ------------- | ------------------ |
| 기밀성(Confidentiality)  | 인가된 자만 접근     | 도청, 정보 유출          |
| 무결성(Integrity)        | 데이터 변조 방지     | 위·변조               |
| **가용성(Availability)** | 필요할 때 사용 가능   | **DoS/DDoS, Land Attack** |

```
[Land Attack]
공격자가 SYN 패킷의 출발지 IP를 피해자 IP로 위조
→ 피해자가 자기 자신에게 응답 시도 → 무한 루프/자원 고갈
→ DoS(서비스 거부)의 일종 → 가용성 침해
```
> 💡 유사 공격 구분: Smurf(ICMP 브로드캐스트), SYN Flooding(연결 미완성 남발), Land(SIP=DIP)

### ✅ 정답

① Land Attack(랜드 어택)  ② 가용성

## Q08. Python — isinstance / type + bool 함정

### 문제

```python
def check(v):
    if isinstance(v, str):     return len(v)
    elif isinstance(v, bool):  return 10
    elif type(v) == int:       return 1
    elif type(v) == float:     return 2
    elif isinstance(v, (list, tuple)): return len(v) * 2
    elif v is None:            return 0
    else:                      return -1
a="AI"; b=True; c=3.14; d=[1,2,3]; e=(10,20); f=None
print(check(a)+check(b)+check(c)+check(d)+check(e)+check(f))
```

### 🤔 핵심 함정 — `isinstance` vs `type`, 그리고 bool

**함정 1: 파이썬에서 `bool`은 `int`의 하위 클래스다**
`isinstance(True, int)` → True. 하지만 여기서는 `isinstance(v, bool)`이 `type(v)==int`보다 **위에** 있어서 bool이 먼저 걸린다.
> 만약 순서가 반대였다면 `True`가 int로 분류돼 다른 답이 나옴 → 순서가 핵심

**함정 2: `isinstance`는 상속 포함, `type()==`은 정확히 그 타입만**

### 단계별 트레이싱

| 변수 | 값        | 걸리는 분기              | 반환값       |
| --- | -------- | ------------------ | --------- |
| a  | "AI"     | isinstance str     | len=**2** |
| b  | True     | isinstance bool    | **10**    |
| c  | 3.14     | type==float        | **2**     |
| d  | [1,2,3]  | isinstance list    | 3×2=**6** |
| e  | (10,20)  | isinstance tuple   | 2×2=**4** |
| f  | None     | v is None          | **0**     |

**합계**: 2 + 10 + 2 + 6 + 4 + 0 = **24**

### ✅ 정답

```
24
```

## Q09. 폭포수 모델 + 개발 단계

### 지문 키워드

> **각 단계가 순차적**으로 진행, 이전 단계 완료 후 다음 단계


## Q11. CPM — 임계경로 + ES / LS / Slack

### 문제 (작업 정보표)

| 작업 | 선행작업 | 소요 기간 |
| --- | ---- | ----- |
| A  | -    | 12    |
| B  | A    | 15    |
| C  | A    | 14    |
| D  | B, C | 16    |
| E  | D    | 13    |
| F  | C    | 11    |

구하는 것: ① 임계경로 ② F의 ES ③ F의 LS ④ F의 Slack

### 핵심 개념 — 전진 계산(ES/EF) & 후진 계산(LS/LF)

- **ES(빠른 착수)** = 선행 작업들의 EF 중 **최댓값**
- **EF(빠른 종료)** = ES + 소요기간
- **LF(늦은 종료)** = 후행 작업들의 LS 중 **최솟값** (없으면 프로젝트 종료일)
- **LS(늦은 착수)** = LF − 소요기간
- **Slack(여유)** = LS − ES = LF − EF

### 단계별 계산 — 전진(Forward)

| 작업 | 선행 EF        | ES     | EF (=ES+기간)    |
| --- | ------------ | ------ | -------------- |
| A  | -            | 0      | 12             |
| B  | A:12         | 12     | 27             |
| C  | A:12         | 12     | 26             |
| D  | max(B27,C26) | 27     | 43             |
| E  | D:43         | 43     | 56             |
| F  | C:26         | **26** | 37             |

→ 프로젝트 종료일 = max(EF) = E의 **56**

### 임계경로

가장 긴 경로 = **A → B → D → E** (12+15+16+13 = **56**)
→ 이 경로 상 작업은 Slack = 0

### F 작업 후진 계산

F는 후행 작업이 없음 → LF = 프로젝트 종료일 = 56
```
LS(F) = LF − 기간 = 56 − 11 = 45
Slack(F) = LS − ES = 45 − 26 = 19
```

### ✅ 정답

① A → B → D → E
② ES = 26
③ LS = 45
④ Slack = 19

## Q13. SQL — 반환 튜플 수 (DISTINCT / COUNT)

### 문제 (학생 STUDENT 테이블 분포)

| DEPT | 인원                    |
| ---- | --------------------- |
| 전자과  | 40명 (1학년 10, 2학년 30)  |
| 전기과  | 60명 (2학년 30, 3학년 30)  |
| 건축과  | 50명 (1학년 20, 2학년 30)  |

### 각 SQL의 반환 튜플 수

**① `SELECT DEPT FROM STUDENT WHERE GRADE = 2;`**
2학년 학생 = 전자과 30 + 전기과 30 + 건축과 30 = **90행**
> ⚠️ DISTINCT 없음 → 학생 한 명당 한 행씩 그대로 반환 → 90

**② `SELECT DISTINCT DEPT FROM STUDENT WHERE GRADE = 1;`**
1학년이 있는 학과 = 전자과(10명), 건축과(20명) → 학과명 중복 제거
> DISTINCT → 서로 다른 DEPT 값만 → 전자과, 건축과 → **2행**

**③ `SELECT COUNT(DISTINCT GRADE) FROM STUDENT WHERE DEPT = '전기과';`**
전기과 학년 = 2학년, 3학년 → DISTINCT GRADE = {2, 3} → 개수 2
> ⚠️ **함정**: 문제는 "반환되는 **튜플 수**"를 물음. `COUNT(...)`는 집계 결과로 **한 줄(값 2)**을 반환 → 튜플 수 = **1**

### 요약표

| SQL | 의미           | 반환 튜플 수 |
| --- | ------------ | ------- |
| ①  | 2학년 전체(중복포함) | **90**  |
| ②  | 1학년 학과(중복제거) | **2**   |
| ③  | 집계 결과 1줄     | **1**   |

### ✅ 정답

① 90  ② 2  ③ 1

## Q14. C언어 — 배열 노드 값 swap + 포인터 연결

### 문제

```c
struct Node { int value; struct Node* next; };
void swapPairs(struct Node arr[], int size) {
    for (int i = 0; i + 1 < size; i += 2) {
        int temp = arr[i].value;
        arr[i].value = arr[i + 1].value;
        arr[i + 1].value = temp;
    }
}
int main() {
    struct Node nodes[3] = {
        {1, &nodes[2]},   // nodes[0]: value=1, next→nodes[2]
        {2, NULL},        // nodes[1]: value=2, next→NULL
        {3, &nodes[1]}    // nodes[2]: value=3, next→nodes[1]
    };
    swapPairs(nodes, 3);
    struct Node* curr = &nodes[0];
    while (curr != NULL) { printf("%d ", curr->value); curr = curr->next; }
}
```

### 🤔 핵심 — value만 바꾸고 next(연결)는 그대로

`swapPairs`는 **인접한 쌍의 value만 교환**한다. next 포인터는 건드리지 않음.

**1단계: swap 전 value 배열**
```
nodes[0].value=1, nodes[1].value=2, nodes[2].value=3
```

**2단계: swapPairs (i는 0, 2 순회, i+1<3 조건)**
```
i=0: nodes[0].value ↔ nodes[1].value  → nodes[0]=2, nodes[1]=1
i=2: i+1=3 < 3 거짓 → 종료 (nodes[2]는 그대로 3)
```
→ swap 후 value: nodes[0]=**2**, nodes[1]=**1**, nodes[2]=**3**

**3단계: next 연결을 따라 순회 (연결은 안 바뀜)**
```
curr = &nodes[0]  → value=2 출력, next→nodes[2]
curr = &nodes[2]  → value=3 출력, next→nodes[1]
curr = &nodes[1]  → value=1 출력, next→NULL
종료
```

### 연결 순서 시각화

```
출력 경로: nodes[0] → nodes[2] → nodes[1] → NULL
value:        2          3          1
```

### ✅ 정답

```
2 3 1 
```
> ⚠️ 배열 인덱스 순서(2 1 3)가 아니라 **next 연결 순서**로 출력됨에 주의

## Q15. C언어 — 포인터 산술 카운팅(도수분포) + 최빈값

### 문제

```c
#define SIZE 11
int analyze(int* arr, int len) {
    int temp[11] = {0};
    for (int i = 0; i < len; i++)
        (*(temp + *(arr + i)))++;          // temp[arr[i]]++
    int peak = 0;
    for (int i = 0; i < 11; i++)
        if (*(temp + i) > peak) peak = *(temp + i);   // 최대 빈도
    printf("결과 : ");
    for (int i = 0; i < 11; i++)
        if (*(temp + i) == peak) printf("%d ", i);     // 최빈값들
    printf("\n");
    return peak;
}
int main() {
    int source[SIZE] = {1, 2, 3, 2, 2, 3, 3, 3, 1, 4, 2};
    analyze(source, SIZE);
}
```

### 🤔 핵심 — 포인터 산술 해석

`(*(temp + *(arr + i)))++` 은 `temp[arr[i]]++` 와 완전히 동일.
즉 **arr에 등장한 숫자를 인덱스로 하는 카운터(도수분포표)를 만드는 것**.

### 단계별 트레이싱

**1단계: 도수 세기 — source = {1,2,3,2,2,3,3,3,1,4,2}**

| 값(인덱스) | 등장 횟수 |
| ------ | ----- |
| 1      | 2     |
| 2      | 4     |
| 3      | 4     |
| 4      | 1     |
| 나머지    | 0     |

→ temp = [0, 2, 4, 4, 1, 0, ...]

**2단계: peak(최대 빈도) 찾기** → max = **4** (값 2와 값 3이 각각 4번)

**3단계: 빈도가 peak(4)와 같은 인덱스 출력** → 인덱스 **2, 3**

### ✅ 정답

```
결과 : 2 3 
```
> 💡 "최빈값이 여러 개면 모두 출력"이 포인트. 값 2와 3이 공동 최빈(4회).

## Q16. Java — String `==` vs `equals` + `intern()`

### 문제

```java
String[] arr1 = {"apple", "banana", new String("apple"), "Banana".toLowerCase()};
String[] arr2 = new String[4];
arr2[0] = "apple";
arr2[1] = "banana";
arr2[2] = arr1[2].intern();
arr2[3] = new String("banana");
for (int i = 0; i < 4; i++) {
    System.out.print((arr1[i] == arr2[i]) + ",");
    System.out.println((arr1[i].equals(arr2[i])));
}
```

### 🤔 핵심 개념 3가지

**① 문자열 리터럴 `"apple"`은 String Pool에 저장** → 같은 리터럴은 **같은 객체** 참조
**② `new String("apple")`은 Heap에 새 객체 생성** → 리터럴과 **다른 객체**
**③ `.intern()`은 그 문자열과 같은 내용의 Pool 객체를 반환**
**④ `.toLowerCase()`처럼 런타임에 만들어진 문자열은 Pool이 아닌 새 객체**

- `==` → **참조(주소) 비교**
- `.equals()` → **내용 비교** (String은 내용 같으면 항상 true)

### 단계별 트레이싱

| i | arr1[i]                       | arr2[i]                | `==` (참조)              | `equals` (내용) |
| - | ----------------------------- | ---------------------- | --------------------- | ------------- |
| 0 | "apple" (Pool)                | "apple" (Pool)         | 같은 Pool → **true**    | **true**      |
| 1 | "banana" (Pool)               | "banana" (Pool)        | 같은 Pool → **true**    | **true**      |
| 2 | new String("apple") (Heap)    | arr1[2].intern()→Pool  | Heap ≠ Pool → **false** | **true**      |
| 3 | "Banana".toLowerCase() (Heap) | new String("banana")(Heap) | 서로 다른 Heap → **false** | **true**      |

> ⚠️ equals는 4개 모두 true (내용은 다 같음). `==`만 각각 다름.

### ✅ 정답

```
true,true
true,true
false,true
false,true
```

```c
void ins_node(Node** head, int data) {      // 맨 앞 삽입

    if (head == NULL) return;
    Node* prev = head;
    Node* now = head->next;
    while (prev != NULL && now != NULL) {
        prev->next = now->next;   // now 건너뛰기
        free(now);
        prev = prev->next;        // 다음 살아있는 노드로
        if (prev != NULL) now = prev->next;
    }
}
int main() {
    Node* head = NULL;
    for (int i = 1; i <= 5; i++) ins_node(&head, i);   // 앞쪽 삽입
    del_node(head);
    printList(head);
}
```

### 1단계 — 리스트 구성 (앞쪽 삽입이라 역순)

`ins_node`는 **맨 앞에 삽입** → 1,2,3,4,5 순으로 넣으면 리스트는 역순:
```
head → 5 → 4 → 3 → 2 → 1 → NULL
```

### 2단계 — del_node 동작 (한 칸 건너 삭제)

`prev`는 유지, `now`(prev 다음)를 삭제하고 건너뛴다. 즉 **짝수 위치 노드가 삭제**됨.

| 반복 | prev(유지) | now(삭제) | 삭제 후 연결                | 다음 prev | 다음 now |
| --- | -------- | ------- | --------------------- | ------- | ------ |
| 1  | 5        | 4       | 5→3 (4 삭제)            | 3       | 2      |
| 2  | 3        | 2       | 3→1 (2 삭제)            | 1       | NULL   |
| 3  | 1        | NULL    | now=NULL → while 종료   | -       | -      |

→ 남은 리스트: **5 → 3 → 1 → NULL**

### ✅ 정답

```
5 3 1 
```
> ⚠️ 두 함정: (1) 앞쪽 삽입이라 **5→4→3→2→1 역순**, (2) del_node는 **한 칸 걸러 삭제** → 5,3,1만 생존

## Q20. Python — 재귀 분할 + 짝/홀 병합

### 문제

```python
def mystery(lst):
    if len(lst) <= 1: return lst
    mid = len(lst) // 2
    left = mystery(lst[:mid])
    right = mystery(lst[mid:])
    return merge(left, right)
def merge(left, right):
    result = []
    while left and right:
        if (left[0] + right[0]) % 2 == 0:   # 합이 짝수 → left에서
            result.append(left.pop(0))
        else:                                # 합이 홀수 → right에서
            result.append(right.pop(0))
    result.extend(left or right)             # 남은 것 붙이기
    return result
print(mystery([5, 8, 2, 7]))
```

### 🤔 핵심 — 병합 정렬 골격이지만 비교 기준이 "합의 짝/홀"

일반 병합정렬처럼 반씩 쪼갠 뒤 merge하지만, merge 기준이 **`(left[0]+right[0])`가 짝수면 left, 홀수면 right**를 꺼낸다.

### 단계별 트레이싱

**분할**
```
mystery([5,8,2,7])
 mid=2 → left=mystery([5,8]), right=mystery([2,7])
```

**left = mystery([5,8])**
```
mid=1 → left=mystery([5])=[5], right=mystery([8])=[8]
merge([5],[8]): 5+8=13 홀수 → right(8) 꺼냄 → result=[8]
  left=[5] 남음 → extend → [8,5]
→ left = [8, 5]
```

**right = mystery([2,7])**
```
mid=1 → [2], [7]
merge([2],[7]): 2+7=9 홀수 → right(7) 꺼냄 → result=[7]
  left=[2] 남음 → extend → [7,2]
→ right = [7, 2]
```

**최종 merge([8,5], [7,2])**

| left    | right  | left[0]+right[0] | 짝/홀 | 꺼낸 것    | result       |
| ------- | ------ | ---------------- | --- | ------- | ------------ |
| [8,5]   | [7,2]  | 8+7=15           | 홀   | right→7 | [7]          |
| [8,5]   | [2]    | 8+2=10           | 짝   | left→8  | [7,8]        |
| [5]     | [2]    | 5+2=7            | 홀   | right→2 | [7,8,2]      |
| [5]     | []     | while 종료 (right 빔) | -   | -       | -            |
| extend(left=[5]) |  |             |     |         | **[7,8,2,5]** |

### ✅ 정답

```
[7, 8, 2, 5]
```

## 📌 이 회차 핵심 암기 / 자주 틀리는 포인트

1. **Q04 생성자 함정**: 부모 생성자 실행 시점엔 자식 필드가 아직 초기화 전 → `info=null, count=0`. 실행 순서 = super() → 자식 필드 초기화 → 자식 생성자 본문. 게다가 부모 생성자 속 `display()`는 오버라이딩된 **자식** 메서드가 실행됨(동적 바인딩).

2. **Q05 WHERE vs HAVING**: 집계값(MAX-MIN) 조건은 **HAVING**. WHERE는 그룹화 전 개별 행, HAVING은 그룹화 후 집계값 필터.

3. **Q06 서브넷 비트**: 호스트 조건 `2^h-2≥1000`→h=10, 서브넷 조건 `2^s≥200`→s=8. 묻는 건 **서브넷 비트 s=8**.

4. **Q08 bool은 int의 자식**: `isinstance(True,int)`도 True. 분기 순서상 bool이 먼저 걸려 10 반환. 순서가 반대면 답 달라짐.

5. **Q11 CPM**: ES=선행 EF의 최댓값, LS=LF-기간, Slack=LS-ES. F는 후행 없어 LF=종료일(56) → LS=45, Slack=19. 임계경로=Slack 0인 경로=A→B→D→E.

6. **Q13 COUNT는 튜플 1개 반환**: ③은 값이 2여도 "반환 튜플 수"는 1. DISTINCT 유무(①90 vs ②2)도 구분.

7. **Q14/Q19 연결 리스트 순서**: Q14는 value만 swap하고 **next 연결 순서로 출력**(2 3 1). Q19는 **앞쪽 삽입→역순(5→4→3→2→1)** + 한 칸 걸러 삭제 → 5 3 1.

8. **Q16 String == vs equals**: 리터럴/`intern()`은 Pool 공유(==true), `new String()`·`toLowerCase()`는 새 객체(==false). equals는 내용만 보므로 전부 true.

9. **Q20 merge 기준**: 병합정렬 골격이지만 비교가 "두 앞원소 합의 짝/홀". 짝→left, 홀→right. 최종 [7,8,2,5].
