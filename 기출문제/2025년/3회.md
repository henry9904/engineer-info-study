## Q01. 패키지 다이어그램 (Package Diagram)

### 지문 키워드
> **"클래스나 유스케이스 등을 일정 기준에 따라 묶어 표현"**
> **"묶인 단위 간의 의존 관계를 점선 화살표로 나타낼 수 있다"**
> **"복잡한 시스템을 여러 단위로 나누어 구조를 단순화"**

### 왜 이게 답인가?

| 지문 키워드 | 연결 포인트 | 정답 |
|------------|------------|------|
| **클래스/유스케이스를 묶어 표현** | Package = 묶음(패키지) 단위 | 패키지 다이어그램 |
| **점선 화살표**로 의존 관계 표현 | UML 의존(dependency) 관계 표기법 | 패키지 다이어그램 |

### 메커니즘

```
[패키지 다이어그램 예시 — 지문 그림]
┌────────────────┐  <<use>>  ┌────────────────┐
│   사용자         │←- - - - -│     결제         │
│ ┌──────────┐  │           │ ┌──────────┐   │
│ │ 계정 관리  │  │           │ │ 카드 결제  │   │
│ │ 장바구니   │  │           │ │ 예치금 사용│   │
│ └──────────┘  │           │ └──────────┘   │
└────────────────┘           └────────────────┘
```

**UML 다이어그램 종류 분류**

| 분류 | 종류 |
|------|------|
| 구조 다이어그램 | **패키지**, 클래스, 컴포넌트, 배치, 객체 |
| 행위 다이어그램 | 유스케이스, 시퀀스, 활동, 상태 |

> **패키지 vs 컴포넌트 구별 포인트**  
> 패키지: 논리적 묶음, 소스코드 수준  
> 컴포넌트: 물리적 컴포넌트, 실행 파일/DLL 수준

## Q02. 조건 커버리지 (Condition Coverage)

### 지문 키워드
> **"각 조건식이 가질 수 있는 모든 경우의 수를 고려"**
> **"각 조건의 결과가 참(True)과 거짓(False)을 각각 한 번 이상 수행"**
> **"조건의 수가 증가할수록 테스트 케이스의 수가 기하급수적으로 증가"**

### 왜 이게 답인가?

| 지문 키워드 | 연결 포인트 | 정답 |
|------------|------------|------|
| **각 조건식**의 T/F 각 1회 이상 | 각 조건(Condition) 개별 검증 | 조건 커버리지 |
| **기하급수적 증가** 단점 | 조건 n개면 최대 2^n 케이스 필요 | 조건 커버리지 |

### 메커니즘

**테스트 커버리지 비교표**

| 커버리지 | 기준 | 강도 | 특징 |
|----------|------|------|------|
| 구문(Statement) | 모든 문장 1회 실행 | 낮음 | 가장 기본 |
| **분기(Branch/Decision)** | 모든 분기(T/F) 1회 | 중간 | 2회차 Q11 |
| **조건(Condition)** | 각 조건식 T/F 각 1회 | 높음 | **이번 문제** |
| 조건/분기(MC/DC) | 각 조건이 독립적으로 결과에 영향 | 매우 높음 | 항공/의료 필수 |
| 경로(Path) | 모든 실행 경로 | 최고 | 현실적으로 어려움 |

```
예시: if (A && B)
구문커버: A=T,B=T → 1케이스로 충분
분기커버: 전체 조건이 T인 경우 + F인 경우
조건커버: A=T, A=F, B=T, B=F → 각 조건 개별로 T/F 모두
```

> **암기법**: `구분조경` 낮음 → 높음 순서

### ✅ 정답
조건 커버리지 (Condition Coverage)

## Q04. 오류 제어 방식 — Hamming/FEC/BEC/Parity/CRC

### 지문 구조 분석
> **① 코드**: 오류를 검출 + **수정까지** 가능  
> **② 방식**: 수신 측에서 **자체적으로 오류 복구**  
> **③ 방식**: 오류 발생 시 **송신 측에 재전송 요청**  
> **④ 검사**: 각 블록에 **1비트 추가**하여 오류 확인  
> **⑤ 검사**: **생성 다항식**을 이용하여 오류 검출

### 왜 이게 답인가?

| 빈칸 | 지문 핵심 키워드 | 연결 포인트 | 정답 |
|------|----------------|------------|------|
| ① | 오류 검출 + **수정 가능** | 해밍 코드 = 오류 위치까지 파악 | **ㅇ. Hamming** |
| ② | 수신 측 **자체 복구** | **F**orward **E**rror **C**orrection | **ㄴ. FEC** |
| ③ | **재전송 요청** 방식 | **B**ackward **E**rror **C**orrection | **ㄷ. BEC** |
| ④ | **1비트 추가**, 홀/짝 확인 | Parity bit = 1비트 추가 | **ㅁ. Parity** |
| ⑤ | **생성 다항식** 이용 | CRC = 다항식 나눗셈 나머지 | **ㄱ. CRC** |

### 메커니즘

```
[오류 처리 방식 전체 구조]

오류 처리
├── FEC (Forward Error Correction, 전진 오류 수정)
│   └── 수신 측이 자체 복구 → 재전송 불필요
│   └── 예: Hamming 코드 (오류 위치 파악 + 수정)
│
└── BEC (Backward Error Correction, 후진 오류 수정)
    └── 재전송 요청 방식
    └── 오류 검출 기법:
        ├── Parity 검사 (1비트 추가, 단순)
        └── CRC 검사 (다항식, 정밀)
```

**오류 검출/수정 기법 비교**

| 기법 | 검출 | 수정 | 원리 |
|------|------|------|------|
| Parity | ✅ | ❌ | 1비트 추가, 홀/짝 |
| CRC | ✅ | ❌ | 생성 다항식 나머지 |
| **Hamming** | ✅ | ✅ | 패리티 비트 여러 개, 위치 파악 |
| MD5 | 무결성 해시 | ❌ | 128비트 해시 (보기에 있지만 오답) |

### ✅ 정답

## Q05. Java — enum + name() + values()

### 문제
```java
enum Tri {
    AB("A"), A("AB"), C("ABC");
    private String code;
    Tri(String a) { this.code = a; }
    public String getCode() { return code; }
    public void setCode(String code) { this.code = code; }
}
public class Main {
    public static void main(String[] args) {
        Tri a = Tri.values()[Tri.A.name().length()];
        System.out.println(a.getCode());
    }
}
```

### 핵심 개념 — enum 메서드

| 메서드 | 역할 | 예시 |
|--------|------|------|
| `.values()` | 모든 enum 상수를 선언 순서대로 **배열로 반환** | `[AB, A, C]` |
| `.name()` | enum 상수의 **이름(선언명) 문자열** 반환 | `Tri.A.name()` → `"A"` |
| `.ordinal()` | enum 상수의 **순서(0부터) 정수** 반환 | `Tri.A.ordinal()` → `1` |

> ⚠️ **핵심 함정**: `name()`은 **상수의 이름** 문자열 반환 (code 필드값이 아님!)  
> `Tri.A.name()` = `"A"` (상수 이름)  
> `Tri.A.getCode()` = `"AB"` (생성자로 넣은 code 필드값)

### 단계별 트레이싱

```
Tri.values() = [AB(0), A(1), C(2)]

Tri.A.name() = "A"        ← 상수의 선언명 그 자체
"A".length() = 1

Tri.values()[1] = A       ← 인덱스 1 = A 상수

A.getCode() = "AB"        ← A 상수의 code 필드값 (생성자에서 "AB" 넣음)
```

**enum 상수 vs code 필드 구분표**

| 상수 | 상수명(name()) | code 필드(getCode()) |
|------|--------------|---------------------|
| AB | "AB" | "A" |
| A | "A" | "AB" |
| C | "C" | "ABC" |

### ✅ 정답
```
AB
```

## Q06. C언어 — 구조체 포인터 연결 + 비트 XOR

### 문제
```c
struct Node { struct Node* x; unsigned int i; };

int main() {
    struct Node t3 = {0,  5u};
    struct Node t2 = {&t3, 7u};
    struct Node t1 = {&t2, 11u};
    struct Node* p = &t1;
    unsigned int c = 0u;

    while (p) {
        c = c * 3u + p->i;
        p = p->x;
    }
    c = (c ^ 42u) + 100u;
    printf("%u", c);
}
```

### 핵심 개념 — 연결 구조 + XOR 연산

```
연결 구조:
p → t1(i=11) → t2(i=7) → t3(i=5) → NULL
```

> ⚠️ **핵심**: `c = c*3 + p->i` — 누적 계산식

### 단계별 트레이싱

**while 루프**

| p | p->i | c 계산 | c 결과 | p->x |
|---|------|--------|--------|------|
| &t1 | 11 | 0×3 + 11 | **11** | &t2 |
| &t2 | 7 | 11×3 + 7 | **40** | &t3 |
| &t3 | 5 | 40×3 + 5 | **125** | NULL |
| NULL | - | 종료 | - | - |

**비트 XOR 계산: 125 ^ 42**

```
125 = 01111101
 42 = 00101010
XOR = 01010111 = 64+16+4+2+1 = 87
```

```
c = 87 + 100 = 187
```

### ✅ 정답
```
187
```

## Q08. Java — implements 키워드

### 지문 키워드
```java
interface Vehicle { void run(); }
class Car ______(①) Vehicle { ... }
```

### 왜 이게 답인가?

| 상황 | 키워드 |
|------|--------|
| 클래스가 **인터페이스를 구현** | `implements` |
| 클래스가 **클래스를 상속** | `extends` |
| 인터페이스가 **인터페이스를 상속** | `extends` |

### 메커니즘

```java
// 클래스 → 인터페이스
class Car implements Vehicle { ... }   ← implements

// 클래스 → 클래스
class Child extends Parent { ... }     ← extends

// 동시에 사용 가능
class Child extends Parent implements Flyable, Swimmable { ... }
//                   클래스 1개         인터페이스 여러 개
```

> **암기법**: interface에는 **impl**ements (impl = 구현), 클래스엔 extends (확장)

### ✅ 정답
implements

## Q09. Python — enumerate + dict + 튜플 출력

### 문제
```python
data = [[3,2,1,5,4], [2,3,5], [4,4,6,1,3], [7,2]]
arr = {}
for index, var in enumerate(data):
    result_sum = sum(var)
    result_len = len(var)
    arr[index] = (result_sum, result_len)
for i in range(len(arr)):
    print(arr[i][0], arr[i][1], end=' ')
```

### 핵심 개념

- `enumerate(data)` → (인덱스, 요소) 쌍 반환
- `arr[index] = (sum, len)` → 딕셔너리에 **튜플** 저장
- `arr[i][0]` → 튜플의 첫 번째 원소(sum), `arr[i][1]` → 두 번째 원소(len)
- `end=' '` → 줄바꿈 대신 공백으로 구분

### 단계별 트레이싱

| index | var | sum | len | arr[index] |
|-------|-----|-----|-----|-----------|
| 0 | [3,2,1,5,4] | **15** | **5** | (15, 5) |
| 1 | [2,3,5] | **10** | **3** | (10, 3) |
| 2 | [4,4,6,1,3] | **18** | **5** | (18, 5) |
| 3 | [7,2] | **9** | **2** | (9, 2) |

**출력 루프 (end=' '로 공백 구분)**

```
i=0: print(15, 5, end=' ') → "15 5 "
i=1: print(10, 3, end=' ') → "10 3 "
i=2: print(18, 5, end=' ') → "18 5 "
i=3: print(9,  2, end=' ') → "9 2 "
```

### ✅ 정답
```
15 5 10 3 18 5 9 2 
```

## Q10. SQL — CROSS JOIN + LIKE 연산

### 지문 내용

**T1**
| NO | NAME |
|----|------|
| 100 | SMITH |
| 101 | ALLEN |
| 102 | SCOTT |

**T2**
| RNO | RULE |
|-----|------|
| 1 | S% |
| 2 | %T% |

```sql
SELECT COUNT(*) AS CNT
FROM T1 A CROSS JOIN T2 B
WHERE A.NAME LIKE B.RULE;
```

### 단계별 트레이싱

**SQL 실행 순서: FROM → WHERE → SELECT**

**1단계: CROSS JOIN — T1×T2 = 3×2 = 6행**

| A.NAME | B.RULE |
|--------|--------|
| SMITH | S% |
| SMITH | %T% |
| ALLEN | S% |
| ALLEN | %T% |

```c
    struct Test test[] = {{1,"AB"}, {2,"DC"}, {3,"EB"}};

```

> `g + n` → 문자열의 **n번째 인덱스** 위치로 포인터 이동

```
p = &test[1]

p->g + (p->i - 1)
= "DC" + (2 - 1)
= "DC" + 1
= 'C'의 주소 (D를 건너뜀)

printf("%s", 'C'의 주소)
```

```
p->g  →  D  C  \0
          ↑  ↑
         +0 +1

%s 출력 → "C"
```

### 지문 키워드

> **③ "데이터의 소유자가 다른 사용자에게 접근 권한을 부여하거나 회수할 수 있는 방식"**

### 왜 이게 답인가?

| 지문 키워드 | 연결 포인트 | 정답 |
|------------|------------|------|
| **미리 정의된 보안 정책**, 임의 변경 불가 | **M**andatory (강제) = 시스템이 강제 | **① MAC** |
| **역할(Role)** 기반 | **R**ole-**B**ased **A**ccess **C**ontrol | **② RBAC** |
| **소유자**가 권한 부여/회수 | **D**iscretionary (임의적) = 소유자 재량 | **③ DAC** |

### 메커니즘

**접근 제어 3종 비교**

| 방식 | 약자 | 권한 결정자 | 특징 |
|------|------|------------|------|
| **MAC** | Mandatory AC | **시스템** (강제) | 군사/보안 등급 시스템, 사용자 변경 불가 |
| **RBAC** | Role-Based AC | **역할** | 관리자, 일반사용자, 손님 등 역할로 관리 |


> **암기법**: `MAC=강제(시스템), RBAC=역할, DAC=소유자`

```
z = 2 = 0000 0010 (이진수)
z >> 1 = 0000 0001 = 1

z & (z>>1):
  0000 0010  (z=2)
& 0000 0001  (z>>1=1)
-----------
  0000 0000  = 0

z = 0
```

> ⚠️ **핵심 함정**: 2와 1의 비트는 겹치는 자리가 없음 → AND 결과 = 0

```
x > 5 → 7 > 5 → true
z <= 3 → 0 <= 3 → true
true && true → true

z = z * x = 0 * 7 = 0
```

| 단계 | 식 | 결과 |
|------|-----|------|
| 1 | 4%3=1, 1>0 → true → z=2 | **z=2** |

## 📌 이 회차 핵심 암기

### 자주 틀리는 포인트

1. **Q05 enum name()**: `Tri.A.name()` = `"A"` (상수 **선언명** 그대로) ≠ `getCode()` 반환값 `"AB"`. 헷갈리면 name은 Name 그 자체라고 기억
2. **Q06 비트 XOR**: 125^42 계산 시 이진수 변환이 필수. 10진수 암산으로 XOR 계산은 오류 발생 가능 → 반드시 이진수로 변환
3. **Q10 SMITH LIKE '%T%'**: SMITH에 T가 있나? S-M-I-**T**-H → 있음 → 카운트 4개 맞음 (3개로 착각하기 쉬움)
4. **Q18 비트 AND**: `2 & 1` = `10 & 01` = 0. 비트가 겹치지 않으면 항상 0 → 이후 삼항연산자도 자동으로 0
5. **Q20 COUNT(컬럼)**: `COUNT(R2)`는 R2=NULL인 행 제외. WHERE 조건 통과한 5행 중 R2=NULL인 1행 제외 → 4

### 1~3회 전체 비교 — 커버리지 3종

| 회차 | 커버리지 종류 | 핵심 |
|------|------------|------|
| 1회 Q15 | **구문(Statement)** | 모든 문장 1회 실행 |
| 2회 Q11 | **분기(Branch)** | 모든 분기(T/F) 1회 실행 |
| 3회 Q02 | **조건(Condition)** | 각 조건식 T/F 각 1회 실행 |

### 시험 직전 핵심 암기
- **접근제어 3종**: MAC(시스템 강제) / RBAC(역할 기반) / DAC(소유자 재량)
- **오류 제어**: FEC(자체 수정) / BEC(재전송 요청) / Hamming(검출+수정) / Parity(1비트) / CRC(다항식)
- **enum**: `values()`=배열 반환, `name()`=선언명 문자열, `ordinal()`=순서 정수
- **super()**: 자식 생성자에서 부모 생성자 호출, 반드시 첫 줄
- **COUNT(컬럼)**: NULL 제외 카운트 / COUNT(*): 전체 카운트
