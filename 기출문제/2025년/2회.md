## Q02. 서브넷 계산 — /26

### 지문 키워드
- IP 주소: **130.97.16.132**
- 서브넷 마스크: **255.255.255.192**
- ① 네트워크 주소: 130.97.16.( ① )
- ② 사용 가능한 호스트 수 (네트워크/브로드캐스트 제외)

### 개념 — /26 서브넷

**255.255.255.192 = /26**
```
11111111.11111111.11111111.11000000
= 네트워크 비트 26개, 호스트 비트 6개
```

### 단계별 계산

**① 네트워크 주소의 마지막 옥텟**
```
132 = 10000100
192 = 11000000
AND = 10000000 = 128

네트워크 주소 = 130.97.16.128
마지막 옥텟 = 128
```

**② 유효 호스트 수**
```
호스트 비트 = 32 - 26 = 6비트
전체 주소 = 2^6 = 64개
- 네트워크 주소 1개 (130.97.16.128)
- 브로드캐스트 주소 1개 (130.97.16.191)
유효 호스트 = 64 - 2 = 62개
```

> ⚠️ **핵심**: 서브넷 마스크 끝 옥텟 192의 의미 — 64씩 블록이 나뉨 (0, 64, 128, 192...)

**서브넷 범위 정리**
```
블록 크기: 256-192 = 64
범위: 130.97.16.128 ~ 130.97.16.191
  네트워크: 130.97.16.128
  유효 호스트: 130.97.16.129 ~ 130.97.16.190
  브로드캐스트: 130.97.16.191
```

### ✅ 정답
① 128  
② 62

## Q05. Java — 람다 + FunctionalInterface + 예외처리

### 문제
```java
@FunctionalInterface
interface F {
    int app(int x) throws Exception;
}

class Main {
    public static int run(F f) {
        try {
            return f.app(3);
        } catch (Exception e) {
            return 7;
        }
    }

    public static void main(String[] args) {
        F f = (x) -> {
            if (x > 2) {
                throw new Exception("x가 2보다 큽니다.");
            }
            return x * 2;
        };

        System.out.print(run(f) + run((int n) -> n + 9));
    }
}
```

### 핵심 개념 — FunctionalInterface + 람다식

**@FunctionalInterface**: 추상 메서드가 **정확히 1개**인 인터페이스. 람다식으로 구현 가능.

> ⚠️ **핵심 함정**: `run(f)`와 `run((int n)->n+9)` — 두 호출은 **서로 다른 람다**를 전달함

### 단계별 트레이싱

**run(f) 분석**
```
f = (x) -> {
    if (x > 2) throw new Exception;
    return x * 2;
}

run(f):
  f.app(3) 호출 → x=3
  3 > 2? ✅ → throw new Exception
  catch(Exception e) → return 7
```

**run((int n) -> n + 9) 분석**
```
람다 = (int n) -> n + 9

run(람다):
  f.app(3) 호출 → n=3
  예외 없음
  return 3 + 9 = 12
```

**최종 계산**
```
System.out.print(run(f) + run((int n) -> n + 9))
= System.out.print(7 + 12)
= System.out.print(19)
```

| 호출 | 람다 내용 | x=3 결과 | 반환값 |
|------|---------|---------|--------|
| run(f) | x>2이면 예외 | 예외 발생 → catch | **7** |
| run((int n)->n+9) | n+9 반환 | 3+9=12 | **12** |

### ✅ 정답
```
19
```

## Q06. CPU 스케줄링 — SJF / SRT

### 지문 키워드
> **① "프로세스의 전체 실행 시간(버스트 타입)을 기준으로 CPU를 할당하는 비선점형(Non-preemptive) 스케줄링 알고리즘으로, CPU 사용 시간이 짧은 프로세스에게 우선순위를 주는 방식"**
> **② "( ① )의 개념을 확장한 선점형(Preemptive) 스케줄링 알고리즘으로, 다음 CPU 버스트 시간(next CPU burst time)이 짧은 프로세스가 CPU를 선점할 수 있다"**

### 왜 이게 답인가?

| 지문 키워드 | 연결 포인트 | 정답 |
|------------|------------|------|
| **전체 실행 시간 기준**, **비선점형**, CPU 시간 짧은 것 우선 | Shortest Job First | **① SJF** |
| **① 의 확장**, **선점형**, 남은 시간이 짧으면 선점 | Shortest Remaining Time | **② SRT** |

### 메커니즘

**CPU 스케줄링 알고리즘 비교**

| 알고리즘 | 방식 | 기준 | 특징 |
|----------|------|------|------|
| FCFS | 비선점 | 도착 순서 | 단순, 호위 효과 |
| **SJF** | **비선점** | **전체 실행 시간** | 최적 평균 대기, 기아 발생 |
| **SRT** | **선점** | **남은 실행 시간** | SJF 선점 버전 |
| Round Robin | 선점 | 시간 할당량 | 공평, 시분할 |
| Priority | 선/비선점 | 우선순위 | 기아 발생 가능 |

> **암기법**: SJF=비선점, SRT=선점 (S**J**F → J는 Job 전체 / S**R**T → R은 Remaining 남은 것)

### ✅ 정답
① SJF (Shortest Job First)  
② SRT (Shortest Remaining Time)

|------------|------------|------|

[Proxy 패턴 구조]

## Q08. Java — String 참조 vs 배열 참조

### 문제
```java
class Main {
    public static void change(String[] data, String s) {
        data[0] = s;
        s = "Z";
    }
    public static void main(String[] args) {
        String[] data = { "A" };
        String s = "B";
        change(data, s);
        System.out.println(data[0] + s);
    }
}
```

### 핵심 개념 — Java 매개변수 전달 방식

> ⚠️ **핵심 함정 1**: Java는 **항상 값 전달(Pass by Value)**  
> - 기본형: 값 복사 → 원본 불변  
> - 참조형: **참조(주소) 값** 복사 → **배열 내부는 변경 가능, 참조 자체 교체는 불가**

> ⚠️ **핵심 함정 2**: `String[]`과 `String`의 동작이 다름!

### 단계별 트레이싱

**change() 호출 전**
```
[main 스택]
data ──▶ ["A"]   (배열 객체를 가리키는 참조)
s    ──▶ "B"     (String 객체를 가리키는 참조)
```

**change(data, s) 실행**
```
[change 스택]
data ──▶ ["A"]   (main의 data와 같은 배열 참조 복사)
s    ──▶ "B"     (main의 s와 같은 String 참조 복사)

1. data[0] = s;
   → 배열 내부 수정 → ["B"]    ✅ main의 data도 영향받음!

2. s = "Z";
   → change의 지역변수 s만 "Z"로 교체
   → main의 s는 여전히 "B"    ❌ main s 영향 없음
```

**change() 실행 후 main 상태**
```
data[0] = "B"  (배열 내부가 변경됨)
s       = "B"  (로컬 참조 교체는 main에 영향 없음)
```

**출력**: `data[0] + s = "B" + "B" = "BB"`

### ✅ 정답
```
BB
```

**ref.id()**
- `id()`는 **static 메서드** → 정적 바인딩

- return **"P"**

```
System.out.print(5 + "P") = "5P"
```

> ⚠️ **1회 Q13과 비교**: 1회는 인스턴스 메서드의 동적 바인딩이 핵심

## Q11. 분기 커버리지 — 테스트 케이스 경로

### 지문 내용 (제어 흐름도)
```
① POINTER = FALSE
     ↓
② X > Y?  ── YES ──▶ ③ POINTER = TRUE ──┐
     │ NO                                  │
     ▼  ◀──────────────────────────────────┘
④ X = X + 1
     ↓
⑤ CALL SUB
     ↓
⑥ X > Y?  ── NO ──▶ ⑦ PRINT RESULT
     │ YES
     ▼
  (①로 돌아감)
```

### 핵심 개념 — 분기 커버리지 (Branch Coverage)

**분기 커버리지**: 모든 분기(TRUE/FALSE)를 **최소 1번씩** 실행

> 이 그래프에서 분기점: ②(X>Y? YES/NO)와 ⑥(X>Y? YES/NO) → 총 4개 분기

### 분기 커버리지 두 경로

**모든 분기(T/F)를 커버하려면 최소 2개 경로 필요**

```
경로① (7단계) — ②YES, ⑥YES 커버:
①→②→③→④→⑤→⑥→①
  ↑YES     ↑YES(루프 재시작)

경로② (6단계) — ②NO, ⑥NO 커버:
①→②→④→⑤→⑥→⑦
  ↑NO     ↑NO(종료)
```

**분기 커버리지 분석**

| 경로 | ②분기 | ⑥분기 | 커버 여부 |
|------|-------|-------|----------|
| 경로① | YES | YES | ②YES✅ ⑥YES✅ |
| 경로② | NO | NO | ②NO✅ ⑥NO✅ |
| 합계 | YES+NO | YES+NO | 모든 분기 커버 ✅ |

### ✅ 정답
경로① (7단계): **① → ② → ③ → ④ → ⑤ → ⑥ → ①**  
경로② (6단계): **① → ② → ④ → ⑤ → ⑥ → ⑦**

## Q12. C언어 — 원형 큐 (Circular Queue)

### 문제
```c
#define SIZE 3
typedef struct {
    int data[SIZE];
    int front, rear, count;
} Qu;

void enq(Qu* q, int value) {
    if (q->count == SIZE) return;
    q->data[q->rear] = value;
    q->rear = (q->rear + 1) % SIZE;
    q->count++;
}
int deq(Qu* q) {
    if (q->count == 0) return -1;
    int value = q->data[q->front];
    q->front = (q->front + 1) % SIZE;
    q->count--;
    return value;
}
int main() {
    Qu q = { {0}, 0, 0, 0 };
    enq(&q, 1); enq(&q, 2); deq(&q); enq(&q, 3);
    printf("%d 그리고 %d\n", deq(&q), deq(&q));
}
```

### 핵심 개념 — 원형 큐

```
원형 큐 (SIZE=3):
인덱스: 0, 1, 2
rear=(rear+1)%SIZE 로 인덱스 순환
front=(front+1)%SIZE 로 앞에서 꺼냄
```

> ⚠️ **핵심 함정**: `printf(deq(&q), deq(&q))` → C언어에서 **함수 인자 평가는 오른쪽→왼쪽 순서**  
> 오른쪽 `deq`가 먼저 실행됨!

### 단계별 트레이싱

**초기 상태**: data=[0,0,0], front=0, rear=0, count=0

| 연산 | 실행 내용 | data[] | front | rear | count |
|------|---------|--------|-------|------|-------|
| 초기 | - | [0,0,0] | 0 | 0 | 0 |
| enq(1) | data[0]=1, rear→1 | [1,0,0] | 0 | 1 | 1 |
| enq(2) | data[1]=2, rear→2 | [1,2,0] | 0 | 2 | 2 |
| deq() | value=data[0]=1, front→1 | [1,2,0] | **1** | 2 | **1** |
| enq(3) | data[2]=3, rear→0 | [1,2,3] | 1 | **0** | **2** |

**최종 상태**: data=[1,2,3], front=1, rear=0, count=2  
큐 내용(front=1부터): **2 → 3**

**printf 인자 평가 (오른쪽→왼쪽)**

| 순서 | deq() | data[] | front | 반환값 |
|------|-------|--------|-------|--------|
| 1st (오른쪽) | data[front=1]=2, front→2, count=1 | [1,2,3] | 2 | **2** |
| 2nd (왼쪽) | data[front=2]=3, front→0, count=0 | [1,2,3] | 0 | **3** |

```
printf("%d 그리고 %d\n", 3, 2)
→ "3 그리고 2"
```

### ✅ 정답
```
3 그리고 2
```

## Q15. Java — 객체 배열 참조 스왑

### 문제
```java
class BO {
    int v;
    BO(int v) { this.v = v; }
}
class Main {
    public static void main(String[] args) {
        BO a = new BO(1);
        BO b = new BO(2);
        BO c = new BO(3);
        BO[] arr = {a, b, c};

        BO t = arr[2];      // t = c 참조
        arr[2] = arr[0];    // arr[2] = a 참조
        arr[0] = t;         // arr[0] = c 참조
        arr[1].v = arr[0].v; // arr[1](=b).v = arr[0](=c).v = 3

        System.out.print(a.v + "a" + b.v + "b" + c.v);
    }
}
```

### 핵심 개념 — 객체 배열은 참조 배열

> ⚠️ **핵심 함정**: Java 배열은 **참조(주소)를 저장**. 배열 원소를 바꾸는 것은 참조 교환이지, 객체 내부 변경이 아님!

### 단계별 트레이싱

**초기 상태**
```
a ──▶ BO{v=1}
b ──▶ BO{v=2}
c ──▶ BO{v=3}
arr = [a참조, b참조, c참조]
```

**스왑 실행**
```
1. BO t = arr[2];
   t ──▶ BO{v=3}  (c와 같은 객체)

2. arr[2] = arr[0];
   arr = [a참조, b참조, a참조]

3. arr[0] = t;
   arr = [c참조, b참조, a참조]
```

**arr[1].v = arr[0].v**
```
arr[0] = c참조 → arr[0].v = c.v = 3
arr[1] = b참조 → arr[1].v = b.v = 3 으로 변경!
   → b.v가 3으로 바뀜 (b 객체 내부 변경)
```

**최종 객체 상태 정리**

| 변수 | 가리키는 객체 | .v 값 |
|------|------------|-------|
| a | BO{v=1} | **1** (변경 없음) |
| b | BO{v=3} | **3** (arr[1].v=arr[0].v로 변경됨) |
| c | BO{v=3} | **3** (변경 없음) |

**출력**: `a.v + "a" + b.v + "b" + c.v = "1a3b3"`

### ✅ 정답
```
1a3b3

## Q18. C언어 — 문자열 역방향 연결리스트

### 문제

typedef struct N {
    char c;
    struct N *p;
} N;

N* a(char *s) {
    N *h = NULL;
    while (*s) {
        N *n = malloc(sizeof(N));
        n->c = *s++;   // 현재 문자 저장, s 다음 문자로 이동

    }

}

> ⚠️ **핵심**: 각 문자를 **맨 앞에 삽입**하므로 결과적으로 **역순** 연결리스트가 만들어짐

| 문자 | n->c | n->p | h (head) | 리스트 상태 |

| 'E' | 'E' | B | E | E→B→NULL |

```
q=T → putchar('T'), q=S

> **"TCP의 3-way 핸드셰이크를 악용하여 다수의 SYN 패킷을 목표 서버로 전송"**
> **"서버가 보낸 SYN-ACK에 대해 ACK를 보내지 않거나 발신지를 위조하여 응답을 차단"**
> **"서버의 연결 대기 큐를 포화시켜 정상적인 접속을 방해하는 서비스 거부(DoS) 공격"**

### 왜 이게 답인가?

| 지문 키워드 | 연결 포인트 | 정답 |
|------------|------------|------|
| **TCP 3-way 핸드셰이크 악용** | SYN 패킷으로 연결 시작 단계 공격 | SYN 플러딩 |
| **SYN 패킷 대량 전송, ACK 안 보냄** | Half-open 연결로 자원 고갈 | SYN 플러딩 |
| **연결 대기 큐 포화** | Backlog 큐 고갈 = DoS | SYN 플러딩 |

### 메커니즘

**정상 3-way 핸드셰이크**
```
클라이언트 ── SYN ──▶ 서버 (큐에 등록)
클라이언트 ◀── SYN-ACK ── 서버
클라이언트 ── ACK ──▶ 서버 (연결 완료, 큐에서 제거)
```

**SYN 플러딩 공격**
```
공격자 ── SYN×1000 (위조 IP) ──▶ 서버 (큐가 꽉 참)
서버 ◀── SYN-ACK 응답하지만 ── 위조 IP로 날아감
ACK가 오지 않음 → 큐 계속 점유

→ 정상 사용자는 큐가 꽉 차서 접속 불가
```

**DoS 공격 비교**

| 공격 | 방식 | 핵심 |
|------|------|------|
| **SYN 플러딩** | TCP 3-way handshake 악용 | 연결 대기 큐 포화 |
| 핑 오브 데스 | 비정상 크기 ICMP 패킷 | 시스템 충돌 |
| 스머프 공격 | 브로드캐스트 ICMP 증폭 | 트래픽 폭주 |
| HTTP Flood | 정상 HTTP 요청 대량 전송 | 웹서버 과부하 |

### ✅ 정답
ㄷ. SYN 플러딩 (SYN Flooding)

## 📌 이 회차 핵심 암기

### 자주 틀리는 포인트

1. **Q10 static 메서드**: 인스턴스 메서드는 실제 객체 타입 기준(동적), **static은 선언 타입 기준(정적)** — 1회 Q13과 반드시 세트로 암기
2. **Q08 String 참조**: `String[] data`는 배열 참조 → 내부 변경 가능 / `String s`는 참조 자체 교체 불가 → main의 s 불변
3. **Q12 원형 큐**: printf 인자는 **오른쪽→왼쪽** 평가 → deq 순서 거꾸로
4. **Q15 객체 배열**: 배열 원소 교환은 **참조 교환** → 원본 객체(a,b,c)의 v가 변하는 건 `arr[1].v = arr[0].v` 줄만
5. **Q17 dict/set 독립성**: dict로 set 만든 뒤 dict 변경해도 set은 **독립적으로 유지됨**

### 1회 vs 2회 핵심 비교

| 개념 | 1회 | 2회 |
|------|-----|-----|
| 동적바인딩 | Q13: static 공유 + 동적바인딩 | Q10: static은 선언타입 기준 |
| 커버리지 | Q15: 구문(Statement) 커버리지 | Q11: 분기(Branch) 커버리지 |
| 연결리스트 | Q18: reconnect (앞으로 이동) | Q18: 문자열 역삽입 |
| 서브넷 | Q09: /22 브로드캐스트 수신 IP | Q02: /26 호스트 수 계산 |

### 시험 직전 공식 정리
- **라운드 로빈 대기시간** = 완료시간 - 도착시간 - 실행시간
- **서브넷 호스트 수** = 2^(호스트비트) - 2
- **분기 커버리지** = 모든 T/F 분기 1회 이상 실행
- **SJF**(비선점) vs **SRT**(선점): Remaining(남은)이 들어가면 SRT
