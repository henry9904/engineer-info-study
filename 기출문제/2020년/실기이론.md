## 비정규화 (Denormalization)

> 1회 Q01

### 지문 키워드
> **"데이터베이스 정규화 이후"**
> **"성능향상과 개발 편의성 등 정규화 기법에 위배되는 수행 기법"**

### 왜 이게 답인가?

| 지문 키워드 | 연결 포인트 | 정답 |
|------------|------------|------|
| **정규화 이후** + **정규화 기법에 위배** | 정규화를 되돌리는 것 = 비(반)정규화 | 비정규화 |
| **성능향상** 목적 | 조인 줄임 → 조회 속도 향상 | 비정규화 |

### 메커니즘

**정규화와 비정규화의 트레이드오프**

```
[정규화 후 상태]
주문 테이블     고객 테이블      상품 테이블
┌────────┐    ┌──────────┐    ┌────────┐
│주문ID   │    │고객ID     │    │상품ID   │
│고객ID(FK)│──▶│고객명     │    │상품명   │
│상품ID(FK)│   │주소       │  ▶│가격     │
│수량     │   └──────────┘    └────────┘
└────────┘
→ 조회 시 3개 테이블 JOIN 필요 → 느림

[비정규화 후 상태]
주문 테이블 (중복 허용)
┌─────────────────────────────┐
│주문ID │고객ID │고객명 │상품명 │가격 │
└─────────────────────────────┘
→ 조회 시 1개 테이블만 접근 → 빠름
→ 단점: 고객명 변경 시 모든 행 갱신 필요
```

**정규화 vs 비정규화 비교**

| 항목 | 정규화 | 비정규화 |
|------|--------|---------|
| 목적 | 중복 제거, 이상 방지 | 조회 성능 향상 |
| 테이블 수 | 많아짐 | 줄어듦 |
| JOIN | 많음 | 적음 |
| 데이터 중복 | 없음 | 의도적 허용 |
| 저장 공간 | 적음 | 많음 |
| 무결성 | 높음 | 낮음 |

> ⚠️ **핵심**: 비정규화는 "잘못된" 설계가 아님. 성능을 위해 **의도적으로** 정규화를 포기하는 것

### ✅ 정답
데이터베이스 정규화 이후, 성능향상과 개발 편의성 등 정규화 기법에 위배되는 수행 기법

## XML / JSON / SOAP — 마크업·데이터 교환 언어

> 1회 Q02, 1회 Q06, 2회 Q03

### 지문 키워드 (1회 Q02 — XML)
> **"SGML의 단순화된 부분집합"**
> **"다른 종류의 시스템끼리 데이터를 쉽게 주고받아 HTML의 한계를 극복"**
> **"텍스트 데이터 형식으로 유니코드를 사용해 전 세계 언어를 지원"**

### 지문 키워드 (1회 Q06 — JSON)
> **"Javascript 객체 문법으로 구조화된 데이터를 표현하기 위한 문자 기반의 표준 포맷"**
> **"XML을 대체해서 데이터 전송에 많이 사용"**
> **"데이터는 이름과 값의 쌍으로 이루어진다"**

### 지문 키워드 (2회 Q03 — SOAP)
> **"HTTP, HTTPS, SMTP 등을 통해 XML 기반의 메시지를 교환하는 프로토콜"**
> **"RPC(Remote Procedure Call) 패턴으로 서버에 메시지를 요청"**

### 왜 이게 답인가?

| 지문 키워드 | 연결 포인트 | 정답 |
|------------|------------|------|
| **SGML의 단순화**, HTML 한계 극복 | 확장 가능한 마크업, 태그 자유 정의 | **XML** |
| **Javascript 객체 문법**, 이름-값 쌍 | JSON = JavaScript Object Notation | **JSON** |
| **XML 기반 메시지**, HTTP/SMTP 전송, RPC | XML 봉투 싸서 전송하는 웹서비스 프로토콜 | **SOAP** |

### 메커니즘

**마크업/데이터 언어 계보**

```
SGML (1986) ← 원조, 복잡하고 무거움
  ├── HTML  ← 웹 표시용, 태그 고정 (p, div, table...)
  └── XML   ← 데이터 교환용, 태그 자유 정의 (<상품>, <가격>...)
               └── JSON ← XML보다 더 경량, REST API 표준
               └── SOAP ← XML을 HTTP로 포장해서 전송하는 프로토콜
```

**실제 표현 비교**

```xml
<!-- XML -->
<person>
  <name>홍길동</name>
  <age>25</age>
</person>
```

```json
// JSON (같은 데이터)
{"name": "홍길동", "age": 25}
```

**XML vs JSON vs SOAP 비교표**

| 항목 | XML | JSON | SOAP |
|------|-----|------|------|
| 형태 | 태그 기반 | 키-값 쌍 | XML + HTTP 봉투 |
| 용량 | 보통 | 가벼움 | 무거움 |
| 주 용도 | 설정파일, 문서 | REST API | 웹서비스 (금융/의료) |
| 파싱 | 복잡 | 단순 | 복잡 |
| 표준 | W3C | RFC | W3C |

> ⚠️ **SOAP는 XML이 아님** — SOAP는 **XML을 메시지 형식으로 사용하는 프로토콜**
> XML과 SOAP의 관계: XML ⊂ SOAP (SOAP 메시지 안에 XML이 들어있음)

### ✅ 정답
1회 Q02: XML  
1회 Q06: JSON  
2회 Q03: SOAP

## 살충제 패러독스 (Pesticide Paradox)

> 1회 Q03

### 지문 키워드
> **"동일한 테스트 케이스로 동일한 절차를 반복 수행하면 새로운 결함을 찾을 수 없다"**

### 왜 이게 답인가?

| 지문 키워드 | 연결 포인트 | 정답 |
|------------|------------|------|
| **동일한 테스트 케이스 반복** | Pesticide(살충제) = 같은 것만 반복 | 살충제 패러독스 |
| **새로운 결함 발견 불가** | 테스트 케이스에 내성 생김 | 살충제 패러독스 |

### 메커니즘

**왜 "살충제"인가?**

```
현실 살충제:  같은 약만 계속 뿌리면 → 해충이 내성 생김 → 효과 없어짐
테스트 케이스: 같은 테스트만 계속 실행 → 소프트웨어가 그 테스트 통과에 최적화
              → 아직 발견 못한 새 버그는 그 테스트로 절대 못 찾음

해결책: 정기적으로 테스트 케이스를 갱신·다양화
```

**소프트웨어 테스트 7원칙**

| 번호 | 원칙 | 핵심 키워드 |
|------|------|------------|
| 1 | 결함 존재 | 결함이 있음을 보여줄 뿐, 없음을 증명 못함 |
| 2 | 완벽 테스트 불가 | 무한한 입력 → 전부 테스트 불가 |
| 3 | 초기 집중 | 결함 80% = 모듈 20%에 집중 (결함 집중) |
| **4** | **살충제 패러독스** | **반복 테스트 → 새 결함 발견 불가** |
| 5 | 정황 의존성 | 소프트웨어 특성에 따라 테스트 방법 다름 |
| 6 | 오류 부재의 오해 | 버그 없어도 요구 미충족이면 의미 없음 |
| 7 | 결함 집중 | 파레토 법칙 — 특정 모듈에 결함 밀집 |

### ✅ 정답
동일한 테스트 케이스로 동일한 절차를 반복 수행하면 새로운 결함을 찾을 수 없다.

## 결합도 / 응집도 (Coupling / Cohesion)

> 1회 Q04

### 지문 키워드
> **① "모듈과 모듈간의 상호 의존 정도"**
> **② "모듈 내부의 기능적인 집중 정도"**
> **"모듈의 독립성을 높이기 위해서는 ①를 낮추고 ②를 높여야 한다"**

### 왜 이게 답인가?

| 지문 키워드 | 연결 포인트 | 정답 |
|------------|------------|------|
| **모듈 간** 상호 의존 정도 | Coupling = 모듈끼리 얼마나 묶여있나 | **① 결합도 (Coupling)** |
| **모듈 내부** 기능 집중 정도 | Cohesion = 모듈 안이 얼마나 뭉쳐있나 | **② 응집도 (Cohesion)** |

### 메커니즘

**좋은 모듈 설계 원칙**

```
독립적인 모듈 = 결합도 ↓ (낮게) + 응집도 ↑ (높게)

[나쁜 설계]
모듈A ─────── 내부변수 직접접근 ───▶ 모듈B
(내용 결합 = 최악)

[좋은 설계]
모듈A ─── 단순 값만 전달 ───▶ 모듈B
(자료 결합 = 가장 느슨)
```

**결합도 6단계 (강한 → 약한)**

| 종류 | 핵심 키워드 | 예시 |
|------|------------|------|
| **내용(Content)** | 다른 모듈 **내부 변수 직접 접근** | 모듈A가 모듈B의 전역변수를 직접 수정 |
| **공통(Common)** | **전역 변수 공유** | 두 모듈이 같은 전역변수를 사용 |
| 외부(External) | 외부 시스템/포맷 공유 | 같은 외부 파일 포맷 참조 |
| 제어(Control) | **플래그/제어값** 전달 | `flag=true` 넘겨서 실행 흐름 제어 |
| 스탬프(Stamp) | **배열/객체/구조체** 전달 | Student 객체 통째로 전달 |
| **자료(Data)** | **단순 값만** 전달 | 정수, 문자열 하나만 전달 |

**응집도 7단계 (높은 → 낮은)**

| 종류 | 핵심 키워드 |
|------|------------|
| **기능(Functional)** | 단 하나의 기능만 수행 |
| **순차(Sequential)** | 앞 작업의 출력이 다음 작업의 입력 |
| **통신(Communication)** | 같은 데이터에 대해 다른 작업 |
| **절차(Procedural)** | 순서는 있지만 데이터 연관 없음 |
| **시간(Temporal)** | 같은 시간대에 실행 (초기화 등) |
| **논리(Logical)** | 비슷한 종류의 기능을 if로 분기 |
| **우연(Coincidental)** | 관련 없는 기능이 묶임 |

> **암기법**:  
> 결합도 강한 순 — `내공외제스자`  
> 응집도 높은 순 — `기순통절시논우`

### ✅ 정답
① 결합도 (Coupling)  
② 응집도 (Cohesion)

## Fan-in / Fan-out

> 1회 Q05

### 지문 내용
```
          A
        / | \
       B  C  D
        \/ \/
        E   F
       / \
      G   H
```
**"Fan-in의 수가 2 이상인 모듈을 쓰시오"**

### 핵심 개념

| 용어 | 의미 | 읽는 방향 |
|------|------|----------|
| **Fan-in** | 해당 모듈을 **호출하는** 모듈 수 | **위**에서 들어오는 화살표 수 |
| **Fan-out** | 해당 모듈이 **호출하는** 모듈 수 | **아래**로 나가는 화살표 수 |

> ⚠️ **암기 포인트**: Fan-in = **"나를 부르는 놈 수"**, Fan-out = **"내가 부르는 놈 수"**

### 풀이

각 모듈의 Fan-in 계산:

| 모듈 | 나를 호출하는 모듈 | Fan-in |
|------|-----------------|--------|
| A | 없음 | 0 |
| B | A → 1개 | 1 |
| C | A → 1개 | 1 |
| D | A → 1개 | 1 |
| E | B, C → 2개 | **2 ✅** |
| F | C, D → 2개 | **2 ✅** |
| G | E → 1개 | 1 |
| H | E → 1개 | 1 |

Fan-in ≥ 2인 모듈 = **E, F**

> ⚠️ **정답지 확인**: 1회 Q05 정답은 **F, H**로 표기되어 있으나, 다이어그램 해석에 따라 F와 H 또는 E와 F가 될 수 있음. 실제 다이어그램의 연결 관계를 정확히 보고 판단할 것

### ✅ 정답
F, H (정답지 기준)

## 성능 측정 지표 3종

> 1회 Q07

### 지문 키워드
> **① "일정 시간 내에 애플리케이션이 처리하는 일의 양"**
> **② "요청을 전달한 시간부터 응답이 도착할 때까지 걸린 시간"**
> **③ "요청을 전달한 시간부터 처리가 완료될 때까지 걸린 시간"**

### 왜 이게 답인가?

| 지문 키워드 | 연결 포인트 | 정답 |
|------------|------------|------|
| **처리하는 일의 양** | Through = 통과, put = 양 | **① 처리량 (Throughput)** |
| **응답이 도착**할 때까지 | Response = 응답, Time = 시간 | **② 응답 시간 (Response Time)** |
| **처리가 완료**될 때까지 | Turn Around = 한 바퀴 돌아오는 | **③ 경과 시간 (Turn Around Time)** |

### 메커니즘

**타임라인으로 이해하기**

```
클라이언트                               서버
   │                                      │
   ├── 요청 전송 ──────────────────────▶  │
   │   (이 시점부터 계산 시작)              │
   │                                      │ ← 처리 시작
   │◀── 응답 첫 바이트 도착 ───────────── │
   │    ↑ 응답 시간 (Response Time) 끝    │ ← 계속 처리 중
   │                                      │
   │◀── 처리 완전 완료 ─────────────────  │
        ↑ 경과 시간 (Turn Around Time) 끝
```

> ⚠️ **응답시간 vs 경과시간 차이**  
> 응답시간 = "첫 응답이 도착"까지만 (웹에서 첫 화면 뜨는 시간)  
> 경과시간 = "완전히 다 처리 완료"까지 (전체 파일 다운로드 완료 시간)

**추가 지표: 자원 사용률 (Resource Usage)**

| 지표 | 설명 |
|------|------|
| Throughput | 단위시간당 처리량 |
| Response Time | 요청→응답 도착 |
| Turn Around Time | 요청→처리 완료 |
| **Resource Usage** | CPU, 메모리, 네트워크 사용률 |

### ✅ 정답
① 처리량 (Throughput)  
② 응답 시간 (Response Time)  
③ 경과 시간 (Turn Around Time)

## 살충제 패러독스 관련 — 블랙박스 / 동등분할 테스트

> 3회 Q05, 4회 Q04

### 지문 키워드 (3회 Q05 — 블랙박스 테스트)
> **"소프트웨어의 기능이 완전히 작동하는 것을 입증"**
> **"동치분할/경계값 분석 이용"**

### 지문 키워드 (4회 Q04 — 동등분할 테스트)
> **"입력값을 넣고, 출력값을 비교하는 테스트 기법"** (테이블로 구간 나눔)

### 핵심 개념

**테스트 기법 분류**

```
테스트 기법
├── 블랙박스 테스트 (Black Box)
│   ← 소프트웨어 내부 구조를 모른 채로 입력-출력만 확인
│   ← 동등분할, 경계값 분석, 결정 테이블, 상태 전이...
│
└── 화이트박스 테스트 (White Box)
    ← 내부 코드/로직을 보면서 확인
    ← 구문/분기/조건 커버리지...
```

**동등분할 (Equivalence Partitioning)**

```
입력값 구간을 나눠서 각 구간 대표값만 테스트

예: 학점 환산 (0~59=F, 60~79=C, 80~89=B, 90~100=A)
테스트 케이스 1: 45 (0~59 구간 대표값)
테스트 케이스 2: 70 (60~79 구간 대표값)
테스트 케이스 3: 85 (80~89 구간 대표값)
테스트 케이스 4: 95 (90~100 구간 대표값)
→ 4개 케이스로 전체 구간 커버
```

> ⚠️ **동등분할 vs 경계값 분석**  
> 동등분할: 구간 **중간값** 테스트  
> 경계값 분석: 구간 **경계** 테스트 (59, 60, 79, 80...)

## Land Attack / DoS 공격

> 1회 Q09

### 지문 키워드
> **"출발지 IP와 목적지 IP가 같은 패킷을 만들어 보내는 공격 방법"**
> **"수신자가 응답을 보낼 때, 목적지 주소가 자기 자신이므로 SYN 신호가 계속 자신의 서버로 돌아와 서버의 자원을 고갈시켜 가용성을 파괴"**

### 왜 이게 답인가?

| 지문 키워드 | 연결 포인트 | 정답 |
|------------|------------|------|
| **출발지 IP = 목적지 IP** | 자기 자신에게 패킷 → Land(착지) | **Land Attack** |
| **SYN이 자신에게 돌아와** 자원 고갈 | 무한 루프로 자원 소진 | **Land Attack** |

### 메커니즘

```
[정상 통신]
공격자 IP: 1.2.3.4  ──── SYN ────▶ 서버 IP: 5.6.7.8
서버 IP: 5.6.7.8    ◀── SYN-ACK ─ 서버 IP: 5.6.7.8 (자신에게 응답)

[Land Attack]
출발지: 5.6.7.8  ──── SYN ────▶ 목적지: 5.6.7.8
(서버 자신이 자신에게 패킷 보냄)
→ SYN-ACK를 자신에게 보냄
→ 또 SYN 생성
→ 무한 루프 → CPU/메모리 고갈
```

**DoS/DDoS 공격 종류 비교**

| 공격명 | 핵심 방법 | 목적 |
|--------|----------|------|
| **Land Attack** | 출발지=목적지 IP 위조 | 자기 루프 → 자원 고갈 |
| **SYN Flooding** | 대량 SYN + ACK 안 보냄 | 연결 대기 큐 고갈 |
| **Smurf** | 브로드캐스트 ICMP 증폭 | 대량 트래픽 |
| **Ping of Death** | 비정상 대용량 ICMP | 시스템 충돌 |
| **Teardrop** | 조각 패킷 재조합 오류 유발 | 시스템 충돌 |

> ⚠️ **Land Attack 핵심**: "출발지 = 목적지" 이 한 문장이 정답 키워드

### ✅ 정답
Land Attack

> **"IP 패킷 전송 중 에러 발생 시 에러 원인을 알려주거나 네트워크 상태를 진단해주는 기능"**

### 왜 이게 답인가?

| 지문 키워드 | 연결 포인트 | 정답 |
|------------|------------|------|
| **IP 계층**, IP 패킷 **암호화** | Internet Protocol **Sec**urity | **IPSec** |
| **다익스트라**, 링크 상태 변화 시에만 전송 | Open Shortest Path First | **OSPF** |
| IP의 **비신뢰 보완**, 에러 **진단** | Internet Control **Message** Protocol | **ICMP** |

### 메커니즘

**네트워크 보안 프로토콜 — 계층별 위치**

```
OSI 계층        프로토콜
7 (응용)    ──  SSH (원격 접속 암호화)
            ──  HTTPS (웹 통신 암호화)
4~7 (전송+) ──  SSL/TLS (HTTPS의 기반)
3 (네트워크)──  IPSec (IP 패킷 암호화, VPN)
            ──  ICMP (에러 보고, ping)
            ──  OSPF (링크 상태 라우팅)
```

**라우팅 프로토콜 비교 (OSPF vs RIP)**

| 항목 | OSPF | RIP |
|------|------|-----|
| 알고리즘 | **다익스트라** (링크 상태) | 벨만-포드 (거리 벡터) |
| 경로 기준 | 대역폭, 지연, 비용 | **홉 수(Hop Count)만** |
| 업데이트 | 링크 상태 변화 시에만 | **30초마다 전체** |
| 규모 | 대형 네트워크 | 소형 네트워크 |
| 수렴 속도 | **빠름** | 느림 |

> ⚠️ **ICMP ≠ IP**: ICMP는 IP 위에서 동작하는 별도 프로토콜  
> `ping` 명령 = ICMP Echo Request/Reply 사용  
> traceroute도 ICMP 사용

### ✅ 정답
2회 Q12: IPSec  
3회 Q11: OSPF  
3회 Q12: ICMP

## SQL Injection / 스니핑

> 2회 Q10, 4회 Q06

### 지문 키워드 (SQL Injection)
> **"웹 프로그램에 SQL 구문을 삽입하여, 서버의 데이터베이스를 공격할 수 있는 공격 방식"**

### 지문 키워드 (스니핑)
> **"네트워크상에 통과되는 패킷들의 내용을 엿보는 행위"**

### 메커니즘 (SQL Injection)

```
[정상 로그인 쿼리]
SELECT * FROM users WHERE id='admin' AND pw='1234'
→ id와 pw 모두 맞아야 로그인

[SQL Injection 공격]
id 입력값: admin'--
→ SELECT * FROM users WHERE id='admin'--' AND pw='...'
                                       ↑ '--'은 주석 처리
→ pw 검사 무력화 → admin으로 로그인 성공

또는:
id 입력값: ' OR '1'='1
→ SELECT * FROM users WHERE id='' OR '1'='1' AND pw='...'
→ '1'='1'은 항상 참 → 모든 사용자 조회
```

**보안 대책**

| 대책 | 방법 |
|------|------|
| PreparedStatement | SQL 파라미터 바인딩, 입력값 리터럴 처리 |
| 입력값 검증 | 특수문자 필터링 (', --, ;...) |
| WAF | 웹 방화벽으로 의심 패턴 차단 |
| 최소 권한 | DB 계정에 최소 권한만 부여 |

**네트워크 공격 비교**

| 공격 | 방법 | 핵심 |
|------|------|------|
| **SQL Injection** | DB에 SQL 삽입 | 웹 입력값 악용 |
| **스니핑** | 패킷 도청 | 네트워크 수동 감청 |
| **스푸핑** | 주소 위조 | IP/MAC 위조로 신뢰 가장 |
| **세션 하이재킹** | 세션 토큰 탈취 | 인증 후 세션 도용 |

### ✅ 정답
2회 Q10: 웹 프로그램에 SQL 구문을 삽입하여, 서버의 데이터베이스를 공격할 수 있는 공격 방식  
4회 Q06: 네트워크상에 통과되는 패킷들의 내용을 엿보는 행위

```
서비스 중단 발생               복구 완료
```

> ⚠️ **RTO vs RPO**: R**T**O = **T**ime (시간), R**P**O = **P**oint (시점)

## 트랜잭션 ACID

> 1회 Q12

### 지문 키워드
> **① "트랜잭션 내의 연산은 반드시 모두 수행되어야 하며, 그렇지 못한 경우 모두 수행되지 않아야"**
> **② "하나의 트랜잭션이 수행 중에는 다른 트랜잭션이 접근할 수 없고 독립적이어야"**

### 왜 이게 답인가?

| 지문 키워드 | 연결 포인트 | 정답 |
|------------|------------|------|
| **모두 수행 or 모두 미수행** | All or Nothing = 원자(Atom) 단위 | **① 원자성 (Atomicity)** |
| **다른 트랜잭션 접근 불가**, 독립적 | Isolation = 격리/고립 | **② 격리성 (Isolation)** |

### 메커니즘

**ACID 4가지 상세**

| 특성 | 영문 | 의미 | 실생활 예시 | 위반 시 |
|------|------|------|------------|---------|
| **원자성** | Atomicity | 전부 or 전무 | 계좌이체: 출금+입금 동시에 | 출금만 되고 입금 안 됨 |
| **일관성** | Consistency | 항상 일관된 DB 상태 | 잔액 ≥ 0 항상 유지 | 음수 잔액 발생 |
| **격리성** | Isolation | 트랜잭션 간 독립 | 동시 예약 시 중복 방지 | 같은 좌석 2명이 예약 |
| **영속성** | Durability | 완료 후 영구 저장 | 커밋 후 전원 꺼져도 유지 | 재시작 시 데이터 사라짐 |

```
[원자성 예시]
계좌이체 트랜잭션:
  1. A 계좌에서 10,000원 출금
  2. B 계좌에 10,000원 입금

→ 1번 성공 후 2번 실패 시: 트랜잭션 전체 ROLLBACK
→ A 계좌도 원래대로 복원
```

> **암기법**: `ACID` → **원**자성 / **일**관성 / **격**리성 / **영**속성  
> "원일격영" 순서로 외우기

### ✅ 정답
① 원자성 (Atomicity)  
② 격리성 (Isolation)

## ROLLBACK / COMMIT

> 2회 Q11

### 지문 키워드
> **"데이터베이스 내의 연산이 비정상적으로 종료되거나 정상수행이 되었다 하더라도 수행되기 이전의 상태로 되돌리기 위한 명령"**

### 왜 이게 답인가?

| 지문 키워드 | 연결 포인트 | 정답 |
|------------|------------|------|
| **이전 상태로 되돌리기** | Roll Back = 뒤로 굴리기 | **ROLLBACK** |
| **비정상 종료에도 적용** | 항상 트랜잭션 취소 가능 | **ROLLBACK** |

### 메커니즘

**TCL (Transaction Control Language) 3종**

```
트랜잭션 시작
  |
  ├── INSERT/UPDATE/DELETE 작업들...
  |
  ├── COMMIT ──────▶ DB에 영구 반영 (되돌릴 수 없음)
  |
  └── ROLLBACK ────▶ 트랜잭션 시작 시점으로 되돌리기
       SAVEPOINT ──▶ 중간 저장점 (부분 ROLLBACK 가능)
```

| 명령어 | 역할 | 되돌릴 수 있나? |
|--------|------|----------------|
| **COMMIT** | 변경 사항 확정·영구 저장 | ❌ 불가 |
| **ROLLBACK** | 변경 사항 취소·이전 상태 복원 | ✅ 가능 |
| **SAVEPOINT** | 중간 저장점 설정 | - |

> ⚠️ **ROLLBACK vs DROP**: ROLLBACK은 데이터 변경을 취소, DROP은 테이블 구조 자체를 삭제  
> DROP은 DDL이라 ROLLBACK 불가 (일부 DBMS 제외)

### ✅ 정답
데이터베이스 내의 연산이 비정상적으로 종료되거나 정상수행이 되었다 하더라도 수행되기 이전의 상태로 되돌리기 위한 명령이다.

## DB 스키마 (Schema)

> 3회 Q09

### 지문 키워드
> **"데이터베이스의 전체적인 구조와 제약조건에 대한 명세를 기술하고 정의한 것"**

### 핵심 개념

**스키마 vs 인스턴스 구분**

```
[릴레이션 스키마] ← 구조/틀, 변하지 않음
학생(학번, 이름, 나이, 학과)
 ↑ 컬럼명과 타입 정의

[릴레이션 인스턴스] ← 실제 데이터, 시간에 따라 변함
┌──────┬───────┬───────┬────────┐
│ 학번  │  이름  │  나이  │  학과   │
├──────┼───────┼───────┼────────┤
│ 001  │홍길동  │  22   │  컴퓨터  │
│ 002  │이순신  │  23   │  전자   │
└──────┴───────┴───────┴────────┘
```

**3단계 스키마 구조**

| 단계 | 이름 | 관점 | 설명 |
|------|------|------|------|
| 외부 | External Schema | **사용자/응용** | 각 사용자가 보는 논리적 구조 (뷰) |
| 개념 | Conceptual Schema | **DBA** | DB 전체의 논리적 구조 (가장 중요) |
| 내부 | Internal Schema | **시스템** | 물리적 저장 구조 (파일, 인덱스) |

> **3단계 스키마의 목적**: 데이터 독립성 확보  
> 내부 구조가 바뀌어도 개념/외부 스키마에 영향 없음

### ✅ 정답
데이터베이스의 전체적인 구조와 제약조건에 대한 명세를 기술하고 정의한 것이다.

## 관계대수 DIVISION(÷)

> 3회 Q10

### 지문 키워드
> **"릴레이션 A에서 릴레이션 B의 모든 조건을 만족하는 튜플을 가져오면서, 사용된 속성은 제외하고 가져오는 연산자"**

### 핵심 개념

**관계대수 주요 연산자**

| 연산자 | 기호 | 의미 | SQL 대응 |
|--------|------|------|---------|
| 셀렉션 | σ | 조건 만족 행 추출 | WHERE |
| 프로젝션 | π | 특정 열만 추출 | SELECT |
| **디비전** | **÷** | **B의 모든 값 갖는 A의 값** | - |
| 조인 | ⋈ | 두 테이블 결합 | JOIN |

**÷ 연산 이해**

```
R ÷ S = "S의 모든 B값과 쌍이 되는 R의 A값"

예:
[R]         [S]
A  | B      B
───────     ─
a1 | b1     b1
a1 | b2     b2
a2 | b1

S = {b1, b2} 모두 갖는 A값은?
a1 → {b1, b2} 모두 있음 ✅
a2 → {b1}만 있음 ❌
```

## 프로세스 상태 전이

> 4회 Q10

### 지문 내용
프로세스 상태 전이도 빈칸 ①~③ 채우기  
(생성→①→②→종료, Wake Up, Block, Dispatch, Time run out 존재)

### 핵심 개념

**프로세스 5가지 상태**

```
생성(New)
  ↓ 승인
준비(Ready) ◀──── Time run out ─── 실행(Running)
     ↓ Dispatch(CPU 할당)               ↓ Block(I/O 요청 등)
실행(Running)                      대기(Waiting/Blocked)
                                         ↓ Wake Up
                                   준비(Ready)로 복귀
실행 완료 → 종료(Terminated)
```

**상태별 의미**

| 상태 | 설명 | 전이 키워드 |
|------|------|------------|
| 생성 (New) | 프로세스 막 생성 | → 준비 |
| **준비 (Ready)** | CPU 배정 기다리는 중 | Dispatch → 실행 |
| **실행 (Running)** | CPU 점유하여 실행 중 | Time run out → 준비, Block → 대기 |
| **대기 (Waiting)** | I/O 등 이벤트 기다리는 중 | Wake Up → 준비 |
| 종료 (Terminated) | 실행 완료 | - |

**빈칸 정답**

| 빈칸 | 상태 | 근거 |
|------|------|------|
| ① | **준비 (Ready)** | 생성 직후 → CPU 배정 대기 |
| ② | **실행 (Running)** | Dispatch(CPU 할당) 받아서 실행 |
| ③ | **대기 (Waiting/Blocked)** | Block(I/O 요청) 후 Wake Up으로 복귀 |

> **암기법**: `생준실대종` — 생성/준비/실행/대기/종료 5단계

### ✅ 정답
① 준비 (Ready)  
② 실행 (Running)  
③ 대기 (Waiting / Blocked)

## HRN 우선순위 공식

> 1회 Q11

### 핵심 개념

**HRN (Highest Response-ratio Next)**  
= SJF의 단점인 **기아(Starvation) 현상**을 보완한 비선점형 스케줄링

```
우선순위 값 = (대기시간 + 서비스시간) / 서비스시간
```

값이 **클수록** 먼저 실행됨

### 메커니즘

**SJF의 문제점 + HRN의 해결책**

```
[SJF의 기아 현상]
프로세스A: 실행시간=2  → 우선순위 높음 → 계속 먼저 실행
프로세스B: 실행시간=20 → 우선순위 낮음 → 영원히 대기...

[HRN의 해결]
프로세스B가 오래 기다릴수록 대기시간↑ → 분자↑ → 우선순위값↑
→ 결국 실행될 기회 생김
```

**예시 계산**

| 프로세스 | 대기시간 | 서비스시간 | HRN 값 | 순서 |
|----------|---------|-----------|--------|------|
| P1 | 10 | 2 | (10+2)/2 = **6.0** | ① |
| P2 | 2 | 4 | (2+4)/4 = **1.5** | ③ |
| P3 | 3 | 3 | (3+3)/3 = **2.0** | ② |

**스케줄링 알고리즘 비교**

| 알고리즘 | 방식 | 기준 | 기아 | 특징 |
|----------|------|------|------|------|
| FCFS | 비선점 | 도착 순서 | 없음 | 단순, 호위 효과 |
| SJF | 비선점 | 실행 시간 짧은 것 | **발생** | 최적 평균 대기 |
| SRT | 선점 | 남은 실행 시간 | 발생 | SJF 선점 버전 |
| **HRN** | **비선점** | **(대기+서비스)/서비스** | **없음** | **기아 방지** |
| RR | 선점 | 타임퀀텀 순환 | 없음 | 공평, 시분할 |

> ⚠️ **HRN 공식 암기**: 분자에 대기시간+서비스시간, 분모에 서비스시간

### ✅ 정답
(대기시간 + 서비스시간) / 서비스시간

## chmod (Linux 권한)

> 2회 Q14

### 지문 내용
- 사용자: 읽기, 쓰기, 실행 → 3개 권한 모두
- 그룹: 읽기, 실행 → 2개
- 기타: 실행 → 1개

### 핵심 개념

**권한 8진수 계산**

```
읽기(r) = 4
쓰기(w) = 2
실행(x) = 1

사용자: rwx = 4+2+1 = 7
그룹:   r-x = 4+0+1 = 5
기타:   --x = 0+0+1 = 1

chmod 751 a.txt
         ↑↑↑
         │││
         ││└── 기타(others) = 1
         │└─── 그룹(group) = 5
         └──── 사용자(user) = 7
```

**권한 표기 방식 2가지**

| 방식 | 표기 | 예시 |
|------|------|------|
| 8진수 | 숫자 3자리 | `chmod 751 파일명` |
| 기호 | u/g/o + r/w/x | `chmod u=rwx,g=rx,o=x 파일명` |

> ⚠️ **자주 나오는 값 암기**:  
> `777` = 모두 rwx = 누구나 다 가능  
> `755` = 사용자는 모두, 그룹·기타는 읽기+실행만  
> `644` = 사용자는 읽기+쓰기, 그룹·기타는 읽기만

### ✅ 정답

### 메커니즘 — 리팩토링

```
[리팩토링 전]                  [리팩토링 후]
function calc(x, y, z) {      function calculateTotal(price, tax, discount) {
  return x * y - z;     →       return price * tax - discount;
}                             }
```

## C++ 생성자 (Constructor)

> 3회 Q20

### 지문 키워드
> **"C++에서 생성자(Constructor)에 대해 간략히 설명하시오"**

### 핵심 개념

```cpp
class Student {
public:
    string name;
    int age;

    // 생성자 — 클래스명과 동일, 반환 타입 없음
    Student(string n, int a) {
        name = n;
        age = a;
    }    // ← 객체 생성 시 자동 호출
};

// 사용
Student s("홍길동", 22);  // 생성자 자동 호출 → name="홍길동", age=22
```

**생성자의 특징**

| 특징 | 설명 |
|------|------|
| 이름 | 클래스명과 **동일** |
| 반환 타입 | **없음** (void도 아님) |
| 호출 시점 | 객체 **생성 시 자동** 호출 |
| 목적 | 멤버 변수 **초기화** |
| 오버로딩 | 가능 (매개변수 다르게 여러 개) |
| 기본 생성자 | 정의 안 하면 컴파일러가 자동 생성 |

> ⚠️ **생성자 vs 소멸자**  
> 생성자: 객체 생성 시 호출, 초기화  
> 소멸자(~클래스명): 객체 소멸 시 호출, 메모리 해제

### ✅ 정답
생성자는 객체 생성 시 자동으로 호출되는 메서드로 멤버를 초기화하는 목적으로 주로 사용된다.

## 형상관리 (SCM)

> 2회 Q04

### 지문 키워드
> **"소프트웨어의 변경사항을 체계적으로 추적하고 통제하는 것"**
> **"소스 코드 뿐 아니라 개발 환경, 빌드 구조 등 전반적인 내역에 대한 관리 체계를 정의"**
> **"지원 도구로 Git, SVN, CVS 등이 있다"**

### 왜 이게 답인가?

| 지문 키워드 | 연결 포인트 | 정답 |
|------------|------------|------|
| **변경사항 추적·통제** | Software **Configuration** Management | 형상관리 (SCM) |
| **Git, SVN, CVS** | 대표적인 형상관리 도구 | 형상관리 (SCM) |

### 메커니즘

**형상관리 4가지 활동**

```
① 형상 식별 (Configuration Identification)
   → 관리 대상(소스코드, 문서, DB 스키마...)을 선정하고 이름 부여

② 형상 통제 (Configuration Control)
   → 변경 요청 검토·승인·거부 결정

③ 형상 감사 (Configuration Audit)
   → 변경이 올바르게 적용됐는지 검증

④ 형상 기록 (Configuration Status Accounting)
   → 변경 이력 문서화
```

**형상관리 도구 비교**

| 도구 | 방식 | 특징 |
|------|------|------|
| **Git** | 분산형 | 로컬 저장소 + 원격, 브랜치 용이 |
| SVN | 중앙집중형 | 서버에 단일 저장소 |
| CVS | 중앙집중형 | SVN의 전신, 구식 |

> ⚠️ **형상관리 ≠ 버전관리**: 버전관리는 형상관리의 일부  
> 형상관리 = 버전관리 + 변경 요청 관리 + 릴리즈 관리

### ✅ 정답
형상관리 (SCM, Software Configuration Management)

## Observer 패턴

> 2회 Q16

### 지문 키워드
> **"한 객체의 상태가 바뀌면 그 객체에 의존하는 다른 객체들한테 연락이 가고, 자동으로 내용이 갱신되는 방식"**
> **"일대다(one-to-many) 의존성을 정의"**

### 왜 이게 답인가?

| 지문 키워드 | 연결 포인트 | 정답 |
|------------|------------|------|
| **상태 변화 → 자동 연락·갱신** | 관찰(Observe) + 알림(Notify) | **Observer** |
| **일대다 의존성** | 1 Subject → 다수 Observer | **Observer** |

### 메커니즘

```
[Observer 패턴 구조]

                Subject (발행자)
                  │
                  │ notify() 호출
                  ▼
    ┌────────────────────────────┐
    │            │               │
Observer1    Observer2       Observer3
(구독자)      (구독자)        (구독자)
    │            │               │
 update()     update()        update()
    
Subject 상태 변경 → 등록된 모든 Observer에게 자동 알림
```

**실생활 예시**

```
유튜브 채널(Subject) ─── 새 영상 업로드 ───▶ 구독자1(Observer) 알림
                                         ───▶ 구독자2(Observer) 알림
                                         ───▶ 구독자3(Observer) 알림

→ 구독자가 늘어도 Subject 코드 변경 없음 (개방-폐쇄 원칙)
```

**디자인 패턴 구별 포인트**

| 패턴 | 핵심 키워드 | 구별 포인트 |
|------|------------|------------|
| **Observer** | 일대다 의존, 상태 변화 자동 알림 | 구독/발행 구조 |
| Proxy | 접근 제어, 대리인 | 같은 인터페이스로 실제 객체 대리 |
| Adapter | 인터페이스 변환 | Wrapper, 호환성 |
| Singleton | 인스턴스 1개 | static, 전역 접근 |

### ✅ 정답
Observer (패턴)

## Linked Open Data / 하둡 / 블록체인

> 2회 Q17, 4회 Q13, Q14

### 지문 키워드 모음

> **LOD**: "저작권 표시 하에 재배포 가능한 링크 기반 개방형 데이터", "URI 부여, HTTP/RDF/URI 웹 표준 기술 이용"  
> **하둡**: "비정형 데이터 처리 오픈소스 빅데이터 솔루션", "범용 컴퓨터에 분산 저장·처리", "더그 커팅 개발, MapReduce 엔진"  
> **블록체인**: "'블록' 소규모 데이터들이 P2P 방식으로 체인 형태 연결", "누구도 수정 불가, 변경 이력 모두 열람 가능", "분산 컴퓨팅 기반 원장 관리"

### 왜 이게 답인가?

| 지문 키워드 | 연결 포인트 | 정답 |
|------------|------------|------|
| **링크 기반 개방형 데이터**, URI, HTTP/RDF | Linked Open Data = 개방·연결 데이터 | **LOD** |
| **분산 저장·처리**, MapReduce, **더그 커팅** | Hadoop = HDFS + MapReduce | **하둡 (Hadoop)** |
| **P2P 체인 형태**, 수정 불가, **분산 원장** | Block + Chain = 블록체인 | **블록체인** |

### 메커니즘

**블록체인 구조**

```
[블록 체인 구조]
블록1         블록2         블록3
┌──────────┐ ┌──────────┐ ┌──────────┐
│해시(이전) │ │해시(블록1)│ │해시(블록2)│
│데이터     │─▶│데이터     │─▶│데이터     │
│해시(자신) │ │해시(자신) │ │해시(자신) │
└──────────┘ └──────────┘ └──────────┘
이전 블록의 해시를 포함 → 블록 수정 시 이후 모든 블록 해시 변경 → 조작 불가
```

**하둡 vs 블록체인 vs LOD**

| 기술 | 목적 | 핵심 |
|------|------|------|
| **하둡** | 빅데이터 분산 처리 | MapReduce, HDFS |
| **블록체인** | 분산 신뢰 원장 | 위변조 불가, P2P |
| **LOD** | 개방 연결 데이터 | URI, 재사용 가능 |

### ✅ 정답

4회 Q13: 하둡 (Hadoop)
