## Q01. 스머프 공격 (Smurf Attack)

### 지문 키워드
> **"공격자는 출발지 IP 주소를 피해자의 IP로 위조"**  
> **"다수의 호스트가 연결된 네트워크 브로드캐스트 주소로 ICMP 에코 요청(Echo Request)을 전송"**  
> **"브로드캐스트 네트워크에 연결된 모든 호스트들이 피해자 IP로 대량의 ICMP 에코 응답(Echo Reply)을 보냄"**  
> **"피해자가 과도한 트래픽으로 인해 서비스 장애"**

### 왜 이게 답인가?

| 키워드 | 연결 포인트 |
|--------|------------|
| **출발지 IP 위조** | IP 스푸핑 |
| **브로드캐스트** 주소로 ICMP 전송 | 증폭 효과 |
| **모든 호스트**가 응답 → 피해자에게 집중 | DDoS 증폭 공격 |

### 메커니즘

```
공격자 (IP를 피해자로 위조)
    ↓ ICMP Echo Request → 브로드캐스트 주소 (255.255.255.255)
    
브로드캐스트 네트워크의 모든 호스트들
    ↓ ICMP Echo Reply → 피해자 IP (위조된 출발지)
    
피해자 ← 수천 개의 응답 쏟아짐 → 서비스 마비 (DDoS)
```

### 유사 공격 비교

| 공격 | 방법 |
|------|------|
| **스머프** | ICMP + IP위조 + 브로드캐스트 증폭 |
| Ping Flood | 단순 대량 ping |
| SYN Flood | TCP 연결 반만 맺어 자원 고갈 |
| DRDoS | 반사 서버 이용 증폭 |

### ✅ 정답
```
스머프 (Smurf, 스머핑, Smurfing)
```

## Q04. 문장 / 분기 / 조건 커버리지

### 지문 키워드
> **① "코드 상의 모든 문장이 최소 한 번 이상 실행되는지를 검사"**  
> **② "if나 switch와 같은 분기문이 각각 정상적으로 수행되는지를 확인"**  
> **③ "각 조건식이 참/거짓으로 조합되도록 확인"**

### 왜 이게 답인가?

| 키워드 | 정답 |
|--------|------|
| **모든 문장** 최소 1회 실행 | **문장 커버리지** (Statement) |
| **분기문** 정상 수행 확인 | **분기 커버리지** (Branch/Decision) |
| **조건식** 참/거짓 조합 | **조건 커버리지** (Condition) |

### 메커니즘 비교

```
if(A > 0 && B > 0) { ... }

문장 커버리지: 위 if 블록 안이 실행되기만 하면 됨
분기 커버리지: if 전체가 true 1번, false 1번 필요
조건 커버리지: A>0 이 T/F, B>0 이 T/F 각각 필요
```

### ✅ 정답
```
① 문장 커버리지 (Statement Coverage)
② 분기 커버리지 (Branch/Decision Coverage)
③ 조건 커버리지 (Condition Coverage)
```

## Q05. 외래키 / 후보키 / 대체키 / 슈퍼키

### 지문 키워드
> **① "한 테이블의 특정 속성이 다른 테이블(또는 동일 테이블)의 튜플을 참조함으로써, 테이블 간의 연관 관계와 무결성을 유지하기 위한 키"**  
> **② "릴레이션에서 각 튜플을 고유하게 식별하며, 속성의 집합이 최소로 구성되어 있는 키"**  
> **③ "위 ②번 키가 여러 개 존재할 경우, 그중 기본키(Primary Key)로 선정되지 않은 키"**  
> **④ "릴레이션에서 모든 튜플을 유일하게 식별할 수 있는 속성들의 부분 집합(적어도 하나 이상의 ②번 키를 포함)으로 구성된 키"**

### 왜 이게 답인가?

| 키워드 | 정답 |
|--------|------|
| **다른 테이블 참조**, 연관 관계 | **외래키 (Foreign Key)** |
| **최소** 속성 집합, 튜플 고유 식별 | **후보키 (Candidate Key)** |
| 후보키 중 **기본키로 선정 안 된** 것 | **대체키 (Alternate Key)** |
| 후보키를 **포함**하는 상위 집합 | **슈퍼키 (Super Key)** |

### 키 포함 관계

```
슈퍼키 (Super Key)          ← 가장 넓은 범위
  ├── 후보키 (Candidate Key) ← 최소성 만족
  │     ├── 기본키 (Primary Key) ← 후보키 중 선택 1개
  │     └── 대체키 (Alternate Key) ← 선택 안 된 나머지
  └── (후보키 아닌 슈퍼키들)

외래키: 다른 테이블의 기본키를 참조
```

### ✅ 정답
```
① 외래키 (Foreign Key)
② 후보키 (Candidate Key)
③ 대체키 (Alternate Key)
④ 슈퍼키 (Super Key)
```

## Q06. URL 구조

### 문제
```
https://user:pass@exam.com:8080/products/items?id=123&s=asc#specs
  ①        ②              ③          ④            ⑤
```

### 개념 — URL 5가지 구성요소

| 요소 | 설명 | 예시 |
|------|------|------|
| **scheme** | 접근 프로토콜 | `https` |
| **authority** | 사용자정보+호스트+포트 | `user:pass@exam.com:8080` |
| **path** | 서버 내 자원 경로 | `/products/items` |
| **query** | 서버에 전달할 추가 데이터 | `id=123&s=asc` |
| **fragment** | 문서 내 특정 위치 | `specs` |

### URL 분해

```
https://user:pass@exam.com:8080/products/items?id=123&s=asc#specs
  ↑①        ↑②                 ↑③              ↑④          ↑⑤

①  scheme    = https
②  authority = user:pass@exam.com:8080
③  path      = /products/items
④  query     = id=123&s=asc
⑤  fragment  = specs
```

### ✅ 정답 (URL에 나타나는 순서)
```
③ scheme → ④ authority → ② path → ① query → ⑤ fragment
```

## Q09. UML 관계 — 연관 / 일반화 / 의존

### 왜 이게 답인가?

| 그림 표현 | 관계 | 설명 |
|----------|------|------|
| ① 차 ↑─ 타이어/바퀴/엔진 | **연관 관계 (Association)** | 전체-부분 포함 (has-a) |

| 관계 | 표기 | 의미 |
|------|------|------|

| **일반화** | ──▷ | 상속 (is-a), 빈 삼각형 |
| **의존** | - - → | 점선 화살표, 일시적 사용 |
| 실체화 | - - ▷ | 인터페이스 구현 |
| 집합 | ◇──── | 약한 포함 |
| 합성 | ◆──── | 강한 포함 |

### ✅ 정답
```
① 연관 관계 (Association)
② 일반화 관계 (Generalization)
③ 의존 관계 (Dependency)
```

## Q11. SQL — 중첩 서브쿼리 + INNER JOIN

### 문제
```sql
SELECT count(*)
FROM employee e inner join project p
ON e.p_id = p.p_id
WHERE p.name IN (
    SELECT name FROM project p
    WHERE p.p_id IN (
        SELECT p_id FROM employee
        GROUP BY p_id
        HAVING count(*) < 2
    )
)
```

**employee**: (1,이홍직,10), (2,이창훈,20), (3,이경직,10)  
**project**: (10,CJOne), (20,Asiana), (30,SignsSEA)

### 단계별 트레이싱

**가장 안쪽 서브쿼리:**
```sql
SELECT p_id FROM employee GROUP BY p_id HAVING count(*) < 2
-- p_id별 인원수:
--   10 → 이홍직, 이경직 = 2명 → 2 < 2 ❌
```

## Q16. Java — 오버라이딩 + 필드 접근

### 문제
```java
class A {
    int x = 3;
    int getX() { return x * 2; }
}
class B extends A {
    int x = 7;
    int getX() { return x * 3; }  // 오버라이딩
}
A a = new B();
B b = new B();
System.out.print(a.getX() + a.x + b.getX() + b.x);
```

### 핵심 분석

**메서드 (getX()): 동적 바인딩**
```
a = new B() → getX() 오버라이딩 → B.getX() 실행
B.getX() = B.x * 3 = 7 * 3 = 21
```

**필드 (x): 정적 바인딩 — 참조 타입 기준!**
```
a.x → a는 A타입 → A.x = 3
b.x → b는 B타입 → B.x = 7
```

### 트레이싱

| 표현 | 바인딩 | 값 |
|------|--------|-----|
| `a.getX()` | 동적 → B.getX() = 7×3 | **21** |
| `a.x` | 정적 → A.x | **3** |
| `b.getX()` | 동적 → B.getX() = 7×3 | **21** |
| `b.x` | 정적 → B.x | **7** |

합계: 21 + 3 + 21 + 7 = **52**

> ⚠️ **필드는 동적 바인딩이 없다!** — a.x는 A타입이므로 A.x=3


## Q17. C언어 — 연결 리스트 버블정렬

### 문제
```c
// 연결 리스트: n1→n3→n2 (1→3→2)
// f() = 인접 노드 값 스왑 (버블 정렬)
struct Node n1={1,NULL}, n2={2,NULL}, n3={3,NULL};
n1.next=&n3; n3.next=&n2;
f(&n1);
// 출력: c(순회) → n1값, n2값, n3값 순서대로
```

### f() 함수 분석 — 버블정렬

```c
while(node != NULL && node->next != NULL) {
    t = node->value;
    node->value = node->next->value;
    node->next->value = t;
    node = node->next->next;  // 두 칸씩 이동
}
```

### 트레이싱

**초기 연결 리스트**: n1(1) → n3(3) → n2(2) → NULL

**f(&n1) 실행:**
```
1단계: node=n1
  t = n1.value = 1
  n1.value = n3.value = 3
  n3.value = t = 1
  node = n3.next = n2

리스트: n1(3) → n3(1) → n2(2)

2단계: node=n2
  t = n2.value = 2
  n2.value = n2.next(NULL).value → NULL이면 종료?
```

> while 조건: `node != NULL && node->next != NULL`  
> node=n2일 때 n2.next=NULL → **조건 불만족 → 종료**

**최종 리스트**: n1(3) → n3(1) → n2(2)

**while(c!=NULL) 순회 출력:**
```
c=n1: printf("3 ")
c=n3: printf("1 ")
c=n2: printf("2 ")
```

### ✅ 정답
```
3 1 2
```

## Q20. Java — 제네릭 + 오버로딩

### 문제
```java
class Printer {
    void print(Integer a) { System.out.print("A" + a); }
    void print(Object a)  { System.out.print("B" + a); }
    void print(Number a)  { System.out.print("C" + a); }
}
class Collection<T> {
    T value;
    public Collection(T t) { value = t; }
    public void print() {
        new Printer().print(value);
    }
}
new Collection<>(0).print();
```

### 핵심 — 제네릭 타입 소거 (Type Erasure)

```java
new Collection<>(0)
// 0은 int → 오토박싱 → Integer
// T = Integer로 추론
// value = (Integer)0

// BUT! 제네릭 타입 T는 런타임에 Object로 소거됨
// Collection<T>의 T → 컴파일 후 Object
// value의 타입 = Object

new Printer().print(value)
// value의 컴파일 타임 타입 = Object (T가 소거됨)
// → print(Object a) 선택
// → "B" + 0 = "B0"
```

> ⚠️ 제네릭 타입 소거: `Collection<T>`에서 T는 런타임에 Object가 됨  
> 오버로딩 선택은 **컴파일 타임** 타입 기준 → Object → `print(Object)` 호출

### ✅ 정답
```
B0
```
