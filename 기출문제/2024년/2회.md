## Q01. 반정규화 (Denormalization)

### 지문 키워드
> **"시스템의 성능 향상, 개발 과정의 편의성 등을 위해 정규화된 데이터 모델을 분할, 통합, 추가하는 과정"**  
> **"의도적으로 정규화에 위배되는 행위"**

### 왜 이게 답인가?

| 키워드 | 연결 포인트 |
|--------|------------|
| **정규화에 위배** | 반(反)정규화 = 정규화 반대 방향 |
| **성능 향상** 목적 | 조인 줄이기, 조회 속도 향상 |
| **분할, 통합, 추가** | 테이블 구조 변형 |

### 정규화 vs 반정규화

| | 정규화 | 반정규화 |
|--|--------|----------|
| 목적 | 중복 제거, 이상 방지 | **성능 향상** |
| 방향 | 테이블 분리 | 테이블 통합/중복 허용 |
| 단점 | 조인 증가 | 데이터 중복 |
| 시점 | 설계 단계 | 성능 이슈 발생 후 |

### ✅ 정답
```
반정규화 (역정규화, 비정규화)
```

## Q02. IPsec

### 지문 키워드
> **"VPN을 통해 안전한 통신을 구현하기 위해 사용되는 프로토콜"**  
> **"네트워크 계층에서 동작"**  
> **"데이터의 기밀성, 무결성, 그리고 인증을 제공"**  
> **"AH(Authentication Header)와 ESP(Encapsulating Security Payload) 두 가지 하위 프로토콜"**  
> **"터널 모드와 전송 모드를 지원"**  
> **"IKE(Internet Key Exchange) 프로토콜과 함께 보안 연결을 설정"**

### 왜 이게 답인가?

| 키워드 | 연결 포인트 |
|--------|------------|
| **AH + ESP** 두 하위 프로토콜 | IPsec의 고유 식별자 |
| **네트워크 계층** 동작 | OSI 3계층 (IP 레벨) |
| **터널/전송 모드** | IPsec의 두 가지 동작 모드 |
| **기밀성+무결성+인증** | IPsec 3대 보안 기능 |

### 메커니즘 — 두 가지 모드

```
[전송 모드] IP헤더 | AH/ESP | 데이터
           ↑ 원본 IP헤더 유지, 데이터만 보호

[터널 모드] 새IP헤더 | AH/ESP | 원본IP헤더 | 데이터
           ↑ 원본 패킷 전체를 새 패킷으로 감쌈 → VPN에 사용
```

### AH vs ESP

| | AH | ESP |
|--|----|----|
| 기밀성(암호화) | ❌ | ✅ |
| 무결성 | ✅ | ✅ |
| 인증 | ✅ | ✅ |

### ✅ 정답
```
IPsec (Internet Protocol Security)
```

## Q03. 순차적 응집도 (Sequential Cohesion)

### 지문 키워드
> **"모듈 내부의 각 기능들이 서로 연관성을 가지고 순차적으로 수행"**  
> **"하나의 작업이 끝나면 그 결과가 다음 작업의 입력으로 사용"**  
> **"기능적 응집도보다는 낮지만, 절차적 응집도보다는 높은 수준"**

### 왜 이게 답인가?

| 키워드 | 연결 포인트 |
|--------|------------|
| **앞 출력 → 뒤 입력** | Sequential = 순차 |
| 기능적 < **순차적** < 절차적 순위 | 응집도 7단계 위치 |

### 응집도 7단계 재정리

```
기능 > 순차 > 통신 > 절차 > 시간 > 논리 > 우연
 1      2      3      4      5      6      7
```

> **순차적**: 앞 모듈의 출력이 뒤 모듈의 **입력**으로 연결 (파이프라인 구조)  
> **절차적**: 순서는 있지만 데이터가 연결되지 않음 (단순 순서 실행)

### ✅ 정답
```
순차적 응집도 (Sequential Cohesion)
```

## Q04. 가상회선 / 데이터그램

### 지문 키워드
> **① "연결 지향적, 데이터 전송 전에 먼저 경로를 설정, 모든 패킷이 동일한 경로로 순차적으로 전달, 패킷이 순서대로 도착하는 것이 보장"**  
> **② "비연결 지향적, 경로를 미리 설정하지 않고, 각 패킷이 독립적으로 전송, 패킷이 목적지에 도착하는 순서가 보장되지 않음"**

### 왜 이게 답인가?

| 키워드 | 정답 |
|--------|------|
| **연결 지향**, 경로 미리 설정, **순서 보장** | **가상회선 (Virtual Circuit)** |
| **비연결 지향**, 패킷 독립 전송, **순서 미보장** | **데이터그램 (Datagram)** |

### 메커니즘

```
[가상회선]
경로 설정 → A→B→C→D (고정)
모든 패킷이 이 경로로만 이동 → 순서 보장

[데이터그램]
패킷1 → A→B→D
패킷2 → A→C→D  ← 다른 경로
패킷3 → A→B→C→D
→ 도착 순서 제각각, 수신 측에서 재조립 필요
```

### ✅ 정답
```
① 가상회선 (Virtual Circuit)
② 데이터그램 (Datagram)
```

> **"동시에 여러 종류의 컬렉션(예: 배열, 리스트, 트리)에서도 동일한 접근 방식을 제공"**

### 왜 이게 답인가?

| 키워드 | 연결 포인트 |

> **"무선 통신, 금융 거래, 저장 데이터의 암호화 등에서 표준으로 채택"**

## Q07. 제어 결합도 (Control Coupling)

### 지문 키워드
> **"두 모듈 사이에서 하나의 모듈이 다른 모듈의 내부 논리 흐름을 제어하기 위해 특정 제어 신호를 전달"**  
> **"모듈 간의 독립성을 저하시켜 유지보수와 확장성에 부정적인 영향"**  
> **"호출하는 모듈이 호출된 모듈의 내부 동작 방식을 너무 자세히 알게 됨"**  
> **"제어 신호가 변경되면 관련된 모든 모듈을 수정해야 하는 상황이 발생"**

### 왜 이게 답인가?

| 키워드 | 연결 포인트 |
|--------|------------|
| **제어 신호** 전달 | Control = 제어 |
| 내부 **논리 흐름 제어** | 상대 모듈 동작 방식 지시 |
| 제어 신호 변경 시 **연쇄 수정** | 강한 의존성 |

### 결합도 7단계 (낮음 → 높음 = 좋음 → 나쁨)

```
자료 < 스탬프 < 제어 < 외부 < 공통 < 내용
  1      2       3      4      5      6
```

> **제어 결합도**: 파라미터로 제어 플래그(flag)를 넘기는 경우  
> 예: `processData(data, true)` — true/false로 내부 동작 제어

### ✅ 정답
```
제어 결합도 (Control Coupling)
```

## Q08. SRT 스케줄링 평균 대기시간

### 문제
| 프로세스 | 도착시간 | 실행시간 |
|---------|---------|---------|
| P1 | 0 | 8 |
| P2 | 1 | 4 |
| P3 | 2 | 9 |
| P4 | 3 | 5 |

**SRT (Shortest Remaining Time)**: 선점형 — 현재 실행 중인 프로세스보다 남은 시간이 짧은 프로세스가 도착하면 **즉시 선점**

### 단계별 SRT 시뮬레이션

| 시간 | 실행 | 이유 |
|------|------|------|
| 0 | P1 | P1만 있음 (잔여:8) |
| 1 | P2 | P2 도착 (잔여:4) < P1 잔여(7) → 선점 |
| 2 | P2 | P3 도착(잔여:9), P2 잔여(3) 가장 짧음 |
| 3 | P2 | P4 도착(잔여:5), P2 잔여(2) 가장 짧음 |
| 4 | P2 | P2 잔여(1) 가장 짧음 |
| 5 | P4 | P2 완료. P4(5) < P1(7) < P3(9) |
| 6 | P4 | |
| 7 | P4 | |
| 8 | P4 | |
| 9 | P4 | |
| 10 | P1 | P4 완료. P1(7) < P3(9) |
| 11~17 | P1 | |
| 17 | P3 | P1 완료 |
| 17~26 | P3 | |

### 완료 시간 및 대기시간 계산

| 프로세스 | 도착 | 실행 | 완료 | 반환시간 | 대기시간 |
|---------|------|------|------|---------|---------|
| P1 | 0 | 8 | 17 | 17 | 17-8 = **9** |
| P2 | 1 | 4 | 5 | 4 | 4-4 = **0** |
| P3 | 2 | 9 | 26 | 24 | 24-9 = **15** |
| P4 | 3 | 5 | 10 | 7 | 7-5 = **2** |

> 대기시간 = 반환시간 - 실행시간 = (완료시간 - 도착시간) - 실행시간

**평균 대기시간** = (9 + 0 + 15 + 2) / 4 = 26 / 4 = **6.5**

### ✅ 정답
```
평균 대기시간: 6.5
```

## Q09. RIP 최적 경로

### 문제
A에서 F까지 RIP(홉 카운트 기준) 최적 경로

```
A ─1─ D ─2─ B ─2─ C ─5─ F
          ↑           ↑
          2           1
          B           E ─2─ F
A ─2─ B
```

실제 연결:
- A-D: 1, A-B: 2
- D-B: 2, B-C: 2, B-D: 2
- C-E: 1, C-F: 5
- E-F: 2

### 경로별 홉 수 계산

| 경로 | 홉 수 |
|------|-------|
| A→B→C→F | 3홉 |
| A→D→B→C→F | 4홉 |
| A→B→C→E→F | 4홉 |
| A→D→B→C→E→F | 5홉 |

> RIP는 비용(숫자)이 아닌 **홉 수(경유 라우터 수)** 기준!

최소 홉: **A→B→C→F = 3홉**

### ✅ 정답
```
A → B → C → F (홉 수: 3)
```

## Q11. SQL — DML 키워드 빈칸

### 문제
```sql
insert into 직원(직원번호, 성명, 나이, 연봉, 팀번호)
( ① ) (101, 'Kim', 25, 3000, 2);

insert into 직원
( ② ) 직원번호, 성명, 나이, 연봉, 팀번호 from 신규직원
where 팀번호 = 2;

delete ( ③ ) 직원 where 직원번호 = 101;

update 직원 ( ④ ) 연봉 = 3500 where 성명 = 'Lee';
```

### 왜 이게 답인가?

| 빈칸 | 문법 구조 | 정답 |
|------|---------|------|
| ① | INSERT INTO ~ **VALUES** (...) | **values** |
| ② | INSERT INTO ~ **SELECT** 컬럼 FROM | **select** |
| ③ | DELETE **FROM** 테이블 | **from** |
| ④ | UPDATE 테이블 **SET** 컬럼=값 | **set** |

### SQL DML 기본 문법 정리

```sql
-- INSERT 방법 1: 직접 값 입력
INSERT INTO 테이블 (컬럼) VALUES (값);

-- INSERT 방법 2: 다른 테이블에서 SELECT
INSERT INTO 테이블 SELECT 컬럼 FROM 다른테이블;

-- DELETE
DELETE FROM 테이블 WHERE 조건;

-- UPDATE
UPDATE 테이블 SET 컬럼=값 WHERE 조건;
```

### ✅ 정답
```
① values
② select
③ from  (또는 생략 가능하나 정답 기준 from)
④ set
```

## Q12. Java — 배열 == 참조 비교

### 문제
```java
public static void check(int[] x, int[] y) {
    if(x==y) System.out.print("O");
    else System.out.print("X");
}
int a[] = new int[] {1,2,3,4};
int b[] = new int[] {1,2,3,4};
int c[] = new int[] {1,2,3};
check(a, b);   // ①
check(b, c);   // ②
check(a, c);   // ③
```

### 핵심 — 배열의 == 비교

> 배열은 **객체**! `==`는 내용이 아닌 **참조(주소) 비교**

```
a → [메모리 주소 0x100] → {1,2,3,4}
b → [메모리 주소 0x200] → {1,2,3,4}  ← 내용 같아도 다른 객체!
c → [메모리 주소 0x300] → {1,2,3}

a == b → 0x100 == 0x200 → false → X
b == c → 0x200 == 0x300 → false → X
a == c → 0x100 == 0x300 → false → X
```

> **내용 비교**가 필요하면 `Arrays.equals(a, b)` 사용

## Q18. Java — 재귀 + 중복 제거

### 문제
```java
String str = "abacabcd";
boolean[] ary = new boolean[256];
System.out.print(fn(str, str.length()-1, ary));

public static String fn(String str, int index, boolean[] ary) {
    if(index < 0) return "";
    char c = str.charAt(index);
    String result = fn(str, index-1, ary);
    if(!ary[c]) {
        ary[c] = true;
        return c + result;
    }
    return result;
}
```

### 핵심 — 재귀 + 중복 체크

- index를 **끝에서 앞으로** 처리 (재귀로 먼저 깊이 들어갔다가 돌아오면서 처리)
- `ary[c]`가 false = 처음 등장 → 출력 + true 표시
- 이미 true면 스킵

### 재귀 호출 구조

```
fn(str, 7) → fn(str, 6) → ... → fn(str, -1) = ""
                                  ↑ 여기서부터 역순으로 돌아오면서 처리

str = "abacabcd"
      0123456 7

index=7(d): fn(6)부터 먼저 처리 → result 받고, d 미등장 → "d"+result
index=6(c): fn(5)부터 처리 → result 받고, c 미등장 → "c"+result
index=5(b): fn(4)부터 처리 → result 받고, b 미등장 → "b"+result
index=4(a): fn(3)부터 처리 → result 받고, a 미등장 → "a"+result
index=3(c): fn(2)부터 처리 → result 받고, c 이미 등장 → result만
index=2(a): fn(1)부터 처리 → result 받고, a 이미 등장 → result만
index=1(b): fn(0)부터 처리 → result 받고, b 이미 등장 → result만
index=0(a): fn(-1) = "", a 이미 등장 → ""
```

### 최종 결과 조립 (역순)

index가 낮은 것부터 처리되므로 첫 등장 순서대로:
- 0:'a' 등장 → ary['a']=true
- 1:'b' 등장 → ary['b']=true
- 2:'a' 중복 → 스킵
- 3:'c' 등장 → ary['c']=true
- 4:'a' 중복 → 스킵
- 5:'b' 중복 → 스킵
- 6:'c' 중복 → 스킵
- 7:'d' 등장 → ary['d']=true

하지만 return 방식이 `c + result` → 앞에 붙임
→ 최종: **"abcd"**

### ✅ 정답
```
abcd
```
