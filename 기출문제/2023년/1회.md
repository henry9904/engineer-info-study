## Q01. AJAX

### 문제
> ( )은 웹 페이지를 새로 고치지 않고 서버로부터 데이터를 비동기적으로 가져오는 기술이다.  
> JavaScript를 이용하여 비동기적으로 서버에 데이터를 요청하고 응답을 처리하는 기술이다.  
> JavaScript를 이용하여 서버로부터 XML 또는 JSON 형식의 데이터를 비동기적으로 받아오는 프로그래밍 기법이다.

### 개념

**AJAX(Asynchronous JavaScript and XML)**

| 항목 | 내용 |
|------|------|
| 핵심 | **페이지 새로고침 없이** 서버와 데이터 교환 |
| 통신 방식 | **비동기(Asynchronous)** — 응답을 기다리지 않고 다른 작업 가능 |
| 데이터 형식 | XML, **JSON** (현재는 JSON이 사실상 표준) |
| 구현 수단 | `XMLHttpRequest` 객체, `fetch()` API |
| 핵심 장점 | 전체 페이지 리로드 없이 필요한 부분만 갱신 → UX 향상 |

### 메커니즘

```
사용자 액션
    ↓
JavaScript에서 XMLHttpRequest 또는 fetch() 호출 (비동기)
    ↓
서버에 HTTP 요청 전송 (GET/POST)
    ↓                        ← 이 동안 브라우저는 다른 작업 가능
서버에서 XML/JSON 응답 반환
    ↓
Callback / Promise로 응답 처리
    ↓
DOM만 부분 업데이트 (페이지 전체 새로고침 X)
```

### ✅ 정답

```
AJAX (Asynchronous JavaScript and XML)
```

## Q02. 가상회선 / 데이터그램

### 문제
> ① 물리적인 연결 선로가 아니라 데이터 패킷이 전송될 경로를 미리 설정하는 방식  
> ② 각각의 패킷이 독립적으로 경로를 찾아 수신자에게 전달되는 방식

### 개념

패킷 교환 방식의 두 가지 유형:

| 항목 | 가상회선 (Virtual Circuit) | 데이터그램 (Datagram) |
|------|------|------|
| 경로 설정 | **전송 전** 경로 미리 설정 | 패킷마다 **독립적**으로 결정 |
| 패킷 순서 | **보장** (같은 경로) | **보장 안 됨** |
| 신뢰성 | 높음 | 낮음 (별도 처리 필요) |
| 유연성 | 낮음 | 높음 |
| 혼잡 대응 | 어려움 | 쉬움 (다른 경로 선택) |
| 예시 프로토콜 | ATM, Frame Relay | IP (인터넷) |

### 메커니즘

```
[가상회선]
A → 경로 예약(Setup) → B
A -패킷→ -패킷→ -패킷→ B  (항상 같은 경로)
A → 연결 해제(Teardown) → B

[데이터그램]
A -패킷1→ 경로1 → B
A -패킷2→ 경로3 → B   ← 경로가 달라도 됨
A -패킷3→ 경로2 → B
B에서 순서 재조립 필요
```

### ✅ 정답

```
① 가상회선 (Virtual Circuit)
② 데이터그램 (Datagram)
```

## Q03. L2TP

### 문제
> VPN 연결을 생성하는 데 사용되는 프로토콜.  
> 마이크로소프트의 PPTP와 시스코의 L2F를 결합해 만들어졌다.  
> 데이터를 안전하게 터널링하는 기능을 제공한다.  
> 암호화를 제공하지 않아 일반적으로 IPsec와 함께 사용된다.

### 개념

**L2TP (Layer 2 Tunneling Protocol)**

| 항목 | 내용 |
|------|------|
| 계층 | OSI 2계층 (데이터링크) 기반 터널링 |
| 탄생 | PPTP(Microsoft) + L2F(Cisco) 결합 |
| 암호화 | **없음** → 반드시 **IPsec**와 함께 사용 |
| 포트 | UDP 1701 |
| 용도 | VPN (Virtual Private Network) 터널 생성 |
| 장점 | 멀티프로토콜 지원, 다양한 OS 호환 |

### VPN 터널링 개념

```
[인터넷(공중망)]

회사 내부망          L2TP 터널(암호화는 IPsec가 담당)          회사 내부망
[직원 PC] ===암호화된 패킷=======================================  [회사 서버]
              ↑
          외부에서 보면 그냥 암호화된 데이터 덩어리
```

### ✅ 정답

```
L2TP (Layer 2 Tunneling Protocol)
```

## Q04. SSH

### 문제
> 네트워크상에서 다른 컴퓨터에 로그인하거나 원격에서 명령을 실행할 수 있는 애플리케이션 프로토콜.  
> 보안성이 뛰어나 데이터 무단 접근 및 변조를 방지한다.  
> 인터넷을 통해 데이터를 전송할 때, **모든 트래픽을 암호화**하여 전송한다.  
> **22번 포트**를 사용한다.

### 개념

**SSH (Secure SHell)**

| 항목 | 내용 |
|------|------|
| 목적 | 원격 로그인, 명령 실행, 파일 전송 |
| 암호화 | **전체 트래픽 암호화** (vs Telnet: 평문) |
| 포트 | **22번** |
| 인증 방식 | 비밀번호, 공개키(Public Key) 인증 |
| 관련 도구 | `ssh`, `scp`, `sftp` |

### Telnet vs SSH 비교

| 항목 | Telnet | SSH |
|------|--------|-----|
| 암호화 | ❌ (평문) | ✅ (암호화) |
| 포트 | 23 | **22** |
| 보안 | 취약 | 강함 |
| 현재 사용 | 거의 안 씀 | 표준 |

### ✅ 정답

```
SSH (Secure SHell)
```

## Q05. 웜 / 트로이목마 / 바이러스

### 문제
> ① 네트워크를 통해 스스로를 복사하고 분산시키는 독립적인 프로그램 — **사용자 액션 없이** 전파  
> ② 유용하거나 흥미로워 보이는 프로그램으로 위장한 악성 코드 — 실행 시 악성 행동 수행  
> ③ 다른 프로그램이나 실행 가능한 소프트웨어에 '붙어' 동작하는 악성 코드 — **사용자의 액션 필요**

### 개념

| 항목 | 웜 (Worm) | 트로이목마 (Trojan Horse) | 바이러스 (Virus) |
|------|------|------|------|
| 자가복제 | ✅ | ❌ | ✅ (숙주 필요) |
| 숙주 프로그램 | 불필요 | 불필요 | **필요** |
| 전파 방법 | **자동** (네트워크) | 사용자가 직접 실행 | 감염 파일 실행 |
| 주요 피해 | 네트워크 마비 | 백도어, 정보유출 | 파일/시스템 손상 |
| 사용자 액션 | **불필요** | 필요 | 필요 |
| 예시 | Morris Worm, 슬래머 | 키로거, RAT | 부트 바이러스 |

### 핵심 구분 포인트

```
자가 전파 여부:
  웜 ✅ (자동) vs 바이러스 ✅ (사용자 액션 필요)

숙주 필요 여부:
  바이러스 ✅ (다른 프로그램에 붙음) vs 웜/트로이목마 ❌

위장 여부:
  트로이목마 ✅ (유용한 척) vs 나머지 ❌
```

### ✅ 정답

```
① 웜 (Worm)
② 트로이 목마 (Trojan Horse)  
③ 바이러스 (Virus)
```

## Q06. ICMP

### 문제
> 인터넷 프로토콜 스위트의 핵심 프로토콜 중 하나.  
> 네트워크 컴퓨터가 정보를 교환하거나 문제를 보고하는 데 사용.  
> 운영 중인 네트워크의 문제를 진단하고, 오류 메시지를 전송하고, 정보를 요청하는 데 사용.  
> TCP와 UDP 같은 다른 프로토콜들이 데이터 전송을 담당하고, 이것들과 함께 동작.

### 개념

**ICMP (Internet Control Message Protocol)**

| 항목 | 내용 |
|------|------|
| 계층 | OSI 3계층 (네트워크) / IP 위에서 동작 |
| 목적 | 오류 보고 및 네트워크 진단 |
| 데이터 전송 | ❌ (TCP/UDP가 담당) |
| 주요 메시지 | Echo Request/Reply (ping), Destination Unreachable, Time Exceeded |
| 관련 도구 | `ping`, `traceroute` |

### ICMP 동작 예시

```
[ping 명령어 사용 시]

PC  →  Echo Request (ICMP)  →  서버
PC  ←  Echo Reply   (ICMP)  ←  서버
         ↑
    왕복 시간(RTT) 측정 → 네트워크 연결 확인
```

### ✅ 정답

```
ICMP (Internet Control Message Protocol)
```

## Q07. Proxy 패턴

### 문제
> 소프트웨어 디자인 패턴 중 하나로, 어떤 객체에 대한 접근을 제어하거나 간접적으로 이루어지도록 하는 패턴.  
> 중개 역할을 하는 객체를 통해 다른 객체에 접근하는 방식 사용.  
> 내부적으로 실제 객체(Real Object)의 참조를 가지고 있으며, 클라이언트의 요청을 받아 실제 객체에 전달하고 결과를 반환.

### 개념

**Proxy 패턴** — GoF 디자인 패턴 중 **구조 패턴**에 해당

| 항목 | 내용 |
|------|------|
| 목적 | 실제 객체에 대한 **대리인** 역할 |
| 구조 | Client → Proxy → Real Object |
| 장점 | 접근 제어, 캐싱, 지연 초기화, 로깅 |
| 관련 패턴 | Decorator (기능 추가), Facade (단순화) |

### 메커니즘

```
[클라이언트]  →  request()  →  [Proxy]
                                   ↓
                          접근 제어 / 캐싱 / 로깅 처리
                                   ↓
                           [Real Object]
                           실제 작업 수행
                                   ↓
                          결과 반환 to Client
```

### Proxy vs 유사 패턴 구분

| 패턴 | 목적 |
|------|------|
| **Proxy** | 접근 **제어** (대리인) |
| Decorator | 기능 **추가** |
| Facade | 인터페이스 **단순화** |
| Adapter | 인터페이스 **변환** |

### ✅ 정답

```
Proxy (프록시 패턴)
```

## Q08. 튜플 / 릴레이션 인스턴스 / 카디널리티

### 문제
> ① 데이터 테이블의 한 행(row)을 말한다.  
> ② 특정 시점에서 관계 또는 테이블의 상태를 나타낸다.  
> ③ 특정 테이블의 행 수를 나타낸다.

### 개념

관계형 데이터베이스 기본 용어:

| DB 용어 | 일반 용어 | 설명 |
|---------|-----------|------|
| **튜플 (Tuple)** | 행 (Row) | 테이블의 한 레코드, 한 개체의 정보 |
| **어트리뷰트 (Attribute)** | 열 (Column) | 속성, 필드 |
| **릴레이션 인스턴스** | 테이블 전체 데이터 | 특정 **시점**의 테이블 상태 |
| **카디널리티 (Cardinality)** | 행 수 | 튜플의 개수 |
| **디그리 (Degree)** | 열 수 | 속성의 개수 |
| **도메인 (Domain)** | 값의 범위 | 각 속성이 가질 수 있는 값의 집합 |

### 시각화

```
학생 테이블
┌──────┬──────┬────┐
│ 학번 │ 이름 │ 학년│  ← 어트리뷰트 (Attribute) / Degree = 3
├──────┼──────┼────┤
│ 001  │ 김철수│  1  │  ← 튜플 (Tuple)
│ 002  │ 이영희│  2  │  ← 튜플 (Tuple)
│ 003  │ 박민수│  3  │  ← 튜플 (Tuple)
└──────┴──────┴────┘
  ↑ 이 전체가 릴레이션 인스턴스 (Cardinality = 3)
```

### ✅ 정답

```
① 튜플 (Tuple)
② 릴레이션 인스턴스 (Relation Instance)
③ 카디널리티 (Cardinality)
```

## Q09. 외부/개념/내부 스키마

### 문제
> ① 사용자 또는 애플리케이션의 관점에서 본 DB의 뷰 — 여러 개가 존재 가능  
> ② 전체 DB의 구조와 제약 조건을 정의 — 모든 앱/사용자가 공유하는 통합 뷰  
> ③ 데이터가 물리적으로 저장되는 방식을 설명

### 개념

데이터베이스의 3단계 스키마 구조 (ANSI/SPARC):

| 스키마 | 별명 | 관점 | 담당자 |
|--------|------|------|--------|
| **외부 스키마** (External Schema) | 서브 스키마, 사용자 뷰 | 사용자/애플리케이션 | 일반 사용자 |
| **개념 스키마** (Conceptual Schema) | 논리적 스키마 | 전체 DB 구조 | DBA |
| **내부 스키마** (Internal Schema) | 물리적 스키마 | 물리 저장 구조 | 시스템 설계자 |

### 계층 구조

```
[사용자 A 뷰]  [사용자 B 뷰]  [앱 C 뷰]
      ↑               ↑              ↑
  ←─────────  외부 스키마 (여러 개)  ─────────→

              ↕ (논리적 데이터 독립성)

  ←───────── 개념 스키마 (하나) ──────────→
         (전체 DB의 논리 구조 정의)

              ↕ (물리적 데이터 독립성)

  ←───────── 내부 스키마 (하나) ──────────→
         (실제 파일, 인덱스, 저장 방식)
```

### ✅ 정답

```
① 외부 스키마 (External Schema)
② 개념 스키마 (Conceptual Schema)
③ 내부 스키마 (Internal Schema)
```

## Q10. 분기 커버리지 (Branch Coverage)

### 문제
> 아래 제어 흐름 그래프가 분기 커버리지를 만족하기 위한 순서를 쓰시오.

```
①(POINTER=FALSE) → ②(X>K?) 
  YES → ③(POINTER=TRUE) 
  NO  → ④(X=X+1) → ⑤(CALL SUB) → ⑥(X>Y?) 
    NO  → ⑦(PRINT RESULT) → ①(루프) 
    YES → ①(루프)
```

### 개념

**테스트 커버리지 종류**

| 종류 | 설명 | 기준 |
|------|------|------|
| 구문 커버리지 | 모든 문장 1회 이상 실행 | 모든 노드 방문 |
| **분기 커버리지** | 모든 분기의 True/False 경로 모두 실행 | 모든 엣지 방문 |
| 조건 커버리지 | 각 조건의 T/F 조합 | - |
| MC/DC | 조건+분기 조합 | - |

### 메커니즘

분기 커버리지 = **모든 분기(결정 포인트)에서 True, False 경로를 각각 최소 1번씩** 실행

이 그래프의 분기 포인트:
- **②번**: `X > K?` → YES 경로(③) / NO 경로(④)
- **⑥번**: `X > Y?` → YES 경로(→①) / NO 경로(⑦)

### 풀이

**경로 1**: ①→②→③(YES)→① 루프 돌다가 ②→④(NO)→⑤→⑥→⑦(NO)→① : `1-2-3-4-5-6-1`  
**경로 2**: ②(NO)→④→⑤→⑥(YES)→① : `1-2-4-5-6-7`

분기 커버리지를 만족하는 테스트 케이스:

```
경로 1: ① → ② → ③ → ① → ② → ④ → ⑤ → ⑥ → ①  =  1-2-3-4-5-6-1
경로 2: ① → ② → ④ → ⑤ → ⑥ → ⑦              =  1-2-4-5-6-7
```

### ✅ 정답

```
1234561, 124567  (또는 1234567, 124561)
```

> 두 경로를 합쳐서 모든 엣지(YES/NO)를 커버하면 정답

## Q13. Java — static 변수 / 후위 증가 연산자

### 문제

```java

    public int a = 20;
    static int b = 0;
}
class Main {
    public static void main(String[] args) {
        int a;
        a = 10;
        TestClass.b = a;                   // (1)
        TestClass tc = new TestClass();    // (2)
        System.out.println(TestClass.b++); // (3)
        System.out.println(tc.b);          // (4)
        System.out.println(a);             // (5)
        System.out.print(tc.a);            // (6)
    }
}
```

### 개념

**static 변수 (클래스 변수)**

| 항목 | 일반 변수 (인스턴스 변수) | static 변수 (클래스 변수) |
|------|------|------|
| 메모리 | 객체마다 별도 | **클래스 하나에 공유** |

```

System.out.println(++b) → 먼저 1 더함 → 증가된 값 출력
```

| (1) | `TestClass.b = a` | `static b = 10` |
| (2) | `new TestClass()` | 객체 생성, `tc.a = 20` |
| (3) | `println(TestClass.b++)` | **b(=10)를 먼저 출력**, 그 후 b=11 |
| (4) | `println(tc.b)` | static b는 공유 → `b = 11` 출력 |
| (5) | `println(a)` | 지역변수 a = 10 → `10` 출력 |
| (6) | `print(tc.a)` | 인스턴스 변수 tc.a = 20 → `20` 출력 |

11
10
20

## Q17. C언어 — 버블정렬 빈칸 채우기

### 문제

```c
void swap(int ary[], int idx1, int idx2) {
    int tmp = ary[idx1];
    ary[idx1] = ary[idx2];
    ary[①] = tmp;          // ① = ?
}
void sort(int ary[], int len) {
    for(int i = 0; i < len; i++) {
        for(int j = 0; j < len - i - 1; j++) {
            if(ary[j] > ary[j+1]) {
                swap(ary, j, j+1);
            }
        }
    }
}
void main() {
    int ary[] = {15, 5, 20, 11, 8};
    int nx = 5;
    sort(ary, ②);           // ② = ?
    for(int i = 0; i < nx; i++) {
        printf("%d\n", ary[i]);
    }
}
```

### 개념

**버블 정렬 (Bubble Sort)**

- 인접한 두 원소를 비교하여 큰 값을 오른쪽으로 이동
- 시간 복잡도: O(n²)
- swap 함수: idx1 ↔ idx2 교환

### 메커니즘 — swap 함수 분석

```
swap(ary, idx1, idx2) 하려면:
tmp = ary[idx1]      → tmp에 idx1 값 임시 저장
ary[idx1] = ary[idx2] → idx2 값을 idx1 자리에
ary[???] = tmp       → tmp(원래 idx1 값)를 idx2 자리에

→ ①은 idx2
```

### 빈칸 결정

| 빈칸 | 이유 |
|------|------|
| ① = `idx2` | swap 완성: ary[idx2] = tmp (원래 idx1 값) |
| ② = `nx` | 배열 길이 nx=5를 sort에 전달 |

### 버블정렬 시뮬레이션 (1패스)

```
초기: [15, 5, 20, 11, 8]
j=0: 15>5  → swap → [5, 15, 20, 11, 8]
j=1: 15<20 → skip
j=2: 20>11 → swap → [5, 15, 11, 20, 8]
j=3: 20>8  → swap → [5, 15, 11, 8, 20]
...최종: [5, 8, 11, 15, 20]
```

### ✅ 정답

```
① idx2
② nx
```

출력 결과:
```
5
8
11
15
20
```

## Q19. Java — 오버라이딩 vs 오버로딩

### 문제

```java
abstract class Vehicle {
    String name;
    public Vehicle(String val) { this.name = val; }
    public String getName() { return "Vehicle name : " + name; }
}
class Car extends Vehicle {
    private String name;
    public Car(String val) { name = super.name = val; }
    public String getName(String val) { return "Car name : " + val; }  // 오버로드
    public String getName(byte val[]) { return "Car name : " + val; }  // 오버로드
}
Vehicle obj = new Car("Spark");
System.out.println(obj.getName());
```

### 개념

**오버라이딩 vs 오버로딩**

| 항목 | 오버라이딩 (Overriding) | 오버로딩 (Overloading) |
|------|------|------|
| 정의 | 부모 메서드를 자식에서 **재정의** | 같은 이름, **다른 매개변수**로 중복 정의 |
| 조건 | 메서드명, 매개변수, 리턴타입 **동일** | 메서드명 같고, 매개변수 **다름** |
| 다형성 | 런타임 다형성 | 컴파일 타임 다형성 |
| 예시 | `getName()` 재정의 | `getName(String)`, `getName(byte[])` |

### 핵심 분석

```java
// Vehicle에 정의된 메서드:
public String getName()  ← 매개변수 없음

// Car에 정의된 메서드:
public String getName(String val)  ← 매개변수 다름 (오버로딩)
public String getName(byte val[])  ← 매개변수 다름 (오버로딩)
// ⚠️ Car는 getName()을 오버라이딩하지 않음!
```

### 메커니즘 — 동적 바인딩

```
Vehicle obj = new Car("Spark");
           ↑참조타입    ↑실제타입

obj.getName() 호출 시:
  1. 컴파일러: obj의 타입(Vehicle)에서 getName() 존재 확인 ✅
  2. 런타임: 실제 객체(Car)에서 getName() 오버라이딩 여부 확인
  3. Car에 getName() (매개변수 없음) 없음!
  4. → Vehicle.getName() 실행
  5. Vehicle.getName()의 name = "Spark" (super.name으로 설정됨)
  6. 결과: "Vehicle name : Spark"
```

### ✅ 정답

```
Vehicle name : Spark
```

| `this()` | **같은 클래스**의 다른 생성자 호출 |
| `super()` | **부모 클래스**의 생성자 호출 |
| 묵시적 super() | 첫 줄에 this()/super() 없으면 **자동으로 super() 삽입** |
| 주의 | `this()` 또는 `super()`는 생성자 첫 줄에만 가능 |

### 단계별 생성자 호출 추적

```
new Child() 호출
  │
  ▼ Child() 실행 → this(5000) → Child(int x) 호출
                                    │
                                    ▼ 첫 줄에 super()/this() 없음
                                       → 묵시적 super() 삽입
                                       → Parent() 호출
                                            │
                                            ▼ this(500) → Parent(int x) 호출
                                                this.x = 500
                                                (Parent.x = 500)
                                            ▲ Parent() 종료
                                       Parent() 종료
                                    ▲ 돌아옴: this.x = 5000
                                      (Child.x = 5000)
  ▲ Child() 종료
```

### 필드 구조 분석

```
obj(Child 객체)
├── Child.x = 5000    ← Child(int x)에서 설정
└── Parent.x = 500    ← Parent(int x)에서 설정

obj.getX()
  → getX()는 Parent에 정의됨
  → return x  ← 여기서 x는 Parent의 x
  → return 500
```

### 핵심 포인트

> `getX()`가 **Parent 클래스**에 정의되어 있으므로, `return x`는 **Parent의 x**(500)를 반환한다.  
> Child의 x(5000)는 별도의 필드로 Parent.x를 가리지 않는다.

### ✅ 정답

```
500
```

## 📌 시험 전 핵심 암기 포인트

### 네트워크 용어
- `AJAX` = 비동기 JS+XML, 페이지 새로고침 없이 데이터 교환
- `L2TP` = PPTP + L2F, 암호화 없음 → IPsec와 함께 사용
- `SSH` = 암호화 원격 로그인, **포트 22**
- `ICMP` = 오류 보고 프로토콜, ping/traceroute

### 악성코드 구분
- `웜`: 자가 전파, **사용자 액션 불필요**
- `트로이목마`: 유용한 척 위장
- `바이러스`: 숙주 필요, 사용자 액션 필요

### DB 핵심
- `튜플` = 행(row), `카디널리티` = 행 수, `디그리` = 열 수
- `HAVING` = 그룹 조건 (집계함수 사용 가능), `WHERE` = 행 조건

### Java 핵심
- `static 변수`: 클래스 공유, `b++`: 출력 후 증가
- `this()`: 같은 클래스 생성자, 묵시적 `super()` 주의
- 오버라이딩: 메서드명+매개변수+리턴타입 **동일**, 오버로딩: 매개변수만 **다름**

### Python set
- `{}` 중괄호, 중복 불허, 순서 없음
- `add()` 한 개, `update()` 여러 개, `remove()` 제거
