## Q01. 조건 커버리지 (Condition Coverage)

### 지문 키워드
> **"각 조건식이 모두 참과 거짓인 경우를 최소 한 번씩 평가"**  
> **"조건문 내의 각각의 논리 조건이 True와 False로 평가될 때 해당 코드의 행동을 테스트"**

### 왜 이게 답인가?
- `소프트웨어의 각 조건식 → True/False` = **조건(Condition) 커버리지**의 정의
- **분기 커버리지**와 헷갈림 주의 → 분기는 전체 결정(결과)의 T/F, 조건은 **개별 조건**의 T/F

### 메커니즘 — 커버리지 3형제 비교

| 종류 | 기준 단위 | 예시: `if(A && B)` |
|------|-----------|---------------------|
| **구문 커버리지** | 모든 문장 1회 실행 | if 안에 들어가기만 하면 됨 |
| **분기 커버리지** | 결정(Decision) T/F | if 전체가 True 1번, False 1번 |
| **조건 커버리지** | **개별 조건** T/F | A=T/F, B=T/F 각각 모두 경험 |
| **MC/DC** | 조건+분기 복합 | 각 조건이 결정에 독립적 영향 |

> 시험 꿀팁: "각 조건식" → 조건 커버리지 / "각 분기" → 분기 커버리지

### ✅ 정답
```
조건 커버리지 (Condition Coverage)
```

## Q02. 템퍼프루핑 (Tamperproofing)

### 지문 키워드
> **"소프트웨어가 불법으로 변경되었을 경우 소프트웨어가 정상 수행되지 않게 하는 기법"**  
> **"특정 기기, 데이터 또는 소프트웨어가 무단 변경, 파괴 또는 조작으로부터 보호"**  
> **"무단 접근을 시도하는 행위를 탐지하고 방지"**

### 왜 이게 답인가?
- `무단 변경 → 정상 수행 안 됨` = **변조 방지** = Tamper(변조) + Proofing(방지)
- 소프트웨어 **무결성 보호** 목적

### 메커니즘

```
[정상 소프트웨어]
      ↓ 해커가 코드 변조 시도
[무결성 검사 (해시/서명 대조)]
      ↓ 변조 감지!
[실행 거부 / 자동 종료]   ← 템퍼프루핑의 핵심
```

| 관련 개념 | 설명 |
|----------|------|
| **템퍼프루핑** | 변조 시 동작 불가 |
| 코드 난독화 | 역공학 방지 (분석 어렵게) |
| 무결성 검사 | 변조 감지 (해시 비교) |

### ✅ 정답
```
템퍼프루핑 (Tamperproofing)
```

## Q04. 싱글톤 / 비지터 패턴

### 지문 키워드
> **① "특정 클래스의 인스턴스가 하나만 생성되도록 보장하고, 그 인스턴스에 어디서든 접근 가능"**  
> **② "객체 구조 내의 요소에 대해 수행할 작업을 캡슐화 — 클래스를 수정하지 않고도 새로운 연산을 추가"**

### 왜 이게 답인가?

| 지문 키워드 | 연결 | 패턴 |
|------------|------|------|
| **인스턴스 하나만**, 전역 접근 | 유일한 인스턴스 보장 | 싱글톤 (Singleton) |
| **클래스 수정 없이** 새 연산 추가, 객체 구조 순회 | 방문자가 연산 담당 | 비지터 (Visitor) |

### 메커니즘

```
[싱글톤]
getInstance() {
    if(instance == null) instance = new 클래스();
    return instance;   ← 항상 같은 객체 반환
}
→ DB 연결, 설정 관리 등에 활용

[비지터]
객체.accept(visitor) → visitor.visit(객체)
→ 객체 건드리지 않고 visitor만 추가하면 새 연산 가능
```

### GoF 패턴 분류 암기

| 생성 패턴 | 구조 패턴 | 행위 패턴 |
|-----------|-----------|-----------|
| **싱글톤** ✅ | Proxy, Adapter | **비지터** ✅, Iterator |
| Factory, Builder | Decorator, Facade | Observer, Strategy |

### ✅ 정답
```
① 싱글톤 (Singleton)
② 비지터 (Visitor)
```

## Q05. 오류 검출/정정 코드

### 지문 키워드
> **① "송수신 과정에서의 단일 비트 오류를 검출하고 정정할 수 있게"**  
> **② "데이터를 전송하기 전에 오류 정정 코드를 추가하여 수신측에서 발생 가능한 오류를 정정"**  
> **③ "오류가 감지된 후 데이터를 복구하기 위해 데이터 송신자에게 재전송을 요청"**  
> **④ "데이터의 무결성을 확인하기 위해 사용되는 간단한 오류 검출"**  
> **⑤ "특정 다항식을 사용하여 검사값(체크섬)을 계산하고, 이 값을 데이터에 추가"**

### 왜 이게 답인가?

| 지문 핵심 키워드 | 연결 포인트 | 정답 |
|----------------|------------|------|
| 단일 비트 오류 **검출 + 정정** 모두 | 위치까지 파악 가능 | **해밍 (Hamming)** |
| 전송 **전** 정정 코드 추가 → 수신측 자체 정정 | 재전송 없음 | **FEC** (Forward Error Correction) |
| 오류 감지 후 **재전송 요청** | 역방향 | **BEC** (Backward Error Correction) |
| **간단한** 오류 검출, 홀짝 비트 | 가장 단순 | **패리티 (Parity)** |
| **다항식** 사용, 체크섬 | 버스트 오류에 강함 | **CRC** (Cyclic Redundancy Check) |

### 메커니즘

```
[패리티]  비트 1개 → 단순 홀짝 → 검출만 가능, 정정 불가
[해밍]    여러 패리티 비트 배치 → 오류 위치까지 파악 → 정정 가능
[CRC]     다항식 나눗셈 나머지 → 강력한 버스트 오류 검출
[FEC]     미리 정정 코드 포함 → 재전송 없이 자체 복구
[BEC/ARQ] 오류 감지 시 재전송 요청 → 신뢰성 높음, 지연 발생
```

### ✅ 정답
```
① 해밍 (Hamming)
② FEC (Forward Error Correction)
③ BEC (Backward Error Correction)
④ 패리티 (Parity)
⑤ CRC (Cyclic Redundancy Check)
```

## Q06. HDLC 프레임 종류

### 지문 키워드
> **① "데이터 전송을 위한 프레임, 사용자 데이터를 포함"**  
> **② "오류 검출, 흐름 제어를 위해 사용, 데이터를 포함하지 않음"**  
> **③ "추가적인 제어 목적이나 특별한 상황 — 설정, 해제, 오류 보고"**  
> **④ "모든 단말이 주국 기능 수행 — 독립적으로 전송/수신, 양방향 통신"**  
> **⑤ "주국이 통신을 제어하지만, 종국도 자체적으로 데이터 전송 시작 가능"**

### 왜 이게 답인가?

| 지문 핵심 키워드 | 연결 포인트 | 정답 |
|----------------|------------|------|
| **사용자 데이터** 포함, 실제 전송 | I = Information | **정보 프레임 (I)** |
| **흐름/오류 제어**, 데이터 없음 | S = Supervisory | **감독 프레임 (S)** |
| **설정/해제/오류 보고** 특수 목적 | U = Unnumbered | **비번호 프레임 (U)** |
| 모든 단말 **동등**, 양방향 독립 | ABM | **비동기 균형 모드** |
| **주국이 제어**, 종국도 개시 가능 | ARM | **비동기 응답 모드** |

### ✅ 정답
```
① 정보 프레임 (I, Information)
② 감독 프레임 (S, Supervisory)
③ 비번호 프레임 (U, Unnumbered) / 비순서
④ 비동기 균형 모드 (ABM)
⑤ 비동기 응답 모드 (ARM)
```

## Q07. 대칭키 / 비대칭키 분류

### 지문 키워드
> **보기: RSA, DES, AES, ECC, ARIA, SEED**  
> **대칭키와 비대칭키로 분류**

### 왜 이게 답인가?

| 항목 | 대칭키 | 비대칭키 |
|------|--------|----------|
| 키 종류 | 암호화 = 복호화 **같은 키** | 공개키 + 개인키 **다른 키** |
| 속도 | **빠름** | **느림** |
| 키 배송 문제 | **있음** | 없음 |
| 알고리즘 | **DES, AES, ARIA, SEED** | **RSA, ECC** |

### 암기 포인트

```
대칭키  : SEED, ARIA (국산), DES, AES
             ↑ 모두 블록 암호 방식

비대칭키: RSA (소인수분해 기반)
          ECC (타원곡선 기반, RSA보다 짧은 키로 동일 보안)
```

> SEED, ARIA = **국산 알고리즘** → 대칭키

### ✅ 정답
```
대칭키  : DES, AES, ARIA, SEED
비대칭키: RSA, ECC
```

## Q08. 해시 (Hash)

### 지문 키워드
> **"보안에서 사용되는 ( )는 데이터의 무결성과 인증을 보장"**  
> **"임의 길이의 데이터를 입력받아 고정된 크기의 값을 출력"**  
> **"원본 데이터에 아주 작은 변화가 생기더라도 출력되는 값은 완전히 달라진다"**  
> **"고유한 지문과 같은 역할"**

### 왜 이게 답인가?

| 지문 키워드 | 해시의 특성 |
|------------|------------|
| **임의 길이 → 고정 길이** | 해시의 핵심 정의 |
| **작은 변화 → 완전히 다른 값** | 눈사태 효과 (Avalanche Effect) |
| **역방향 불가** | 단방향 함수 |
| **무결성 확인** | 해시의 대표 용도 |

### 메커니즘

```
"Hello" ──→ [SHA-256] ──→ "2CF24DB5..." (항상 256비트 고정)
"hello" ──→ [SHA-256] ──→ "2D711642..." (대소문자 하나 차이 → 완전히 다름!)

특성:
  같은 입력 → 항상 같은 출력 (결정적)
  역산 불가 (단방향)
  충돌 저항성 (다른 입력에서 같은 출력 찾기 불가)
```

### ✅ 정답
```
해시 (Hash)
```

## Q09. SQL — DROP VIEW CASCADE

### 지문 키워드
> **"학생뷰를 참조하는 모든 다른 뷰들을 삭제"**  
> `DROP VIEW 학생 ( )`

### 왜 이게 답인가?

| 옵션 | 동작 | 사용 시점 |
|------|------|-----------|
| **CASCADE** | 종속된 객체 **모두 함께 삭제** | 연쇄 삭제 원할 때 |
| RESTRICT | 참조 객체 있으면 **삭제 거부** | 안전하게 확인 후 삭제 |

### 메커니즘

```
학생뷰 ←─ 참조 ─ 학생성적뷰
        ←─ 참조 ─ 학생정보뷰

DROP VIEW 학생 CASCADE
→ 학생뷰 + 참조하는 모든 뷰 동시 삭제 ✅

DROP VIEW 학생 RESTRICT
→ 참조하는 뷰가 있으므로 삭제 거부 ❌
```

### ✅ 정답
```sql
DROP VIEW 학생 CASCADE;
```

## Q10. SQL — INSERT

### 지문 키워드
> **테이블 구조와 삽입할 데이터가 주어짐**

```
테이블: STUDENT (num int, name varchar(20), grade int, major varchar(30), tel varchar(20))
삽입 데이터: num=123, name=lee, grade=3, major=com, tel=010
```

### 왜 이게 답인가?

INSERT 문법:
```sql
INSERT INTO 테이블명 (컬럼1, 컬럼2, ...)
VALUES (값1, 값2, ...);
```

- 문자열(varchar) → **작은따옴표** `'...'`
- 숫자(int) → 따옴표 없음

### 컬럼-값 대응

| 컬럼 | 타입 | 값 | 따옴표 |
|------|------|-----|--------|
| num | int | 123 | 없음 |
| name | varchar | lee | `'lee'` |
| grade | int | 3 | 없음 |
| major | varchar | com | `'com'` |
| tel | varchar | 010 | `'010'` |

### ✅ 정답
```sql
INSERT INTO STUDENT (num, name, grade, major, tel)
VALUES (123, 'lee', 3, 'com', '010');
```

## Q11. 드라이버 / 스텁

### 지문 키워드
> **① "아직 개발되지 않은 모듈의 상위 모듈을 시뮬레이션 — 하위 모듈에 입력을 제공하고 출력을 받아들여 테스트"**  
> **② "아직 개발되지 않은 모듈의 하위 모듈을 시뮬레이션 — 상위 모듈이 독립적으로 테스트되도록 돕는다"**

### 왜 이게 답인가?

| 지문 핵심 | 연결 포인트 | 정답 |
|----------|------------|------|
| **상위** 모듈 시뮬레이션, **하위** 모듈 테스트 | 위에서 호출해줌 | **드라이버 (Driver)** |

## Q14. C언어 — 포인터와 gets()

### 문제
```c

char* getname() {
    printf("입력:");
    gets(n);          // 전역 n에 입력
    return n;         // 전역 n의 주소 반환
}
int main() {
    char* n1 = getname();  // 입력: 홍길동
    char* n2 = getname();  // 입력: 김철수
    char* n3 = getname();  // 입력: 박영희
    printf("%s\n", n1);
    printf("%s\n", n2);
    printf("%s\n", n3);
}
```

### 왜 이게 답인가?

**핵심**: `n`은 **전역 배열** — n1, n2, n3가 모두 **같은 주소**를 가리킴

```
전역 char n[30]의 주소: 0x1000
         ↑
n1 ──→ 0x1000  (getname 1번째 호출)
n2 ──→ 0x1000  (같은 주소!)
n3 ──→ 0x1000  (같은 주소!)

getname() 3번 호출 시:
  1번: n = "홍길동"
  2번: n = "김철수" (덮어씀)
  3번: n = "박영희" (덮어씀)

출력 시점: n은 "박영희" → n1, n2, n3 모두 "박영희" 출력
```

### 단계별 트레이싱

| 호출 | gets(n) 입력 | 전역 n 내용 | n1/n2/n3 |
|------|-------------|------------|----------|
| 1번 | 홍길동 | "홍길동" | 모두 동일 주소 |
| 2번 | 김철수 | "김철수" (덮어씀) | 모두 동일 주소 |
| 3번 | 박영희 | "박영희" (덮어씀) | 모두 동일 주소 |
| 출력 | — | "박영희" | **모두 박영희** |


## Q15. C언어 — switch fall-through

### 문제
```c
int n[3] = {73, 95, 82};
int sum = 0;
for(int i=0; i<3; i++) { sum += n[i]; }
switch(sum/30) {
    case 10:
    case 9:  printf("A");
    case 8:  printf("B");
    case 7:
    case 6:  printf("C");
    default: printf("D");
}
```

### 개념 — switch fall-through

C언어 switch는 **break가 없으면 아래 case로 계속 실행** (fall-through)

### 단계별 트레이싱

```
sum = 73 + 95 + 82 = 250
sum / 30 = 8  (정수 나눗셈, 소수점 버림: 250÷30=8.33 → 8)

case 8 진입 → printf("B")  ← 출력: B
↓ break 없음! fall-through
case 7: (출력 없음)
↓ fall-through
case 6: printf("C")          ← 출력: C
↓ fall-through
default: printf("D")         ← 출력: D
```

→ 최종 출력: **BCD**

### ✅ 정답
```
BCD
```

## Q18. Java — String == vs equals

### 문제
```java
String str1 = "Programming";
String str2 = "Programming";
String str3 = new String("Programming");

System.out.println(str1==str2);           // ①
System.out.println(str1==str3);           // ②
System.out.println(str1.equals(str3));    // ③
System.out.println(str2.equals(str3));    // ④
```

### 개념 — String Pool vs Heap

| 생성 방법 | 저장 위치 | 동일 리터럴이면? |
|----------|-----------|----------------|
| `"..."` 리터럴 | **String Pool** | 같은 객체 공유 |
| `new String("...")` | **Heap** | 항상 새 객체 생성 |

### 메커니즘

```
String Pool:  ┌─────────────────┐
              │ "Programming"   │ ← str1, str2 모두 여기 가리킴
              └─────────────────┘

Heap:         ┌─────────────────┐
              │ "Programming"   │ ← str3 (new로 생성, 별도 객체)
              └─────────────────┘

str1 == str2      → 같은 Pool 주소  → true
str1 == str3      → Pool vs Heap   → false  ← 주소가 다름!
str1.equals(str3) → 내용 비교       → true   ← "Programming" 동일
str2.equals(str3) → 내용 비교       → true
```

> **핵심**: `==` = 주소(참조) 비교 / `equals()` = 내용(값) 비교

### ✅ 정답
```
true
false
true
true
```

## Q19. C언어 — 스택 push/pop 추적

### 문제
```c
// MAX_SIZE=10, top=-1 초기화
push(5); push(2);
while(!isEmpty()) {
    printf("%d", pop());
    push(4); push(1);
    printf("%d", pop());
    push(3);
    printf("%d", pop());
    printf("%d", pop());
    push(6);
    printf("%d", pop());
    printf("%d", pop());
}
```

### 개념 — 스택 (LIFO: Last In First Out)

- `push(x)`: 최상단에 x 추가 → `stack[++top] = x`

| push(4) | [5, 4] | |
| push(1) | [5, 4, **1**] | |
| pop() | [5, 4] | **1** |
| push(3) | [5, 4, **3**] | |
| pop() | [5, 4] | **3** |
| pop() | [5] | **4** |
| push(6) | [5, **6**] | |
| pop() | [5] | **6** |
| pop() | [] | **5** |
