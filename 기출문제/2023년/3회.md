## Q01. OAuth

### 지문 키워드
> **"비밀번호를 공개하지 않고도 정보를 다른 웹사이트에서 안전하게 접근할 수 있도록 허용하는 개방형 표준"**  
> **"소셜 미디어 로그인, 데이터 공유 API 등 다양한 분야에서 널리 사용"**

### 왜 이게 답인가?

| 지문 키워드 | 연결 포인트 |
|------------|------------|
| **비밀번호 공개 없이** 다른 서비스 접근 권한 위임 | Open Authorization |
| **소셜 미디어 로그인** | "구글로 로그인" = OAuth |

### 메커니즘

```
사용자 → "구글 계정으로 로그인" 클릭
      → 구글이 "이 앱에 권한을 허용하시겠습니까?" 확인
      → 승인 시 Access Token 발급
      → 앱은 Token으로 API 호출 (비밀번호는 절대 앱에 전달 X)
```

### ✅ 정답
```
OAuth
```

## Q02. chmod 권한 설정

### 지문 키워드
> **"사용자에게 읽기/쓰기/실행 권한, 그룹에는 읽기/실행 권한, 그 이외에는 실행 권한만을 text.txt 파일에 부여"**

### 왜 이게 답인가?

**권한 숫자 계산:**

| 권한 | 숫자 |
|------|------|
| 읽기 (r) | **4** |
| 쓰기 (w) | **2** |
| 실행 (x) | **1** |

| 대상 | 권한 | 계산 | 숫자 |
|------|------|------|------|
| 소유자 | 읽기+쓰기+실행 | 4+2+1 | **7** |
| 그룹 | 읽기+실행 | 4+1 | **5** |
| 기타 | 실행만 | 1 | **1** |

```
chmod  7  5  1  text.txt
       ↑  ↑  ↑
    소유자 그룹 기타
```

### ✅ 정답
```
chmod 751 text.txt
```

## Q03. SQL — UNION / ORDER BY

### 문제
```sql
SELECT A FROM T1
UNION
SELECT B FROM T2
ORDER BY A DESC;
```

T1: {A, 4, 1, 3} / T2: {B, 4, 2, 3}

### 개념

| 연산 | 동작 |
|------|------|
| **UNION** | 두 결과 합치기 + **중복 제거** |
| UNION ALL | 합치기 + 중복 포함 |

### 트레이싱

```
UNION 결과 (중복 제거): {A, B, 1, 2, 3, 4}
ORDER BY A DESC (내림차순): A, 4, 3, 2, 1
(문자 A는 숫자보다 뒤, B도 포함 시 B>A 순)
```

### ✅ 정답
```
A
4
3
2
1
```

## Q04. MAC / DAC / RBAC

### 지문 키워드
> **① "리소스마다 보안 레이블, 사용자마다 보안 등급 — 주로 국방·정부 기관"**  
> **② "데이터 소유자가 자신의 자원에 대한 접근 권한을 직접 관리"**  
> **③ "사용자의 역할에 따라 접근 권한을 부여"**

### 왜 이게 답인가?

| 지문 핵심 키워드 | 정답 |
|----------------|------|
| **보안 등급/레이블**, 국방·정부 | **MAC** (Mandatory Access Control) |
| **소유자**가 직접 권한 관리 | **DAC** (Discretionary Access Control) |
| **역할(Role)** 기반 권한 | **RBAC** (Role-Based Access Control) |

### 메커니즘

```
MAC  → 시스템이 강제 (소유자도 마음대로 변경 불가)
DAC  → 소유자가 결정 (chmod처럼 내 파일 권한 내가 설정)
RBAC → 역할로 묶음 (관리자 역할 = 모든 권한, 사원 역할 = 읽기만)
```

### ✅ 정답
```
① MAC (Mandatory Access Control)
② DAC (Discretionary Access Control)
③ RBAC (Role-Based Access Control)
```

## Q05. ATM

### 지문 키워드
> **"광대역 멀티미디어 통신 지원"**  
> **"데이터를 53바이트 크기의 고정 길이 셀로 전송"**  
> **"5바이트는 헤더, 48바이트는 페이로드"**  
> **"데이터 전송 전에 논리적 연결이 먼저 설정"**

### 왜 이게 답인가?

**53바이트 고정 셀** = ATM의 가장 강력한 식별 키워드

```
┌──────────┬────────────────────────────────┐
│ 헤더(5B) │         페이로드(48B)           │
└──────────┴────────────────────────────────┘
←───────────── 53바이트 고정 ──────────────→
```

- 고정 셀 크기 → 스위칭 빠름 → 실시간 음성/영상 적합
- 논리적 연결 먼저 (가상회선 방식) → 순서 보장

### ✅ 정답
```
ATM (Asynchronous Transfer Mode)
```

## Q08. 동치 분할 (Equivalence Partitioning)

### 지문 키워드
> **"전체 입력 데이터를 여러 개의 '등가 클래스'로 나눈다"**  
> **"한 클래스의 하나의 값을 테스트하면, 해당 클래스의 다른 모든 값도 동일한 결과를 보일 것"**  
> **"테스트 케이스의 수를 줄일 수 있다"**

### 왜 이게 답인가?

| 키워드 | 연결 |
|--------|------|
| **등가 클래스** | Equivalence = 동치/동등 |
| **대표값 하나**로 클래스 전체 테스트 | 효율적 테스트 |
| **케이스 수 줄임** | 테스트 최소화 |

### 메커니즘

```
점수 입력 (1~100):
[유효]   1 ≤ 점수 ≤ 100 → 대표값 50 하나만 테스트
[무효1]  점수 < 1       → 대표값 0 하나만 테스트
[무효2]  점수 > 100     → 대표값 101 하나만 테스트

→ 100개 테스트 → 3개로 압축!
```

### ✅ 정답
```
동치 분할 (Equivalence Partitioning)
```

## Q11. 관계 대수 기호

### 지문 키워드
> ① JOIN / ② PROJECT / ③ SELECT / ④ DIVISION

### 왜 이게 답인가?

| 연산 | 기호 | SQL 대응 | 역할 |
|------|------|---------|------|
| **SELECT** | **σ (시그마)** | WHERE | **행** 필터링 |
| **PROJECT** | **π (파이)** | SELECT 절 | **열** 선택 |
| **JOIN** | **⋈** | JOIN | 두 테이블 합치기 |
| **DIVISION** | **÷** | — | 나눗셈 연산 |

### 핵심 암기

```
σ(시그마) = SELECT  → 조건으로 행(Row) 선택
π(파이)   = PROJECT → 속성(Column) 선택
⋈         = JOIN    → 테이블 합체
```

> **주의**: σ=행선택(SELECT), π=열선택(PROJECT) — 헷갈리기 쉬움!

### ✅ 정답
```
① ⋈  (JOIN)
② π  (PROJECT, 파이)
③ σ  (SELECT, 시그마)
④ ÷  (DIVISION)
```

## Q13. Java — 오버라이딩 / super / this 추적

### 문제
```java
class A {
    public void paint() { System.out.print("A"); draw(); }
    public void draw()  { System.out.print("B"); draw(); }  // 동적 바인딩!
}
class B extends A {
    public void paint() {
        super.draw();          // A의 draw() 강제 호출
        System.out.print("C");
        this.draw();           // B의 draw() 호출
    }
    public void draw() { System.out.print("D"); }
}
A b = new B();
b.paint();   // B의 paint() 실행
b.draw();    // B의 draw() 실행
```

### 핵심 개념 — 동적 바인딩

> `A b = new B()` → 실제 객체는 B  
> 오버라이딩된 메서드 → **항상 실제 객체(B) 기준**  
> `super.draw()` → **명시적으로 A의 draw() 강제 호출**

### 단계별 트레이싱

**b.paint() → B의 paint() 실행:**

| 단계 | 실행 코드 | 출력 |
|------|----------|------|
| 1 | `super.draw()` → A의 draw() 강제 호출 | |
| 2 | A.draw(): `print("B")` | **B** |
| 3 | A.draw(): `draw()` → 동적 바인딩 → B의 draw() | |
| 4 | B.draw(): `print("D")` | **D** |
| 5 | B.paint() 복귀: `print("C")` | **C** |
| 6 | `this.draw()` → B의 draw() | |
| 7 | B.draw(): `print("D")` | **D** |

**b.draw() → B의 draw():**

| 단계 | 실행 | 출력 |
|------|------|------|
| 1 | B.draw(): `print("D")` | **D** |

→ 전체: **BDCD** + **D** = **BDCDD**

### ✅ 정답
```
BDCDD
```

## Q14. C언어 — 포인터 산술

### 문제
```c
char* p = "KOREA";
printf("%s\n", p);       // ①
printf("%s\n", p+1);     // ②
printf("%c\n", *p);      // ③
printf("%c\n", *(p+3));  // ④
printf("%c", *p+4);      // ⑤ ← 핵심 함정!
```

### ⚠️ 핵심 — `*(p+4)` vs `*p+4`

```
"KOREA" 인덱스:
K(0) O(1) R(2) E(3) A(4)
K의 ASCII = 75

*(p+4) → p에서 4칸 이동 → 'A' 출력
*p+4   → *p='K'=75, 75+4=79='O' 출력  ← 이게 지문의 코드!
```

### 트레이싱

| 코드 | 동작 | 출력 |
|------|------|------|
| `%s, p` | p부터 문자열 | **KOREA** |
| `%s, p+1` | p+1부터 문자열 | **OREA** |
| `%c, *p` | p[0]='K' | **K** |
| `%c, *(p+3)` | p[3]='E' | **E** |
| `%c, *p+4` | 'K'(75)+4=79='O' | **O** |

### ✅ 정답
```
KOREA
OREA
K
E
O
```

## Q15. C언어 — 구조체 포인터 접근자

### 문제
```c
struct User { char *name; int age; };
struct User *d2;
d2(  )name = "Lee";    // ← 빈칸
d2(  )age = 45;
```

### 왜 이게 답인가?

| 변수 종류 | 접근 방법 |
|----------|-----------|
| 일반 구조체 변수 `d1` | `d1.name` (점) |
| 구조체 **포인터** `d2` | `d2->name` (화살표) |

```
d2->name  ==  (*d2).name
   ↑               ↑
```

## Q17. Java — static 메서드 오류 찾기

### 문제
```java
class Person {
    private String name;
    public Person(String val) { name = val; }
    public static String get() {
        return name;    // ← 오류!
    }
    public void print() { System.out.println(name); }
}
```

### 왜 이게 답인가?

**static 메서드 제약:**
- static 메서드는 **클래스 로딩 시** 존재
- 인스턴스 변수(`name`)는 **객체 생성 후** 존재
- static 메서드에서 인스턴스 변수 접근 → **컴파일 오류**

```
public static String get() {
    return name;  // name은 인스턴스 변수 → static에서 접근 불가!
}
// 오류: "non-static variable name cannot be referenced from a static context"
```

### ✅ 정답
```java
return name;
```

## Q18. Java — 오버라이딩 재귀 추적

### 문제
```java
class Parent {
    int com(int n) {
        if(n <= 1) return n;
        return com(n-1) + com(n-2);   // 피보나치
    }
}
class Child extends Parent {
    int com(int n) {
        if(n <= 1) return n;           // 음수도 그대로 반환!
        return com(n-1) + com(n-3);   // 변형 재귀
    }
}
Parent obj = new Child();
System.out.print(obj.com(7));
```

### 핵심 — 동적 바인딩

`Parent obj = new Child()` → 모든 `com()` 호출은 **Child의 com()** 실행

### 트레이싱 (Child.com 기준)

| 호출 | 계산 | 결과 |
|------|------|------|
| com(-1) | -1≤1 → return -1 | **-1** |
| com(0) | 0≤1 → return 0 | **0** |
| com(1) | 1≤1 → return 1 | **1** |
| com(2) | com(1)+com(-1) = 1+(-1) | **0** |
| com(3) | com(2)+com(0) = 0+0 | **0** |
| com(4) | com(3)+com(1) = 0+1 | **1** |
| com(5) | com(4)+com(2) = 1+0 | **1** |
| com(6) | com(5)+com(3) = 1+0 | **1** |
| com(7) | com(6)+com(4) = 1+1 | **2** |

### ✅ 정답
```
2
```

```
     = 7 × 6 × 5 × 4 × 3 × 2 × f(1)
     = 7 × 6 × 5 × 4 × 3 × 2 × 1
     = 5040
```
