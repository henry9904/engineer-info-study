## Q01. 관계대수 — σ, π, ∧

### 문제

> Student(SID, SName, Major, Score)에서 전공='컴퓨터공학' **AND** 점수 ≥ 80인 학생의 **학번·이름** 조회
> (σ, π, ∧만 사용)

### 핵심 개념 — 관계대수 기호

| 기호       | 이름     | 역할                    |
| -------- | ------ | --------------------- |
| **σ**(시그마) | 선택(Selection) | **행(조건)** 필터 |
| **π**(파이) | 추출(Projection) | **열(속성)** 선택 |
| **∧**    | AND    | 조건 결합                 |

### 표현식 구성

```
1) 조건으로 행 선택:  σ Major='컴퓨터공학' ∧ Score≥80 (Student)
2) 원하는 열만 추출:  π SID, SName ( ... )
```

### ✅ 정답

```
π SID, SName (σ Major='컴퓨터공학' ∧ Score≥80 (Student))
```
> ⚠️ σ(조건)를 안쪽에, π(열 선택)를 바깥에. 순서 바뀌면 안 됨(먼저 행 거르고 열 뽑기).

## Q02. C언어 — 비트/논리 연산 + 단락 평가

### 문제

```c
int a = 6, b = 4, c = 9, d = 2;
int result;
result = (a++ & ~b) || (--c ^ (d++ & a)) && (b-- | ~(c--));
result = result && (a++ | b++) || (c++ & d++);
printf("출력 : %d, %d, %d, %d, %d\n", result, a, b, c, d);
```

### 🤔 핵심 함정 — `||`와 `&&`의 단락 평가

`||`는 왼쪽이 참(0 아님)이면 오른쪽 **평가 안 함**. 그 안의 증감(`--c`, `d++` 등)도 실행 안 됨.

### 단계별 트레이싱

**첫 번째 문장**
```
좌변 (a++ & ~b):
  a++ → 식엔 6 사용, a→7
  ~b = ~4 = -5 (…11111011)
  6 & -5 = 6 & …1011 = 0110 & 1011 = 0010 = 2 (0 아님 → 참)
→ || 왼쪽이 참 → 오른쪽 전체 단락(평가 안 함)
   (--c, d++, b--, c-- 모두 실행 안 됨 → c=9, d=2, b=4 그대로)
result = 1 (참)
```
상태: a=7, b=4, c=9, d=2, result=1

**두 번째 문장**
```
result && (a++ | b++):
  result=1(참) → 오른쪽 평가
  a++ → 7 사용, a→8;  b++ → 4 사용, b→5
  7 | 4 = 0111|0100 = 0111 = 7 (0 아님 → 참)
  1 && 7 → 참(1)
→ || 왼쪽이 참 → (c++ & d++) 단락(평가 안 함) → c=9, d=2 그대로
result = 1
```
최종: result=1, a=8, b=5, c=9, d=2

### ✅ 정답

```
출력 : 1, 8, 5, 9, 2
```
> ⚠️ 두 문장 모두 `||` 왼쪽이 참이라 오른쪽 증감이 대거 생략됨 → c, d가 초기값 유지.

## Q03. C언어 — 구조체 배열 인덱스 조작

### 문제 (요지)

```c
struct Container { int data[5]; int pos; };
void processData(struct Container* c) {
    c->pos = -1;
    for (int i = 1; i <= 5; i++) {
        if (c->pos < 4) {
            if (i % 2 == 0) c->data[++(c->pos)] = i * 2;
            else            c->data[++(c->pos)] = i + 3;
        }
        if (i == 3 || i == 5) { if (c->pos >= 0) c->pos--; }
    }
    // 이후 마지막 원소 꺼내 가공 + 9 push
}
```

### 🤔 이 함수가 하는 일

`pos`는 스택의 top(꼭대기 인덱스)처럼 동작. `++(c->pos)`는 **먼저 pos를 1 증가시킨 뒤 그 자리에 저장**(push). i==3, i==5일 땐 pos를 하나 내려(pop) 다음 push가 그 자리를 덮어쓴다.

### 한 줄씩 따라가기 — for 루프 (pos는 -1에서 시작)

```
i=1 (홀수): pos<4 참 → data[++pos=0] = 1+3 = 4      pos=0, data=[4]
            i==3||5? X
i=2 (짝수): pos<4 참 → data[++pos=1] = 2*2 = 4      pos=1, data=[4,4]
            i==3||5? X
i=3 (홀수): pos<4 참 → data[++pos=2] = 3+3 = 6      pos=2, data=[4,4,6]
            i==3 → pos>=0 → pos-- → pos=1  (6은 남아있지만 top은 1)
i=4 (짝수): pos<4 참 → data[++pos=2] = 4*2 = 8      pos=2, data=[4,4,8]  (data[2]=6→8 덮어씀)
            i==3||5? X
i=5 (홀수): pos<4 참 → data[++pos=3] = 5+3 = 8      pos=3, data=[4,4,8,8]
            i==5 → pos>=0 → pos-- → pos=2
```
루프 종료: data=[4,4,8,8], **pos=2** (top은 data[2]=8)

### 한 줄씩 따라가기 — 루프 후 처리

```
pos>=0 참 →
   temp = data[pos--] = data[2] = 8,  pos→1
   temp(8) 짝수 && pos<4 참 →
      data[++pos=2] = temp/2 = 8/2 = 4    pos=2, data=[4,4,4,8]
마지막 if: pos<4 참 →
   data[++pos=3] = 9                       pos=3, data=[4,4,4,9]
```
최종: data=[4,4,4,9], **pos=3**

### show() — pos부터 0까지 역순 출력

```
i=pos=3: data[3]=9 → "9 "
i=2:     data[2]=4 → "4 "
i=1:     data[1]=4 → "4 "
i=0:     data[0]=4 → "4 "
```

### ✅ 정답

```
9 4 4 4 
```
> ⚠️ i==3,5에서 pos를 되돌려(pop) 다음 push가 이전 값을 덮어씀. 역순 출력이라 9가 먼저.

## Q04. OCP + LSP (SOLID 원칙)

### 지문 키워드 → 정답

| 빈칸 | 지문 키워드                                | 정답        |
| --- | ------------------------------------ | --------- |
| ①  | 기존 모듈 **변경 없이 기능 확장**                | **OCP(개방-폐쇄 원칙)** |
| ②  | 상위 타입을 하위 타입으로 **치환해도 문제없음**         | **LSP(리스코프 치환 원칙)** |

### 메커니즘 — SOLID 5원칙

| 약자 | 원칙            | 핵심                    |
| --- | ------------- | --------------------- |
| S  | 단일 책임(SRP)    | 클래스는 하나의 책임           |
| **O** | **개방-폐쇄(OCP)** | **확장엔 열림, 변경엔 닫힘** |
| **L** | **리스코프 치환(LSP)** | **자식이 부모를 대체 가능** |
| I  | 인터페이스 분리(ISP) | 필요한 인터페이스만            |
| D  | 의존 역전(DIP)    | 추상에 의존                |

### ✅ 정답

① OCP(개방-폐쇄 원칙)  ② LSP(리스코프 치환 원칙)

## Q05. SQL — EXISTS 상관 서브쿼리

### 문제

```sql
SELECT S.SID, S.SName
FROM Student S
WHERE S.Major = 'Computer'
AND EXISTS (
    SELECT 1 FROM Course C
    WHERE C.SID = S.SID AND C.CName = 'Database' AND C.Score >= 80
);
```
Student: S001 Alice(Computer), S002 Bob(Math), S003 Charlie(Computer), S004 David(Eng)
Course: S001 Database 85, S001 Database 90, S002 Database 78, S003 Database 79, S003 Algorithm 90, S004 Database 88

### 🤔 핵심 — 두 조건을 **모두** 만족해야

1. Major = 'Computer' (Alice, Charlie만)
2. **AND** EXISTS: 그 학생이 Database를 80점 이상 수강한 기록이 있어야

### 단계별 트레이싱

| 학생  | Computer? | Database ≥80 기록                  | 최종 |

| Alice(S001)  | ✅ | Database 85, 90 → 있음 ✅          | **포함** |

| Charlie(S003) | ✅       | Database 79(80 미만), Algorithm 90 → **Database ≥80 없음** ❌ | 제외 |
| David(S004)  | ❌ Eng    | Database 88 있지만 전공 아님           | 제외 |

> ⚠️ Charlie는 Computer지만 Database 점수가 79(80 미만) → EXISTS 실패. David는 Database 88이지만 전공이 Eng → Major 조건 실패.

## Q06. Java — Set/Map + 정렬

### 문제 (요지)

```java
List<String> list = [Apple, Banana, Cherry, Banana];
Set<String> set = new HashSet<>(list);   // 중복 제거 → {Apple, Banana, Cherry}
set.add("Date");                          // {Apple, Banana, Cherry, Date}
Map<Integer,String> map = new HashMap<>();
int idx = 1;
for (String s : set) if (s.startsWith("B")) map.put(idx++, s);  // B로 시작
map.put(3, "Berry");
// map의 value들을 정렬 후 출력
```

### 단계별 트레이싱

```
set = {Apple, Banana, Cherry, Date}  (중복 Banana 제거)
"B"로 시작하는 것 = "Banana"뿐 → map.put(1, "Banana")
map.put(3, "Berry") → map = {1:Banana, 3:Berry}
result = [Banana, Berry]
Collections.sort → [Banana, Berry] (사전순)
출력: "Banana Berry "
```
> 💡 HashSet 순서는 불규칙하지만, "B"로 시작하는 원소가 Banana 하나뿐이라 결과가 순서에 무관.

### ✅ 정답

```
Banana Berry 
```

## Q07. Python — 중첩 리스트 컴프리헨션

### 문제

```python
a = [i for i in range(5) if i % 2 == 0]           # 짝수
b = [x+1 for x in a if x < 4]                      # 4 미만 +1
c = [y*2 if y % 3 == 0 else y+3 for y in b]        # 3배수면 *2, 아니면 +3
print(c)
```

### 단계별 트레이싱

```
a = [0, 2, 4]                        (0~4 중 짝수)
b: a에서 x<4인 것(0,2) → +1 → [1, 3]  (4는 제외)
c: 각 y에 대해
   y=1: 1%3≠0 → 1+3 = 4
   y=3: 3%3=0 → 3*2 = 6
→ c = [4, 6]
```

| 단계 | 결과      |
| -- | ------- |
| a  | [0,2,4] |
| b  | [1,3]   |
| c  | [4,6]   |

### ✅ 정답

```
[4, 6]
```

## Q09. 페이지 교체 — LRU vs LFU 부재 횟수

### 문제

> 참조열: 7,0,1,2,0,3,0,4,2,3,0,3,2 / 프레임 3개 / 동률 시 가장 먼저 적재된 것 제거(FIFO)

### ① LRU (Least Recently Used) — 가장 오래 안 쓴 것 제거

| 참조 | 프레임 상태(→최근)  | 부재? |
| -- | ------------ | --- |
| 7  | [7]          | F   |
| 0  | [7,0]        | F   |
| 1  | [7,0,1]      | F   |
| 2  | [0,1,2] (7제거) | F   |
| 0  | [1,2,0]      | hit |
| 3  | [2,0,3] (1제거) | F   |
| 0  | [2,3,0]      | hit |
| 4  | [3,0,4] (2제거) | F   |
| 2  | [0,4,2] (3제거) | F   |
| 3  | [4,2,3] (0제거) | F   |
| 0  | [2,3,0] (4제거) | F   |
| 3  | [2,0,3]      | hit |
| 2  | [0,3,2]      | hit |

부재 = 7,0,1,2,3,4,2,3,0 → **9회**

### ② LFU (Least Frequently Used) — 가장 적게 쓴 것 제거(동률 FIFO)

| 참조 | 처리                             | 부재? |
| -- | ------------------------------ | --- |
| 7  | [7:1]                          | F   |
| 0  | [7:1,0:1]                      | F   |
| 1  | [7:1,0:1,1:1]                  | F   |
| 2  | 동률→FIFO 7제거 [0:1,1:1,2:1]      | F   |
| 0  | 0:2                            | hit |
| 3  | 최소 1:1,2:1 →FIFO 1제거 [0:2,2:1,3:1] | F   |
| 0  | 0:3                            | hit |
| 4  | 최소 2:1,3:1 →FIFO 2제거 [0:3,3:1,4:1] | F   |
| 2  | 최소 3:1,4:1 →FIFO 3제거 [0:3,4:1,2:1] | F   |
| 3  | 최소 4:1,2:1 →FIFO 4제거 [0:3,2:1,3:1] | F   |
| 0  | 0:4                            | hit |
| 3  | 3:2                            | hit |
| 2  | 2:2                            | hit |

부재 = 7,0,1,2,3,4,2,3 → **8회**

### ✅ 정답

① LRU = 9회  ② LFU = 8회

## Q10. 워킹 셋 + 스래싱

### 지문 키워드 → 정답

| 빈칸 | 지문 키워드                          | 정답        |
| --- | ------------------------------ | --------- |
| ①  | 프로세스가 **자주 참조하는 페이지 집합**       | **워킹 셋(Working Set)** |
| ②  | 페이지 교체에 시간 대부분 소비, 성능 급락       | **스래싱(Thrashing)** |

### 메커니즘

```
워킹 셋: 최근 일정 시간(윈도우) 내 참조된 페이지 집합
        → 이만큼 메모리를 주면 페이지 부재↓
스래싱 : 메모리 부족 → 페이지 부재 폭증 → CPU가 교체에만 매달림
         → 다중프로그래밍 정도가 임계 넘으면 발생
해결: 워킹셋 모델, 페이지 부재 빈도(PFF) 조절
```

### ✅ 정답

① 워킹 셋(Working Set)  ② 스래싱(Thrashing)

## Q11. 옵저버 패턴 (Observer Pattern)

### 지문 키워드

> 객체 간 **일대다(One-to-Many)** 의존, 주체(Subject) 상태 변경 시 연결된 **의존 객체들에 자동 통지**, 결합도 낮춤

### 왜 이게 답인가?

| 지문 키워드                       | 정답        |
| ---------------------------- | --------- |
| 일대다 의존 + 상태 변경 자동 알림(관찰)     | **옵저버 패턴** |

### 메커니즘 — 디자인 패턴 분류(행위 패턴)

```
옵저버(Observer): Subject ─(상태변경)→ 등록된 Observer들에게 notify()
  예: 이벤트 리스너, 구독/발행(pub-sub), MVC의 뷰 갱신
```

## Q12. Java — 참조 전달 + 메서드 상호 호출

### 문제

```java
class Ref {
    int a;
    Ref(int x) { a = x; }
    int modify(Ref obj) {
        int temp = this.a;
        this.a = obj.a + 1;
        obj.a = temp - 1;
        return this.a + obj.a;
    }
    int chain(Ref obj) {
        int k = this.modify(obj);
        k += obj.modify(this);
        return k;
    }
}
Ref obj1 = new Ref(5);  Ref obj2 = new Ref(7);
int result = obj1.chain(obj2);
```

### 🤔 핵심 — 객체는 참조 전달, modify가 두 객체를 동시에 바꿈

### 단계별 트레이싱

시작: obj1.a=5, obj2.a=7. `obj1.chain(obj2)` 실행.

**k = obj1.modify(obj2)** (this=obj1, obj=obj2)
```
temp = obj1.a = 5
obj1.a = obj2.a + 1 = 7+1 = 8
obj2.a = temp - 1 = 5-1 = 4
return obj1.a + obj2.a = 8+4 = 12
→ k=12,  현재 obj1.a=8, obj2.a=4
```

**k += obj2.modify(obj1)** (this=obj2, obj=obj1)
```
temp = obj2.a = 4
obj2.a = obj1.a + 1 = 8+1 = 9
obj1.a = temp - 1 = 4-1 = 3
return obj2.a + obj1.a = 9+3 = 12
→ k = 12 + 12 = 24,  현재 obj1.a=3, obj2.a=9
```

result = 24, obj1.a = 3, obj2.a = 9

### ✅ 정답

```
result=24, obj1.a=3, obj2.a=9
```

## Q14. C언어 — char 포인터 + 16진수 값

### 문제

```c
char arr[] = {0x1A, 0x2B, 0x3C, 0x4D, 0x5E};
char *p = arr;  char *q = &arr[2];
printf("%d, ", *p);        // arr[0]
printf("%d, ", *(p + 3));  // arr[3]
printf("%d, ", *(q - 1));  // arr[1]
printf("%d, ", *(q + 1));  // arr[3]
*q = *(p + 4) + *(p);      // arr[2] = arr[4]+arr[0]
*(p + 1) = *(q) - *(p + 3);// arr[1] = arr[2]-arr[3]
printf("%d, ", arr[1]);
printf("%d ", arr[2]);
```

### 16진수 → 10진수

```
0x1A=26, 0x2B=43, 0x3C=60, 0x4D=77, 0x5E=94
```

### 단계별 트레이싱

`q = &arr[2]`.

| 출력식        | 대상        | 값 |
| ---------- | --------- | - |
| `*p`       | arr[0]    | **26** |
| `*(p+3)`   | arr[3]    | **77** |
| `*(q-1)`   | arr[1]    | **43** |
| `*(q+1)`   | arr[3]    | **77** |

**대입 연산**
```
*q = *(p+4) + *(p) → arr[2] = 94 + 26 = 120
   (char라 오버플로 가능하나 결과 120은 char 범위 내)
*(p+1) = *(q) - *(p+3) → arr[1] = arr[2](120) - arr[3](77) = 43
```

| 출력식      | 값 |
| -------- | - |
| `arr[1]` | **43** |
| `arr[2]` | **120** |

### ✅ 정답

```
26, 77, 43, 77, 43, 120 
```
> ⚠️ `*q = 94+26 = 120`을 먼저 반영한 뒤 arr[1] 계산에 arr[2]=120이 사용됨.

## Q15. SQL — CROSS JOIN + LIKE + LENGTH

### 문제

```sql
SELECT COUNT(*) AS CNT
FROM T1 CROSS JOIN T2
WHERE T1.NAME LIKE T2.RULE
AND LENGTH(T1.NAME) = LENGTH(T2.RULE) + 1;
```
T1.NAME: KING, JONES, CLARK, BLAKE
T2.RULE: `_I%`, `%E`, `%A%`

### 🤔 핵심 — LIKE 매칭 **AND** 길이 조건 둘 다

- `_I%`: 두 번째 글자가 I (앞 1글자 + I + 나머지)
- `%E`: E로 끝남
- `%A%`: A 포함
- **추가 조건**: NAME 길이 = RULE 길이 + 1

### 단계별 트레이싱 — 4×3=12 조합 중 필터

| NAME  | RULE  | LIKE 매칭          | LEN(NAME)=LEN(RULE)+1 | 통과 |
| ----- | ----- | ---------------- | --------------------- | -- |
| KING  | `_I%` | ✅ (K**I**NG)     | 4 = 3+1 ✅            | ✅  |
| KING  | `%E`  | ❌ (E로 안 끝남)      | -                     | ❌  |
| KING  | `%A%` | ❌ (A 없음)         | -                     | ❌  |
| JONES | `_I%` | ❌ (2번째 O)        | -                     | ❌  |
| JONES | `%E`  | ❌ (S로 끝남)        | -                     | ❌  |
| JONES | `%A%` | ❌ (A 없음)         | -                     | ❌  |
| CLARK | `%A%` | ✅ (CL**A**RK)    | 5 = 3+1=4? ❌ (5≠4)   | ❌  |
| BLAKE | `%E`  | ✅ (BLAK**E**)    | 5 = 2+1=3? ❌         | ❌  |
| BLAKE | `%A%` | ✅ (BL**A**KE)    | 5 = 3+1=4? ❌         | ❌  |

> 💡 LIKE로는 여러 개 매칭되지만 **길이 조건(NAME = RULE+1)**까지 만족하는 건 KING/`_I%`뿐.
> `_I%`의 길이 3, KING 길이 4 → 4=3+1 ✅. CLARK/BLAKE는 길이 5라서 탈락.

### ✅ 정답

| CNT |
| --- |
| 1   |

## Q16. Python — 클래스 변수 vs 인스턴스 + classmethod

### 문제

```python
class Parent:
    num = 30
    def __init__(self, val): self.num = val      # 인스턴스 변수 num 생성
    @classmethod
    def modify(cls, x): cls.num += x             # 클래스 변수 수정
class Child(Parent):
    num = 20
    def __init__(self, val): self.num = val
    @classmethod
    def modify(cls, x): cls.num -= x
a = Parent(10); b = Child(20)
a.show(); b.show()          # 인스턴스 num
a.modify(10); b.modify(10)  # 클래스 num
a.show(); b.show()          # 인스턴스 num (변화 없음)
print(Parent.num, Child.num) # 클래스 num
```

### 🤔 핵심 함정 — 인스턴스 `self.num`과 클래스 `cls.num`은 별개

`__init__`의 `self.num=val`이 **인스턴스 변수**를 만들어 클래스 변수를 가림. `show()`는 인스턴스 num을 봄. `modify()`는 **클래스 변수**만 바꿈 → 인스턴스 num엔 영향 없음.

### 단계별 트레이싱

```
a = Parent(10) → a.num(인스턴스)=10, Parent.num(클래스)=30
b = Child(20)  → b.num(인스턴스)=20, Child.num(클래스)=20

a.show() → 인스턴스 a.num = 10
b.show() → 인스턴스 b.num = 20

a.modify(10) → Parent.num += 10 → 30→40 (클래스 변수만)
b.modify(10) → Child.num -= 10 → 20→10 (클래스 변수만)

a.show() → 인스턴스 a.num = 10 (여전히)
b.show() → 인스턴스 b.num = 20 (여전히)

print(Parent.num, Child.num) → 40 10
```

### ✅ 정답

```
10
20
10
20
40 10
```
> ⚠️ modify는 클래스 변수(cls.num)만 바꿔 인스턴스 값(10,20)엔 무영향. 마지막 클래스 변수만 40/10.

| 1 | B      | 홀수, 1%3≠0 → 없음  | -     | "E#D@C#B"    |
