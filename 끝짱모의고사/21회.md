## Q02. C언어 — 연결 리스트 노드 교환

### 문제

```c
// add로 앞쪽 삽입: 1,2,3,4 → 리스트 4→3→2→1
// swap(&h, 2, 4): 값 2와 4 노드를 통째로 교환(포인터 조작)
```

### 🤔 이 함수가 하는 일

값 x, y를 가진 노드의 **위치(포인터)를 통째로 교환**한다.

### 1단계 — 리스트 구성

```
add 1,2,3,4 (앞쪽 삽입) → 4 → 3 → 2 → 1
```

### 2단계 — swap(&head, 2, 4)

```
현재:  head → [4] → [3] → [2] → [1] → NULL

찾기: 값4 노드(맨 앞), 값2 노드(뒤쪽). 이 두 "노드 자체"의 위치를 바꾼다.
      값을 복사하는 게 아니라 next 링크(포인터)를 재배치.

배치 후:
      head → [2] → [3] → [4] → [1] → NULL
       ▲      ▲            ▲
       │    옛날 4 자리   옛날 2 자리
    head가 이제 [2]를 가리킴
```
> 포인터 교환의 핵심: 노드 안의 data는 그대로(2,4)이고, 누가 누구를 가리키느냐(next)만 바뀐다. 그래서 값 2와 4가 통째로 자리를 맞바꾼 효과.

### ✅ 정답

```
2 3 4 1 
```
> ⚠️ data 복사가 아닌 노드 포인터(위치) 교환 → 2 3 4 1.

## Q04. SQL — LEFT JOIN + 상관 서브쿼리

### 문제

```sql
SELECT e.name AS A, m.name AS B, e.salary AS C
FROM employees e LEFT JOIN employees m ON e.m_id = m.id
WHERE e.salary > (SELECT AVG(salary) FROM employees WHERE dept = e.dept);
```
영업부: 홍길동(6000), 이영희(8000) → 평균 7000
개발부: 김철수(5000), 박민수(7500), 최수진(4800) → 평균 5766.7

### 단계별 트레이싱

```
홍길동 6000 > 영업부평균 7000? ✗
이영희 8000 > 7000? ✓ → 관리자 m_id=104(박민수) → 이영희, 박민수, 8000
김철수 5000 > 개발부평균 5766? ✗
박민수 7500 > 5766? ✓ → m_id=NULL → LEFT JOIN → 박민수, NULL, 7500
최수진 4800 > 5766? ✗
```

### ✅ 정답

| A | B | C |
| - | - | - |
| 이영희 | 박민수 | 8000 |
| 박민수 | NULL | 7500 |

> ⚠️ 박민수는 m_id가 NULL이라 LEFT JOIN으로 B=NULL. 각자 자기 부서 평균과 비교.

## Q06. Python — 재귀 병합(홀짝 기준)

### 문제

```python
def tricky(arr):
    if len(arr) <= 1: return arr
    mid = len(arr) // 2
    left = tricky(arr[:mid]); right = tricky(arr[mid:])
    result = []
    while left and right:
        if left[0] % 2 == right[0] % 2:   # 두 앞원소 홀짝 같으면 left
            result.append(left.pop(0))
        else:
            result.append(right.pop(0))
    result.extend(left or right)
    return result
arr = [5, 3, 2, 9, 1, 4]
```

### 한 단계씩 따라가기

```
tricky([5,3,2,9,1,4]): mid=3 → left=tricky([5,3,2]), right=tricky([9,1,4])
  tricky([5,3,2]): mid=1 → left=[5], right=tricky([3,2])
     tricky([3,2]): mid=1 → [3],[2]
        merge([3],[2]): 3홀 vs 2짝 다름 → right(2) → [2], left=[3] 남음 → extend → [2,3]
     merge([5],[2,3]): 5홀 vs 2짝 다름 → right(2) → [2]; 5홀 vs 3홀 같음 → left(5) → [2,5]; right=[3] → extend → [2,5,3]
  tricky([9,1,4]): mid=1 → [9], tricky([1,4])
     tricky([1,4]): [1],[4] → 1홀 vs 4짝 다름 → right(4) → [4], left=[1] → [4,1]
     merge([9],[4,1]): 9홀 vs 4짝 다름 → right(4) → [4]; 9홀 vs 1홀 같음 → left(9) → [4,9]; right=[1] → [4,9,1]
  최종 merge([2,5,3],[4,9,1]):
     2짝 vs 4짝 같음 → left(2) → [2]
     5홀 vs 4짝 다름 → right(4) → [2,4]
     5홀 vs 9홀 같음 → left(5) → [2,4,5]
     3홀 vs 9홀 같음 → left(3) → [2,4,5,3]
     left=[] → extend(right=[9,1]) → [2,4,5,3,9,1]
```

### ✅ 정답

```
[2, 4, 5, 3, 9, 1]
```

## Q07. 문장/분기/경로 커버리지 경로

### 🤔 핵심 — 커버리지별 필요 경로

- **문장 커버리지**: 모든 문장(노드)을 1회 이상 → 가장 긴 한 경로면 충분
- **분기 커버리지**: 모든 분기(T/F)를 1회 이상
- **경로 커버리지**: 가능한 모든 독립 경로

### ✅ 정답 (제어 흐름도 기준)

```
① 문장: 1 → 2 → 3 → 4 → 5
② 분기: 1 → 2 → 3 → 4 → 5, 1 → 3 → 5
③ 경로: 1→2→3→4→5, 1→2→3→5, 1→3→4→5, 1→3→5
```
> 💡 문장은 1경로, 분기는 T/F 모두, 경로는 모든 조합.

## Q10. Java — 정적/인스턴스 초기화 + 가상 호출

### 문제 (요지)

```java
class Alpha { static{"A"} {"B"} Alpha(){"C"; show();} void show(){"X"} }
class Beta extends Alpha { static{"D"} {"E"} Beta(){"F"; show();} void show(){"Y"} }
class Gamma extends Beta { static{"G"} {"H"} Gamma(){"I"; show();} }
new Gamma();
```

### 🤔 초기화 순서 3규칙

```
① 정적 블록 static{}: 클래스 최초 로딩 때 "딱 1회", 상위→하위 순 (A→D→G)
② 인스턴스 블록 {} + 생성자: 객체 만들 때마다, super 먼저 완전히 끝난 뒤 자기 것
   → 한 클래스 안에서는 [인스턴스 블록] → [생성자 본문] 순서
③ show()는 인스턴스 메서드 → "실제 객체 타입"으로 찾아감 (가상 호출)
   객체는 Gamma → Gamma에 show 없음 → 부모로 올라가 Beta.show()="Y"
```
> 함정: 생성자가 아직 다 안 끝났어도 show()는 이미 **실제 타입(Gamma)** 기준으로 호출된다. "생성 중 가상 호출" → 항상 Beta.show()="Y".

### 1단계: 클래스 로딩 (정적 블록, 1회)

```
Alpha 로딩 "A" → Beta 로딩 "D" → Gamma 로딩 "G"   →  "ADG"
```

### 2단계: new Gamma() — super 먼저 타고 내려갔다 올라오기

| 순서 | 실행 | show() 대상 | 출력 |
| -- | -- | -- | -- |
| Alpha 인스턴스 블록 | "B" | | **B** |
| Alpha() 본문 | "C" → show() | 객체=Gamma → Beta.show() | **C Y** |
| Beta 인스턴스 블록 | "E" | | **E** |
| Beta() 본문 | "F" → show() | 객체=Gamma → Beta.show() | **F Y** |
| Gamma 인스턴스 블록 | "H" | | **H** |
| Gamma() 본문 | "I" → show() | 객체=Gamma → Beta.show() | **I Y** |

생성 출력: **B C Y E F Y H I Y**

**전체** = "ADG"(정적) + "BCYEFYHIY"(생성) = **ADGBCYEFYHIY**

### ✅ 정답

```
ADGBCYEFYHIY
```
> ⚠️ 정적 블록은 최초 1회(ADG). show()는 실제타입 Gamma지만 Gamma에 없어 상속된 Beta.show()="Y" 세 번.

## Q13. C언어 — 이중 포인터 + 단락 평가

### 문제

```c
int arr[3][2] = {{10,20},{30,40},{50,60}};
int *ptrArr[3] = {arr[2], arr[1], arr[0]};   // {50..}, {30..}, {10..}
int **dp = ptrArr;  int x = 0;
if (**dp > 40 && ++x) printf("%d ", x); else printf("%d ", **dp);      // ①
printf("%d ", *(*(arr+1)+1));                                          // ②
if (**(ptrArr+2) < 5 || ++x) printf("%d ", x); else printf(...);       // ③
dp++;
if (**dp == 30 && ++x) printf("%d ", x); else printf(...);             // ④
printf("%d", *(dp[1]+1));                                              // ⑤
```

### 🤔 이중 포인터 메모리 그림

```
arr(2차원)          ptrArr(포인터 배열)         dp(이중 포인터)
arr[0]={10,20} ◄─── ptrArr[2] ─┐
arr[1]={30,40} ◄─── ptrArr[1] ─┼─ dp ──► 처음엔 &ptrArr[0]
arr[2]={50,60} ◄─── ptrArr[0] ─┘

*dp   = ptrArr[?] (포인터 하나)      → 한 번 풀면 int* (행)
**dp  = 그 행의 [0]번째 int          → 두 번 풀면 실제 값
```
> `ptrArr`는 각 행을 "거꾸로" 가리킴: [0]→arr[2], [1]→arr[1], [2]→arr[0].

### 단락 평가(short-circuit) 규칙

```
&& : 앞이 거짓이면 뒤(++x)는 "실행 안 함"
|| : 앞이 참이면   뒤(++x)는 "실행 안 함"
→ ++x가 실제로 도는지 매 줄 확인해야 x 값이 맞음
```

### 한 줄씩 따라가기 (x 값 추적)

| 줄 | 식 풀이 | 판정 | x | 출력 |
| -- | -- | -- | -- | -- |
| 초기 | | | 0 | |
| ① | **dp = *ptrArr[0]=arr[2][0]=**50** | 50>40 참 → ++x 실행(x=1), 참 | 1 | **1** |
| ② | *(*(arr+1)+1)=arr[1][1] | =40 | 1 | **40** |
| ③ | **(ptrArr+2)=arr[0][0]=**10** | 10<5 거짓 → ++x 실행(x=2), 참 | 2 | **2** |
| dp++ | dp = &ptrArr[1] (→arr[1]) | | 2 | |
| ④ | **dp = arr[1][0]=**30** | 30==30 참 → ++x 실행(x=3) | 3 | **3** |
| ⑤ | dp[1]=ptrArr[2]=arr[0], +1, * | =arr[0][1]=20 | 3 | **20** |

### ✅ 정답

```
1 40 2 3 20
```

```java
class Node { int data; Node next; static int total=0;
   Node(int d){data=d; total+=data;}
   int update(Node o){ int sum=data+o.data; data+=3; o.data-=2; if(next!=null)next.data+=1; total+=sum; return sum; }
   int cascade(Node n1,Node n2){ int r1=update(n1); int r2=n1.update(n2); int r3=n2.update(this); data=data-n1.data+n2.data; return r1+r2+r3; } }
A=Node(5), B=Node(10), C=Node(15); A→B, B→C, C→A;
```

### 🤔 update()가 한 번에 바꾸는 3곳 + total

```
update(o) 는 부작용이 3개 + 누적 1개:
  this.data += 3     (자기 자신 +3)
  o.data    -= 2     (인자로 받은 객체 -2)
  next.data += 1     (자기가 가리키는 다음 노드 +1, next 있을 때만)
  total     += (this.data+o.data)  ← "바꾸기 전" 값들의 합을 누적
연결: A→B→C→A (원형). total은 static이라 모든 노드가 공유.
```
> 순서 주의: sum은 **바꾸기 전** data로 계산 → 그 다음 this·o·next를 수정.

### 객체 상태판 (한 줄 실행할 때마다 A/B/C/total)

| 실행 | 계산 | A | B | C | total |
| -- | -- | -- | -- | -- | -- |
| 생성 | 5+10+15 | 5 | 10 | 15 | **30** |
| **B.update(C)** | sum=10+15=25 | 5 | 13 | 13→+1(next)=14 | +25=**55** |
| → return 25 | | 5 | **13** | **14** | |
| **C.cascade(A,B)** ↓ | | | | | |
| r1=C.update(A) | sum=14+5=19 | 3→+1(C.next=A)=4 | 13 | 17 | +19=**74** |
| r2=A.update(B) | sum=4+13=17 | 7 | 11→+1(A.next=B)=12 | 17 | +17=**91** |
| r3=B.update(C) | sum=12+17=29 | 7 | 15 | 17→+1(B.next=C)... 15 | +29=**120** |
| C.data 재계산 | C(16)−A(7)+B(15) | 7 | 15 | **24** | |

- res1 = 25 → print
- cascade return = r1+r2+r3 = 19+17+29 = **65**
- total = **120**
- 최종 A,B,C = **7, 15, 24**

> 💡 r3에서 B.update(C): C.data는 17→15(−2)로 먼저 바뀐 뒤 B.next=C라 +1 되어 16. 이후 마지막 줄 `C.data = 16−7+15 = 24`.

### ✅ 정답

```
30
25
5, 13, 14
65
120
7, 15, 24
```
> ⚠️ update가 this·other·next 3개를 바꾸고 total 누적. cascade는 세 번 update + 마지막 재계산.

## Q15. SQL — CASE WHEN + GROUP BY

### 문제

```sql
SELECT dept, CASE WHEN AVG(salary)>7000 THEN 'H' WHEN AVG(salary)>4000 THEN 'M'
   WHEN AVG(salary)>0 THEN 'L' ELSE '-' END AS grade
FROM employees GROUP BY dept;
```

### 단계별 트레이싱

```
영업부: (6000+8000)/2 = 7000 → 7000>7000? ✗ → >4000? ✓ → 'M'
개발부: (5000+7500+2800)/3 = 5100 → >4000 → 'M'
인사부: 강민지 NULL → AVG(NULL 제외) = NULL → NULL>0 거짓 → ELSE '-'
```

### ✅ 정답

| dept | grade |
| ---- | ----- |
| 영업부 | M |
| 개발부 | M |
| 인사부 | - |

> ⚠️ 영업부 평균 정확히 7000 → `>7000`은 거짓이라 'H' 아니고 'M'. 인사부는 전부 NULL이라 AVG=NULL → '-'.
